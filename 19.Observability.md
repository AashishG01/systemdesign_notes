## 1. What is Observability?

**Observability** is the ability to understand a system's internal state by looking at its external outputs (logs, metrics, traces). It answers: "Why is the system broken?" — not just "Is the system broken?"

> **💡 Analogy — Car Dashboard:**
> - **Monitoring** = the dashboard warning lights (engine temperature, fuel, battery). Tells you SOMETHING is wrong.
> - **Logging** = the black box recorder. Records everything that happened so you can investigate AFTER the crash.
> - **Tracing** = GPS tracking. Shows the exact path a journey took, how long each segment took, and where the delay happened.

```
Without Observability:       With Observability:
  "The site is slow."          "Request /api/orders took 5.2s.
                                2ms in API server.
   ...why?                      5100ms in database.
   ...which service?             Query: SELECT * FROM orders
   ...which query?               WHERE status = 'pending'
   ...since when?                Missing index on 'status' column.
   ...how many affected?         Affecting 23% of requests since
                                 yesterday's deploy."
```

## 2. The 3 Pillars of Observability

| Pillar | What It Captures | Question It Answers | Tool Examples |
| :--- | :--- | :--- | :--- |
| **Logs** | Discrete events with context | "What exactly happened?" | ELK Stack, Loki, CloudWatch Logs |
| **Metrics** | Numeric measurements over time | "How is the system performing right now?" | Prometheus + Grafana, Datadog, CloudWatch |
| **Traces** | End-to-end request journey across services | "Where is the bottleneck in this request?" | Jaeger, Zipkin, OpenTelemetry, Datadog APM |

## 3. Logging

### 3.1 Structured vs Unstructured Logs

```
❌ UNSTRUCTURED (hard to search):
  [2024-03-01 10:15:23] ERROR: Failed to process order for user john_doe

✅ STRUCTURED (JSON — searchable, parsable):
  {
    "timestamp": "2024-03-01T10:15:23.456Z",
    "level": "ERROR",
    "service": "order-service",
    "message": "Failed to process order",
    "user_id": "user_123",
    "order_id": "order_789",
    "error": "Payment gateway timeout",
    "duration_ms": 5200,
    "trace_id": "abc-123-def-456"
  }
```

### 3.2 Log Levels

| Level | When to Use | Example |
| :--- | :--- | :--- |
| **DEBUG** | Detailed internal state (development only) | `"Processing item 3 of 5, price=29.99"` |
| **INFO** | Normal business events | `"Order #789 created for user #123"` |
| **WARN** | Something unexpected but not broken | `"Redis cache miss. Falling back to DB."` |
| **ERROR** | Something failed, needs attention | `"Payment gateway returned 500. Retrying."` |
| **FATAL** | System is crashing | `"Database connection pool exhausted. Shutting down."` |

## 4. Monitoring & Metrics

### 4.1 The 4 Golden Signals (Google SRE)

| Signal | What It Measures | Alert If |
| :--- | :--- | :--- |
| **Latency** | Time to serve a request | p99 latency > 500ms |
| **Traffic** | Requests per second | RPS drops > 50% from baseline |
| **Errors** | Rate of failed requests | Error rate > 1% of total requests |
| **Saturation** | How "full" your resources are | CPU > 80%, Memory > 90%, DB connections > 80% |

### 4.2 RED Method (for request-driven services)

| Metric | Description |
| :--- | :--- |
| **R**ate | Requests per second |
| **E**rrors | Errors per second |
| **D**uration | Time per request (latency histogram) |

### 4.3 USE Method (for infrastructure)

| Metric | Description | Example |
| :--- | :--- | :--- |
| **U**tilization | % of resource being used | CPU at 75% |
| **S**aturation | Work queued because resource is busy | 50 requests waiting in thread pool |
| **E**rrors | Count of errors | 12 disk I/O errors/min |

### 4.4 Percentile Latencies (Critical Concept)

```
Why AVERAGE latency is MISLEADING:

  99 requests take 50ms, 1 request takes 5000ms.
  Average = (99×50 + 5000) / 100 = 99.5ms  ← looks fine!
  
  But 1% of users wait 5 SECONDS. At 10M requests/day, that's 100,000 angry users.

USE PERCENTILES:
  p50 (median):  50ms    ← half of requests are faster than this
  p95:           60ms    ← 95% of requests are faster
  p99:           5000ms  ← 1% of requests are this slow ← THIS IS YOUR PROBLEM
```

## 5. Distributed Tracing

### 5.1 Why You Need It

In microservices, a single user request hops through 5-10 services. When it's slow, which service is the bottleneck?

```
User request: GET /api/checkout

  → API Gateway (2ms)
    → Auth Service (15ms)
      → Order Service (20ms)
        → Inventory Service (30ms)
        → Payment Service (4500ms)  ← 💥 BOTTLENECK!
          → Stripe API (4480ms)     ← External API is slow
      → Email Service (async, 0ms impact)
    
  Total: 4567ms

WITHOUT tracing: "The checkout is slow." (Where? No idea.)
WITH tracing:    "Payment Service → Stripe API took 4.48s." (Exact root cause.)
```

### 5.2 How It Works

```
1. Request enters the system → generate a unique TRACE ID: "trace-abc-123"
2. Each service receives the trace ID in a header (X-Trace-ID)
3. Each service creates a SPAN (start time, end time, service name, metadata)
4. All spans are sent to a tracing backend (Jaeger)
5. Jaeger stitches all spans into a TRACE TIMELINE:

  ├── API Gateway ──────── 2ms ──────────────────────────────────────┤
  │   ├── Auth Service ── 15ms ───┤                                  │
  │   ├── Order Service ─── 20ms ──────────────────────────────────┤│
  │   │   ├── Inventory ── 30ms ─┤                                 ││
  │   │   ├── Payment ────────────── 4500ms ─────────────────────┤ ││
  │   │   │   └── Stripe API ────── 4480ms ────────────────────┤ │ ││
  │   │   └── Email (async) ── 0ms ┤                             │ ││
```

## 6. Alerting Best Practices

### The Alerting Pyramid

```
             ╱ ╲
            ╱ P1 ╲        PAGE (wake someone up at 3 AM)
           ╱ PAGE ╲        → System is DOWN or data is being lost
          ╱────────╲
         ╱   P2     ╲     URGENT TICKET
        ╱  TICKET    ╲    → Degraded performance, needs fix within hours
       ╱──────────────╲
      ╱     P3         ╲  WARNING / SLACK NOTIFICATION
     ╱   NOTIFICATION   ╲ → Something unusual, investigate during work hours
    ╱────────────────────╲
   ╱      DASHBOARD       ╲  DASHBOARD ONLY
  ╱     (no alert)         ╲  → Informational metrics, no action needed
 ╱──────────────────────────╲
```

### Alert Fatigue (The #1 Problem)

```
❌ BAD: 200 alerts/day → engineers ignore ALL of them → miss the real emergency.

✅ GOOD: Every alert is ACTIONABLE.
  → If receiving the alert doesn't change your behavior → DELETE the alert.
  → Every alert should have a RUNBOOK: "When you see this, do steps 1-2-3."
```

## 7. The Observability Stack

| Stack | Components | Best For |
| :--- | :--- | :--- |
| **ELK** | Elasticsearch + Logstash + Kibana | Log aggregation and search |
| **PLG** | Promtail + Loki + Grafana | Lightweight log aggregation (cheaper than ELK) |
| **Prometheus + Grafana** | Prometheus (metrics DB) + Grafana (dashboards) | Infrastructure and application metrics |
| **Jaeger / Zipkin** | Distributed tracing backends | Request tracing across microservices |
| **OpenTelemetry** | Vendor-neutral SDK for logs, metrics, traces | Unified instrumentation (send to any backend) |
| **Datadog / New Relic** | All-in-one SaaS platform | Everything (logs + metrics + traces + APM) |

## 8. Interview Questions

### ❓ Q1: "Your API is slow but you don't know why. How do you debug it?"

**Answer:** "Use the 3 pillars in order:
1. **Metrics** (Grafana) → Is p99 latency high? Which endpoint? Since when?
2. **Traces** (Jaeger) → Find a slow request. View the trace → which downstream service is the bottleneck?
3. **Logs** → Filter by the trace ID → see the exact error or slow query in that service."

### ❓ Q2: "How would you set up monitoring for a new microservices system?"

**Answer:** "Instrument the 4 Golden Signals for every service: RPS, error rate, latency (p50/p95/p99), and saturation (CPU/memory/DB connections). Use Prometheus for metrics, Grafana for dashboards, Jaeger for tracing, and Loki for logs. Set alerts: page if error rate > 5% or p99 > 2 seconds. Every alert has a runbook."

### ❓ Q3: "What's the difference between monitoring and observability?"

**Answer:** "Monitoring tells you WHEN something is broken (alert: CPU > 90%). Observability tells you WHY it's broken (trace shows the DB query is doing a full table scan because of a missing index). Monitoring is a subset of observability."
