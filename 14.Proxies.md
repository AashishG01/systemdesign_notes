## 1. What is a Proxy?

A **proxy** is an intermediary server that sits between a client and a server, forwarding requests on behalf of one of them.

```
WITHOUT PROXY:
  Client в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶ Server

WITH PROXY:
  Client в”Җв”Җв–¶ PROXY в”Җв”Җв–¶ Server
             (intermediary)
```

There are two types, and they sit on **opposite sides**:

```
  FORWARD PROXY                         REVERSE PROXY
  (sits near CLIENT)                    (sits near SERVER)

  Client в”Җв”Җв–¶ [Forward Proxy] в”Җв”Җв–¶ Internet в”Җв”Җв–¶ Server
  
  Client в”Җв”Җв–¶ Internet в”Җв”Җв–¶ [Reverse Proxy] в”Җв”Җв–¶ Server
```

## 2. Forward Proxy

### 2.1 What It Is

A Forward Proxy sits in front of **clients** and forwards their requests to the internet. The server sees the proxy's IP, not the client's IP.

> **рҹ’Ў Analogy:** You want to buy something anonymously. You ask your friend (forward proxy) to buy it for you. The shopkeeper sees your friend, not you.

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ                в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ              в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ Client A в”Ӯв”Җв”Җв”Җв”җ            в”Ӯ               в”Ӯ              в”Ӯ          в”Ӯ
в”Ӯ Client B в”Ӯв”Җв”Җв”Җв”јв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶в”Ӯ FORWARD PROXY в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶в”Ӯ  Server  в”Ӯ
в”Ӯ Client C в”Ӯв”Җв”Җв”Җв”ҳ            в”Ӯ               в”Ӯ              в”Ӯ          в”Ӯ

Server sees: Request from Proxy (1.2.3.4)
```

### 2.2 Use Cases

| Use Case | How |
| :--- | :--- |
| **Anonymity / Privacy** | Client hides behind proxy IP. Server can't identify the real client. (VPN is a type of forward proxy) |
| **Content Filtering** | Company blocks employees from accessing social media. Proxy intercepts requests to facebook.com вҶ’ rejects. |
| **Bypass Geo-Restrictions** | Client in India connects to a US-based proxy вҶ’ accesses US-only Netflix content. |
| **Caching** | Proxy caches frequently requested resources. 100 employees access same page вҶ’ proxy serves cached copy, saves bandwidth. |
| **Logging / Monitoring** | Company logs all internet traffic through the proxy for compliance. |

## 3. Reverse Proxy

### 3.1 What It Is

A Reverse Proxy sits in front of **servers** and handles all incoming client requests. The client talks to the proxy, not knowing which backend server actually processes the request.

> **рҹ’Ў Analogy:** You call a customer service number. You don't know which agent will answer. The call router (reverse proxy) connects you to an available agent. You never know the agent's direct number.

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ              в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ              в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ          в”Ӯ              в”Ӯ               в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶в”Ӯ Server A в”Ӯ
в”Ӯ  Client  в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶в”Ӯ REVERSE PROXY в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–¶в”Ӯ Server B в”Ӯ

Client sees: proxy.example.com
Client does NOT see: Server A (10.0.0.1), Server B (10.0.0.2)
```

### 3.2 Use Cases

| Use Case | How |
| :--- | :--- |
| **Load Balancing** | Distributes requests across multiple backend servers. (NGINX, HAProxy) |
| **SSL Termination** | Proxy handles HTTPS encryption/decryption. Backend servers use plain HTTP (faster). |
| **Caching** | Proxy caches API responses. Repeated requests served without touching backend. |
| **Security** | Backend servers are hidden. Attackers can only see the proxy's IP. No direct access to backends. |
| **Compression** | Proxy compresses responses (gzip) before sending to client. Reduces bandwidth. |
| **Rate Limiting** | Proxy enforces rate limits before requests reach the backend. |
| **A/B Testing** | Proxy routes 10% of traffic to new version (Server C), 90% to stable (Server A, B). |

## 4. Forward vs Reverse Proxy вҖ” Key Differences

| Feature | Forward Proxy | Reverse Proxy |
| :--- | :--- | :--- |
| **Sits in front of** | Clients | Servers |
| **Hides** | Client's identity from server | Server's identity from client |
| **Who configures it** | Client / client's organization | Server / server's organization |
| **Purpose** | Client anonymity, content filtering, bypass restrictions | Load balancing, security, caching, SSL termination |
| **Client awareness** | Client KNOWS it's using a proxy (configured in browser/OS) | Client does NOT know a proxy exists (transparent) |
| **Examples** | Squid, corporate proxies, VPNs | NGINX, HAProxy, Cloudflare, Traefik |

```
FORWARD PROXY (CLIENT-SIDE):

  [Your Company Network]
  Employee A в”Җв”Җв”җ
  Employee B в”Җв”Җв”јв”Җв”Җв–¶ [Forward Proxy] в”Җв”Җв–¶ Internet в”Җв”Җв–¶ google.com
  Employee C в”Җв”Җв”ҳ    (filters, logs)

REVERSE PROXY (SERVER-SIDE):

  Internet в”Җв”Җв–¶ [NGINX Reverse Proxy] в”Җв”Җв–¶ App Server 1
                                    в”Җв”Җв–¶ App Server 2
                                    в”Җв”Җв–¶ App Server 3
```

## 5. Reverse Proxy vs Load Balancer

This is **the most commonly confused** topic. Here's the truth:

| | Reverse Proxy | Load Balancer |
| :--- | :--- | :--- |
| **Primary job** | Intermediary between client and servers (many features) | Distribute traffic across servers (one feature) |
| **Features** | SSL termination, caching, compression, rate limiting, routing, security | Traffic distribution algorithms (round-robin, least connections) |
| **Operates with** | Even a single backend server (still useful) | Multiple backend servers (pointless with 1 server) |
| **Relationship** | A reverse proxy CAN act as a load balancer | A load balancer IS a subset of reverse proxy features |

> **рҹ’Ў Key Insight:** Every load balancer is a reverse proxy, but not every reverse proxy is a load balancer. NGINX is a reverse proxy that INCLUDES load balancing. AWS ALB is primarily a load balancer that includes some reverse proxy features.

## 6. Real-World Use Cases

### NGINX as Reverse Proxy (Most Common Setup)

```nginx
# NGINX configuration вҖ” reverse proxy + load balancer
upstream backend_servers {
    server 10.0.0.1:3000;   # App Server 1
    server 10.0.0.2:3000;   # App Server 2
    server 10.0.0.3:3000;   # App Server 3
}

server {
    listen 443 ssl;                    # SSL termination at proxy
    server_name api.example.com;

    # SSL certificate (handled by proxy, not backend)
    ssl_certificate     /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    # Proxy to backends (plain HTTP internally)
    location / {
        proxy_pass http://backend_servers;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # Serve static files directly from proxy (no backend hit)
    location /static/ {
        root /var/www/static;
        expires 30d;
    }
}
```

### Cloudflare as Reverse Proxy (Global Scale)
```
Every website using Cloudflare is behind a reverse proxy:

User вҶ’ Cloudflare Edge (reverse proxy) вҶ’ Your Origin Server

Cloudflare provides:
  вң… DDoS protection (blocks attacks at the edge)
  вң… CDN (caches static content at 200+ locations)
  вң… SSL/TLS (free HTTPS certificate)
  вң… WAF (blocks SQL injection, XSS)
  вң… Rate limiting
  вң… Bot detection
  
All this happens BEFORE the request reaches your server.
```

## 7. Interview Questions

### вқ“ Q1: "What's the difference between a forward proxy and a reverse proxy?"

**Answer:** "A forward proxy sits in front of **clients** and hides the client's identity вҖ” like a VPN. A reverse proxy sits in front of **servers** and hides the server's identity вҖ” like NGINX or Cloudflare. Forward proxy is configured by the client. Reverse proxy is configured by the server. In system design, we almost always talk about **reverse proxies** (load balancing, SSL termination, caching)."

### вқ“ Q2: "Why use a reverse proxy even with a single backend server?"

**Answer:** "Even with 1 server, a reverse proxy provides: SSL termination (backend doesn't handle HTTPS), caching (reduce server load), compression (smaller responses), rate limiting (protect from abuse), security (hide server's real IP), and it makes adding a 2nd server trivial later."

### вқ“ Q3: "Your API is slow. Would you add a CDN or a reverse proxy cache?"

**Answer:** "Both, at different layers:
- **CDN** for geographically distributed users accessing static/cacheable content (images, HTML, public API responses).  
- **Reverse proxy cache** (NGINX `proxy_cache`) for API responses that are expensive to compute but don't need global distribution (DB-heavy endpoints serving users in the same region)."
