## 1. The CAP Theorem

### 1.1 The Definition

The **CAP Theorem** (Brewer's Theorem) states that a distributed system can provide **at most 2 out of 3** guarantees simultaneously:

| Letter | Guarantee | Meaning |
| :--- | :--- | :--- |
| **C** | **Consistency** | Every read receives the most recent write. All nodes see the same data at the same time. |
| **A** | **Availability** | Every request receives a response (success or failure). No request is ignored. |
| **P** | **Partition Tolerance** | The system continues to function even if network communication between nodes is lost. |

> **💡 Intuition — The Two ATMs:**
> You and your friend both have cards for the same bank account ($1000 balance). You go to ATM in Delhi, friend goes to ATM in Mumbai. The network cable between the two bank data centers gets cut.
>
> - **Consistency:** Both ATMs show the same balance. But if the network is cut, one ATM must STOP working (sacrifice Availability) to prevent showing stale data.
> - **Availability:** Both ATMs keep working. But they might show different balances (sacrifice Consistency) because they can't sync.
> - **You can't have both** when the network is partitioned.

## 2. Understanding C, A, and P

### 2.1 Consistency (C)

```
CONSISTENT (All nodes return same data):

  Client writes: "balance = 500" → Node A
  
  Node A ──sync──▶ Node B ──sync──▶ Node C
  
  Any node you read from returns: balance = 500 ✅
  
NOT CONSISTENT (Stale read):

  Client writes: "balance = 500" → Node A
  Network lag: Node B still has "balance = 1000"
  
  Read from Node B → Returns 1000 ❌ (stale!)
```

### 2.2 Availability (A)

```
AVAILABLE: Every request gets a response (even if it's stale data).
  → Client asks Node B for balance → Node B responds: "$1000" (even if stale)
  → System is AVAILABLE ✅ (response given, even if not perfectly accurate)

NOT AVAILABLE: Node refuses to respond.
  → Client asks Node B for balance → Node B: "I can't answer right now. 
     I haven't synced with Node A yet." → 503 error.
  → System is UNAVAILABLE ❌ (no response)
```

### 2.3 Partition Tolerance (P)

```
PARTITION: Network between nodes is broken.

  ┌────────┐         BROKEN          ┌────────┐
  │ Node A │ ────── ✕ ────── ────── │ Node B │
  │ (Delhi)│       network           │(Mumbai)│
  └────────┘     partition           └────────┘
  
  Node A can't talk to Node B. They're isolated.
  
PARTITION TOLERANT: System keeps working despite the broken link.
NOT PARTITION TOLERANT: System goes down entirely.
```

> **💡 Critical Insight:** In any distributed system, **network partitions WILL happen**. Cables get cut. Data centers lose connectivity. Cloud AZs go down. **P is not optional** — you MUST have Partition Tolerance. So the real choice is: **CP or AP**.

## 3. Why You Can Only Pick 2 (Actually, It's CP or AP)

### 3.1 The Real Choice

Since P is mandatory in any distributed system, the choice is:

```
┌───────────────────────────────────────────────────────┐
│   CP: Consistency + Partition Tolerance                │
│   "During a partition, STOP serving some requests     │
│    rather than serve stale data."                     │
│   → Some users get errors, but no one gets WRONG data │
│                                                       │
│   AP: Availability + Partition Tolerance              │
│   "During a partition, KEEP serving all requests      │
│    even if data might be stale."                      │
│   → Everyone gets a response, but data might be OLD   │
└───────────────────────────────────────────────────────┘
```

### 3.2 CP vs AP Scenarios

| Scenario | Choose CP | Choose AP |
| :--- | :--- | :--- |
| **Bank balance** | ✅ Wrong balance = legal nightmare | ❌ Showing stale balance = overdraft fraud |
| **Inventory count** | ✅ Overselling = shipping product you don't have | ❌ Showing "out of stock" when there's actually 1 left |
| **Social media feed** | ❌ Blocking users from seeing feed is worse than stale feed | ✅ "Oops, missed a post for 5 seconds" is fine |
| **DNS** | ❌ DNS downtime = entire internet breaks | ✅ Stale DNS record for 60 seconds is fine |
| **Shopping cart** | ❌ Cart being unavailable loses the sale | ✅ Temporarily stale cart merges later |

## 4. Consistency Models

### 4.1 Strong Consistency

Every read returns the **most recent write**. As if there's only one copy of the data.

```
Time 0: Client A writes X = 5 to Node 1
Time 1: Node 1 synchronously replicates to Node 2 and Node 3
Time 2: Client B reads X from Node 3 → gets 5 ✅ (always up-to-date)

Trade-off: SLOW writes (must wait for all replicas to confirm)
Used by: PostgreSQL (single-node), Google Spanner, CockroachDB
```

### 4.2 Eventual Consistency

If no new writes happen, **eventually** all replicas will converge to the same value. But reads MAY return stale data temporarily.

```
Time 0: Client A writes X = 5 to Node 1
Time 1: Client B reads X from Node 2 → gets OLD VALUE (3) ⚠️ (stale!)
Time 2: Node 1 asynchronously replicates to Node 2
Time 3: Client B reads X from Node 2 → gets 5 ✅ (now consistent)

Staleness window: Time 1 → Time 3 (usually milliseconds to seconds)

Trade-off: FAST writes, LOW latency, HIGH availability. But stale reads possible.
Used by: DynamoDB, Cassandra, DNS, S3
```

### 4.3 Causal Consistency

If Event A **causes** Event B, then everyone sees A before B. Unrelated events can be seen in any order.

```
Alice posts: "I got the job!" (Event A)
Bob replies: "Congratulations!" (Event B — caused by A)

Causal consistency guarantees:
  Everyone sees Alice's post BEFORE Bob's reply. ✅
  No one sees "Congratulations!" without seeing "I got the job!" first.

But two UNRELATED posts (Alice's post and Charlie's post) may appear 
in different orders for different users. That's fine.

Used by: MongoDB (with causal sessions)
```

### 4.4 Read-Your-Writes Consistency

After a client writes data, that **same client** is guaranteed to see the write on subsequent reads. Other clients may see stale data.

```
Client A writes: "name = Aashish"
Client A reads: → "Aashish" ✅ (always sees own write)
Client B reads: → might still see old name ⚠️ (eventually consistent for others)

Used by: Facebook (you see your own post immediately, friends see it seconds later)
```

### 4.5 Consistency Model Comparison

| Model | Staleness | Latency | Availability | Best For |
| :--- | :--- | :--- | :--- | :--- |
| **Strong** | None | High (sync replication) | Lower | Banking, inventory |
| **Eventual** | Seconds | Low (async replication) | Highest | Social media, DNS |
| **Causal** | None for related events | Medium | High | Chat, comments |
| **Read-Your-Writes** | None for the writing user | Low | High | User profiles, posts |

## 5. PACELC Theorem (The Extended CAP)

CAP only talks about behavior **during partitions**. PACELC extends it to ask: "What do you optimize for when there is **NO partition**?"

```
PACELC:
  IF (Partition) → Choose C or A  (same as CAP)
  ELSE           → Choose Latency or Consistency

  PA/EL: During partition → Available, Else → Low Latency
          (DynamoDB, Cassandra) — always optimizes for speed
  
  PC/EC: During partition → Consistent, Else → Consistent
          (PostgreSQL, Google Spanner) — always optimizes for correctness

  PA/EC: During partition → Available, Else → Consistent
          (MongoDB default) — available during failures, consistent otherwise
```

| :--- | :--- | :--- | :--- |
| **PostgreSQL** | CP (single-node: CA) | Strong | Transactions, ACID, banking |

| **MongoDB** | CP (default) or AP | Strong or Eventual (configurable) | Flexible documents, startups |
| **Cassandra** | AP | Eventual (tunable) | Write-heavy, time-series, IoT |

| **Google Spanner** | CP | Strong (globally!) | Global-scale transactions (Google-level) |
| **CockroachDB** | CP | Strong | Distributed SQL, PostgreSQL-compatible |
| **Elasticsearch** | AP | Near-real-time | Search, logs, analytics |

## 7. Interview Questions & Scenarios

### ❓ Q1: "Is CAP really 'pick 2 of 3'?"

**Answer:** "No, that's a simplification. In any real distributed system, **network partitions are inevitable** — so Partition Tolerance (P) is not optional. The real choice is **CP or AP** during a partition. When the network is healthy, you can have all three."

---

### ❓ Q2: "Your e-commerce site shows a product as 'In Stock' but when the user clicks Buy, it says 'Out of Stock'. What consistency model caused this?"

**Answer:** "This is **Eventual Consistency**. The product catalog is cached or served from a read replica that hasn't yet received the latest inventory update. The inventory DB shows 0 stock, but the cache/replica still shows 1.

**Fix options:**
1. Strong consistency for inventory checks at checkout (query the primary DB, not cache).
2. Optimistic UI: show 'In Stock' but validate at checkout time. If out of stock, apologize.
3. Reserve stock for 5 minutes when user adds to cart (Redis TTL)."

---

### ❓ Q3: "When would you choose Eventual Consistency over Strong Consistency?"

**Answer:** "When **availability and latency matter more than perfect accuracy**:
- Social media feed (seeing a post 2 seconds late is fine)
- Product reviews, likes, view counts (approximate is acceptable)
- DNS (stale record for 60 seconds is fine vs DNS being down entirely)
- Shopping cart (merge conflicts later, never block the user)

I'd use Strong Consistency for:
- Financial transactions (wrong balance = legal issue)
- Inventory at checkout (overselling = shipping air)
- User authentication (stale auth = security hole)"

---

### ❓ Q4: "How does DynamoDB offer both AP and CP?"

**Answer:** "DynamoDB lets you choose **per-read**:
- `ConsistentRead: false` (default) → **Eventual Consistency** (AP). Reads from any replica. Fast, potentially stale.
- `ConsistentRead: true` → **Strong Consistency** (CP). Reads from the leader node. Guaranteed latest. Slightly slower and more expensive.

This is the 'tunable consistency' approach — you pick the right model per query, not per database."
