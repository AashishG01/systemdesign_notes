# Part 1: Why Caching Exists

---

## 1. Why Caching Helps — The Core Problem

### 1.1 The Fundamental Truth About Computers

**Every performance problem in computing boils down to one thing: the speed gap between different storage layers.**

| Storage Layer | Speed | Cost per GB | Persistence |
| :--- | :--- | :--- | :--- |
| **CPU Register** | ~0.3 nanoseconds | Priceless | ❌ Volatile |
| **L1/L2/L3 CPU Cache** | 1-10 nanoseconds | ~$100 | ❌ Volatile |
| **RAM (Main Memory)** | ~100 nanoseconds | ~$5 | ❌ Volatile |
| **SSD (Solid State Drive)** | ~100 microseconds (100,000 ns) | ~$0.10 | ✅ Persistent |
| **HDD (Hard Disk Drive)** | ~10 milliseconds (10,000,000 ns) | ~$0.02 | ✅ Persistent |
| **Network (DB on another server)** | 1-100 milliseconds | Varies | ✅ Persistent |

> **💡 Key Insight:** RAM is roughly **100,000x faster** than a network round-trip to a database. Caching is the art of keeping frequently accessed data in fast storage (RAM) to avoid slow storage (disk/network).

### 1.2 The Problem Without Caching

Imagine Amazon's homepage. Every time a user visits:

```
User Request ──▶ App Server ──▶ PostgreSQL ──▶ Read from Disk ──▶ Return 50ms later
```

*   100,000 users visit simultaneously.
*   100,000 identical `SELECT * FROM featured_products` queries hit PostgreSQL.
*   Each query takes 50ms of disk I/O.
*   The DB has a maximum of ~5,000 connections.
*   **Result: 95,000 users get "Connection Refused" errors. The site crashes.**

### 1.3 The Solution With Caching

```
User #1 ──▶ App Server ──▶ Cache MISS ──▶ PostgreSQL (50ms) ──▶ Save to Redis
User #2 ──▶ App Server ──▶ Cache HIT ──▶ Redis (0.5ms) ──▶ Done ✅
User #3 ──▶ App Server ──▶ Cache HIT ──▶ Redis (0.5ms) ──▶ Done ✅
   ...
User #100,000 ──▶ App Server ──▶ Cache HIT ──▶ Redis (0.5ms) ──▶ Done ✅
```

*   **Only 1 query** hits the database.
*   **99,999 queries** are served from RAM in 0.5ms.
*   Database load drops by **99.999%**.
*   Page response time drops from 50ms to 0.5ms.

> **💡 The One-Line Summary:** Caching trades a tiny amount of RAM (cheap) for a massive reduction in database load and response time (expensive).

## 2. Where Caches Live — The Memory Hierarchy

Caching happens at **every layer** of a modern system, not just Redis.

```
┌───────────────────────────────────────────────────────────────────────┐
│                           USER'S BROWSER                             │
│  ┌─────────────────────────────────────────────────────────────────┐  │
│  │ Browser Cache: HTML, CSS, JS, images stored locally. Next      │  │
│  │ visit loads from disk in 0ms, no network request at all.       │  │
│  └─────────────────────────────────────────────────────────────────┘  │
└───────────────────────────────┬───────────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────────┐
│                              CDN                                      │
│  Edge servers worldwide cache static assets (images, videos, CSS).    │
│  User in Tokyo gets the image from a Tokyo server in 5ms instead      │
│  of crossing the Pacific to New York (200ms).                         │
└───────────────────────────────┬───────────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────────┐
│                        APPLICATION SERVER                             │
│  In-process memory cache (e.g., Node.js Map, Java HashMap).          │
│  Fastest possible cache (no network hop) but limited to the memory   │
│  of a single server and lost on restart.                              │
└───────────────────────────────┬───────────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────────┐
│                     DISTRIBUTED CACHE (Redis)                         │
│  Shared across ALL app servers. Lives on its own dedicated server.    │
│  Survives app server restarts. The most important caching layer       │
│  in System Design.                                                    │
└───────────────────────────────┬───────────────────────────────────────┘
                                │
                                ▼
┌───────────────────────────────────────────────────────────────────────┐
│                          DATABASE                                     │
│  PostgreSQL has its own internal buffer pool (shared_buffers) that    │
│  caches recently accessed disk pages in RAM. But this is limited     │
│  and not under your direct control.                                   │
└───────────────────────────────────────────────────────────────────────┘
```

## 3. Types of Caches in a System

| Cache Type | Where it Lives | Scope | Example |
| :--- | :--- | :--- | :--- |
| **Browser Cache** | User's machine | Single user | `Cache-Control: max-age=3600` in HTTP headers |
| **CDN Cache** | Edge servers globally | All users in a region | CloudFlare, AWS CloudFront caching images and static assets |
| **In-Process Cache** | Inside the app server's RAM | Single server instance | Node.js `Map`, Java `ConcurrentHashMap`, Python `dict` |
| **Distributed Cache** | Separate dedicated server(s) | ALL app servers share it | **Redis**, Memcached |
| **Database Buffer** | Inside the DB engine | Single DB instance | PostgreSQL `shared_buffers`, MySQL InnoDB Buffer Pool |

> **💡 Interview Insight:** When an interviewer says "add a cache," they almost always mean a **Distributed Cache (Redis)**. But mentioning CDN caching and browser caching for static assets shows advanced reasoning.

# Part 2: Caching Patterns & Strategies

---

## 4. Cache-Aside Pattern (Lazy Loading) ⭐

**The most common caching pattern in the industry.** If you learn only one, learn this one.

### 4.1 How it Works

The **application code** is responsible for talking to both the cache AND the database. The cache sits passively "aside" — it doesn't know the database exists.

```
┌──────────┐  1. GET user:123   ┌─────────┐
│   App    │ ──────────────────▶│  Redis  │
│  Server  │                    │  Cache  │
│          │◀── 2a. Cache HIT ──│         │  → Return data. Done. ✅
│          │    (data found!)   └─────────┘
│          │
│          │◀── 2b. Cache MISS ─┐
│          │    (data NOT found) │
│          │                     │
│          │  3. Query DB        │
│          │ ──────────────────▶┌──────────┐
│          │◀── 4. DB Response ─│ Postgres │
│          │                    └──────────┘
│          │
│          │  5. SET user:123   ┌─────────┐
│          │ ──────────────────▶│  Redis  │  → Populate cache for next time
│          │                    └─────────┘
│          │
│          │  6. Return data to client
└──────────┘
```

### 4.2 Pseudocode

```javascript
async function getUser(userId) {
  // Step 1: Check cache first
  const cached = await redis.get(`user:${userId}`);
  
  if (cached) {
    return JSON.parse(cached);  // Cache HIT — return instantly
  }
  
  // Step 2: Cache MISS — query the database
  const user = await db.query('SELECT * FROM users WHERE id = $1', [userId]);
  
  // Step 3: Populate cache for future requests (with 5 min TTL)
  await redis.set(`user:${userId}`, JSON.stringify(user), 'EX', 300);
  
  return user;
}
```

### 4.3 Pros and Cons

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| **Resilience:** If Redis crashes, the app falls back to the DB. It's slower but doesn't die. | **Cache Miss Penalty:** The first request for any data always hits the DB (cold start). |
| **Only caches what's needed:** Only data that is actually requested gets cached. No wasted memory on unused data. | **Stale Data Risk:** If the DB is updated directly (e.g., admin changes a price), the cache still serves the old value until TTL expires. |
| **Simple to implement:** Minimal application logic required. | **Thundering Herd:** If the cache expires and 10,000 users request the same data simultaneously, all 10,000 cache-miss and hit the DB at once. |

## 5. Write-Through Cache

### 5.1 How it Works

Writes go to the **cache AND the database simultaneously**. The cache is always perfectly up-to-date.

```
┌──────────┐  1. Write data   ┌─────────┐
│   App    │ ─────────────────▶│  Redis  │ ← Cache is updated FIRST
│  Server  │                   └─────────┘
│          │  2. Write data   ┌──────────┐
│          │ ─────────────────▶│ Postgres │ ← DB is updated SECOND
└──────────┘                   └──────────┘
```

### 5.2 Pros and Cons

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| **Cache is NEVER stale.** Every read from cache is guaranteed to be the latest data. | **Higher write latency.** Every write operation must wait for BOTH cache + DB to complete. |
| **No cache miss on reads** (if all writes go through this pattern). | **Caches data that may never be read.** If you cache every write but no one reads it, you're wasting RAM. |

> **Best For:** Systems where data consistency is *critical* and reads are extremely frequent — e.g., user session data, configuration values.

---

## 6. Write-Behind (Write-Back) Cache

### 6.1 How it Works

Writes go to the **cache only**. The cache asynchronously flushes changes to the database in the background, often in batches.

```
┌──────────┐  1. Write data    ┌─────────┐
│   App    │ ──────────────────▶│  Redis  │ ← Write completes here instantly
│  Server  │  (returns immediately)  │         │
└──────────┘                   └────┬────┘
                                    │
                            2. Async batch flush
                              (every 5 seconds)
                                    │
                                    ▼
                               ┌──────────┐
                               │ Postgres │ ← DB is updated LATER
                               └──────────┘
```

### 6.2 Pros and Cons

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| **Blazing fast writes.** The app only waits for the cache write (0.5ms), not the DB write (50ms). | **Data loss risk.** If Redis crashes before flushing to the DB, those writes are permanently lost. |
| **Reduces DB write load** by batching multiple writes into one. | **Complexity.** You need a reliable background flush mechanism. |

> **Best For:** High-write-throughput systems where losing a few seconds of data is acceptable — e.g., analytics event tracking, page view counters, sensor data ingestion.

---

## 7. Read-Through Cache

### 7.1 How it Works

Similar to Cache-Aside, but the **cache itself** is responsible for loading data from the DB on a miss. The application only talks to the cache, never directly to the DB.

```
┌──────────┐  1. GET user:123   ┌─────────┐
│   App    │ ──────────────────▶│  Cache  │
│  Server  │                    │ (Smart) │
│          │                    │         │──▶ 2. Cache MISS → Cache queries DB
│          │                    │         │◀── 3. DB returns data
│          │◀── 4. Return data ─│         │──▶ 5. Cache stores data internally
└──────────┘                    └─────────┘
```

**Key Difference from Cache-Aside:** In Cache-Aside, your application code handles the miss logic. In Read-Through, the cache library/proxy handles it transparently.

> **Best For:** When you want a clean separation of concerns — your application code never needs to know about the database. Common with cache proxies like **Varnish** or cache-integrated ORMs.

---

## 8. Pattern Comparison — When to Use What

| Pattern | Reads | Writes | Stale Data? | Data Loss Risk? | Use Case |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Cache-Aside** | App checks cache → DB on miss | App writes to DB only | ⚠️ Yes (until TTL) | ❌ None | **Most web apps.** The default choice. |
| **Write-Through** | Always from cache | App writes to cache + DB | ✅ Never stale | ❌ None | User sessions, configs, critical settings |
| **Write-Behind** | Always from cache | App writes to cache only, async flush to DB | ✅ Never stale | ⚠️ Yes (if cache crashes before flush) | Analytics, counters, high-write IoT |
| **Read-Through** | Cache loads from DB on miss | Varies | ⚠️ Yes (until TTL) | ❌ None | Clean architecture, cache proxies |

> **💡 Interview Answer:** "In most production systems, I use **Cache-Aside** for general data with a TTL of 60-300 seconds. For critical user session data, I use **Write-Through** to ensure the cache is never stale. For analytics counters like page views, I use **Write-Behind** to minimize DB load."

# Part 3: TTL & Cache Invalidation

## 9. TTL (Time-To-Live)

### 9.1 What is TTL?

**TTL** is the maximum lifespan of a cached entry. After the TTL expires, the cache automatically deletes the entry. The next request will be a cache miss, forcing a fresh read from the database.

```javascript
// Cache this product for 300 seconds (5 minutes)
await redis.set('product:456', JSON.stringify(product), 'EX', 300);

// After 300 seconds, Redis automatically deletes 'product:456'
// Next request → Cache MISS → Fresh data from DB
```

### 9.2 Choosing the Right TTL

| TTL Duration | When to Use | Example |
| :--- | :--- | :--- |
| **5-30 seconds** | Data changes very frequently and staleness is noticeable | Stock prices, live sports scores, ticket availability count |
| **1-5 minutes** | A good default for most web applications | Product pages, user profiles, search results |
| **1-24 hours** | Data rarely changes | Blog posts, static content, configuration values |
| **No TTL (∞)** | Data is immutable (never changes after creation) | Historical transactions, past order receipts, uploaded images metadata |

### 9.3 The TTL Balancing Act

```
Short TTL (5s)                              Long TTL (24h)
◄──────────────────────────────────────────────────────────────►
More DB Load                                Less DB Load
Fresher Data                                Staler Data
Lower Cache Hit Rate                        Higher Cache Hit Rate
```

> **💡 Rule of Thumb:** Start with a **5-minute TTL**. Monitor your cache hit rate. If it's below 80%, increase the TTL. If users are complaining about stale data, decrease it.

## 10. The Cache Invalidation Problem

> *"There are only two hard things in Computer Science: cache invalidation and naming things."* — Phil Karlton

### 10.1 Why is it Hard?

The moment data in the DB changes, the cached version becomes **stale** (outdated). The user sees old data until the cache is refreshed. The question is: *How do you keep the cache in sync with the database?*

### 10.2 The 3 Invalidation Strategies

#### Strategy 1: TTL-Based Expiration (Passive)
*   **How:** Set a TTL. Let stale data exist until it expires naturally.
*   **Staleness Window:** Up to the full TTL duration.
*   **Complexity:** Zero. Set it and forget it.
*   **Best For:** Data where a few minutes of staleness is acceptable (product catalog, blog posts).

```javascript
// Product is cached for 5 minutes
redis.set('product:456', data, 'EX', 300);
// If admin changes the price at T+1min, users see the old price for 4 more minutes
```

#### Strategy 2: Active Invalidation (Event-Driven)
*   **How:** When the DB is updated, your application code **immediately deletes** the cached entry.
*   **Staleness Window:** Near-zero (milliseconds).
*   **Complexity:** Medium. Every write path must know about the cache.
*   **Best For:** Data where freshness is important (inventory counts, user settings).

```javascript
async function updateProductPrice(productId, newPrice) {
  // Step 1: Update database
  await db.query('UPDATE products SET price = $1 WHERE id = $2', [newPrice, productId]);
  
  // Step 2: IMMEDIATELY invalidate the cache
  await redis.del(`product:${productId}`);
  
  // Next read will cache-miss and fetch the fresh price from DB
}
```

#### Strategy 3: Pub/Sub Invalidation (Distributed)
*   **How:** When the DB is updated, an event is published to a message channel. All cache instances subscribe and invalidate their local copy.
*   **Staleness Window:** Near-zero.
*   **Complexity:** High. Requires message infrastructure.
*   **Best For:** Multi-region deployments where multiple cache instances exist globally.

```
DB Updated ──▶ Publish "product:456:updated" to Redis Pub/Sub channel
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
  Cache (NY)   Cache (London) Cache (Tokyo)
  DEL product:456  DEL product:456  DEL product:456
```

### 10.3 Invalidation Strategy Comparison

| Strategy | Staleness | Complexity | DB Load | Best For |
| :--- | :--- | :--- | :--- | :--- |
| **TTL-Based** | Up to TTL duration | ✅ None | Low | Blog posts, catalogs, search results |
| **Active Invalidation** | Near-zero | Medium | Medium | Prices, user settings, inventory |
| **Pub/Sub** | Near-zero | High | Medium | Multi-region caches, real-time systems |

## 11. Cache Eviction Policies

When the cache runs out of memory (RAM is full), it must decide **which entries to throw away** to make room for new ones. This is the **eviction policy**.

### 11.1 Common Eviction Algorithms

| Policy | Full Name | How it Works | Best For |
| :--- | :--- | :--- | :--- |
| **LRU** | Least Recently Used | Evicts the entry that hasn't been accessed for the longest time. | ⭐ **The default choice for most systems.** If you haven't looked at it recently, you probably won't need it again. |
| **LFU** | Least Frequently Used | Evicts the entry that has been accessed the fewest total times. | Data with clear "hot" and "cold" items (e.g., celebrity profiles vs. inactive accounts). |
| **FIFO** | First In, First Out | Evicts the oldest entry, regardless of how often it's been accessed. | Simple queue-like caches where order of insertion matters. |
| **Random** | Random Eviction | Evicts a random entry. | Surprisingly effective in some workloads with no clear access pattern. |
| **TTL-Based** | Time-To-Live | Evicts entries that are closest to their TTL expiration. | When every entry has a meaningful TTL set. |

### 11.2 LRU Visualized (Most Common)

```
Cache Capacity: 4 items

Access: A → B → C → D → B → E

State after each access:
  A:  [A]
  B:  [A, B]
  C:  [A, B, C]
  D:  [A, B, C, D]     ← Cache is FULL
  B:  [A, C, D, B]     ← B accessed again, moves to "most recent" end
  E:  [C, D, B, E]     ← Cache full! Evict A (Least Recently Used). Add E.
```

> **💡 Redis uses an approximated LRU** by default (`maxmemory-policy allkeys-lru`). It samples a small random subset of keys and evicts the least recently used among the sample. This is much faster than true LRU while being practically equivalent.

## 12. Common Cache Failure Scenarios

These are the three catastrophic failure patterns every engineer must know.

### 12.1 Cache Stampede (Thundering Herd)

**What Happens:** A very popular cache key expires. 10,000 simultaneous requests all cache-miss and hit the database at the exact same moment, crashing it.

```
TTL expires for "homepage_data"
  │
  ├── Request 1 → Cache MISS → Query DB
  ├── Request 2 → Cache MISS → Query DB
  ├── Request 3 → Cache MISS → Query DB
  │   ... (10,000 simultaneous DB queries)
  └── Request 10,000 → Cache MISS → Query DB
  
  💥 DATABASE OVERLOADED. ALL QUERIES FAIL.
```

**Solution — Mutex Lock (Cache Locking):**
```javascript
async function getWithLock(key) {
  const cached = await redis.get(key);
  if (cached) return JSON.parse(cached);
  
  // Try to acquire a lock. Only ONE request can pass.
  const lockAcquired = await redis.set(`lock:${key}`, '1', 'NX', 'EX', 10);
  
  if (lockAcquired) {
    // I won the lock! I'll query the DB and populate the cache.
    const data = await db.query('SELECT ...');
    await redis.set(key, JSON.stringify(data), 'EX', 300);
    await redis.del(`lock:${key}`);
    return data;
  } else {
    // Someone else has the lock. Wait 100ms and retry from cache.
    await sleep(100);
    return getWithLock(key);  // Retry — cache should be populated by now
  }
}
```

### 12.2 Cache Penetration

**What Happens:** Users repeatedly request data that **doesn't exist** in the DB (e.g., `user_id = -999`). Every request cache-misses, queries the DB, finds nothing, and doesn't cache anything. The attacker keeps hammering the DB.

**Solution — Cache the NULL:**
```javascript
const data = await db.query('SELECT * FROM users WHERE id = $1', [-999]);

if (!data) {
  // Cache the fact that this ID doesn't exist (with short TTL)
  await redis.set('user:-999', 'NULL', 'EX', 60);
}
```

**Solution — Bloom Filter:** Use a probabilistic data structure that can tell you *definitively* "this key does NOT exist" without hitting the DB.

### 12.3 Cache Avalanche

**What happens:** A massive number of cache keys expire at the **exact same time**, causing a flood of simultaneous cache misses.

**Solution — Jittered TTL:**
```javascript
// Instead of: TTL = 300 (all expire at the same second)
// Use:        TTL = 300 + random(0, 60)  (spread expiration over 60 seconds)

const jitter = Math.floor(Math.random() * 60);
await redis.set(key, data, 'EX', 300 + jitter);
```

## 13. Redis Basics — What, Why, and How

### 13.1 What is Redis?

**Redis** (Remote Dictionary Server) is an open-source, **in-memory** data structure store. It stores everything in RAM, making it orders of magnitude faster than disk-based databases.

| Aspect | Redis | PostgreSQL |
| :--- | :--- | :--- |
| **Storage** | RAM (in-memory) | Disk (HDD/SSD) |
| **Read Speed** | ~0.1ms | ~5-50ms |
| **Data Model** | Key-Value + specialized structures | Relational tables |
| **Persistence** | Optional (RDB snapshots / AOF log) | Always persistent |
| **Best For** | Caching, sessions, real-time data | Permanent storage, transactions, complex queries |

### 13.2 Why Redis, Not Just a HashMap?

You might think: "Why not just use a JavaScript `Map` or Java `HashMap` inside my app server?"

| Feature | In-Process HashMap | Redis |
| :--- | :--- | :--- |
| **Shared across servers** | ❌ Server A's HashMap is invisible to Server B | ✅ All servers read/write to one shared Redis |
| **Survives app restart** | ❌ Destroyed when the process dies | ✅ Persists (optionally) via RDB/AOF |
| **Memory management** | ❌ No eviction. Can cause OOM crash | ✅ Built-in eviction policies (LRU, LFU) |
| **Advanced data structures** | ❌ Only key-value | ✅ Sorted Sets, Lists, HyperLogLog, Streams |
| **Atomic operations** | ❌ Race conditions in multi-threaded apps | ✅ Single-threaded — every command is atomic |

### 13.3 Core Redis Commands

```bash
# === String Operations ===
SET user:123 '{"name":"Alice","age":30}'       # Store a value
GET user:123                                    # Retrieve it
SET user:123 '...' EX 300                       # Store with 300 second TTL
DEL user:123                                    # Delete
TTL user:123                                    # Check remaining TTL (-1 = no TTL, -2 = expired)

# === Atomic Counters ===
SET page_views 0
INCR page_views                                 # Atomically increment to 1
INCR page_views                                 # Atomically increment to 2
INCRBY page_views 100                           # Atomically increment by 100 → 102

# === Key Management ===
EXISTS user:123                                 # Does this key exist? (1 = yes, 0 = no)
KEYS user:*                                     # Find all keys matching pattern (⚠️ NEVER in production — blocks Redis)
SCAN 0 MATCH user:* COUNT 100                   # Safe alternative to KEYS — iterates in batches
EXPIRE user:123 600                             # Set TTL to 600 seconds on existing key
```

## 14. Redis Data Structures

Redis is NOT just a key-value store. It has **6 specialized data structures**, each purpose-built for specific use cases.

### 14.1 Strings (The Default)

The simplest type. A key maps to a single string value (can be text, JSON, number, or binary data up to 512MB).

```bash
SET greeting "Hello, World!"
GET greeting  # → "Hello, World!"

# Use case: Caching JSON responses, session tokens, simple counters
```

### 14.2 Hashes (Mini-Objects)

A key maps to a **hash map of field-value pairs**. Like a row in a database table.

```bash
HSET user:123 name "Alice" age "30" email "alice@dev.com"
HGET user:123 name       # → "Alice"
HGETALL user:123         # → name:"Alice", age:"30", email:"alice@dev.com"
HINCRBY user:123 age 1   # Atomically increment age to 31
```

**Use Case:** Storing structured objects. More memory-efficient than storing a serialized JSON string because you can read/update individual fields without deserializing the entire object.

### 14.3 Lists (Ordered Sequences)

A key maps to a **doubly-linked list**. O(1) push/pop from head or tail.

```bash
LPUSH notifications:user:123 "New message from Bob"
LPUSH notifications:user:123 "Order #456 shipped"
LRANGE notifications:user:123 0 9       # Get the 10 most recent notifications
LLEN notifications:user:123              # Count of notifications
```

**Use Case:** Activity feeds, notification queues, recent items lists, chat message history.

### 14.4 Sets (Unique Collections)

A key maps to an **unordered collection of unique strings**. No duplicates allowed.

```bash
SADD online_users "alice" "bob" "carol"
SISMEMBER online_users "alice"           # → 1 (true)
SCARD online_users                       # → 3 (count)
SMEMBERS online_users                    # → {"alice", "bob", "carol"}

# Set Operations
SADD friends:alice "bob" "carol" "dave"
SADD friends:bob "alice" "carol" "eve"
SINTER friends:alice friends:bob         # → {"carol"} (mutual friends!)
```

**Use Case:** Online user tracking, tag management, mutual friends, deduplication.

### 14.5 Sorted Sets (Ranked Collections) ⭐

A key maps to a **set of unique members**, each associated with a **score** (floating-point number). Members are automatically sorted by score. The most powerful Redis data structure.

```bash
ZADD leaderboard 1500 "alice" 2300 "bob" 1800 "carol"
ZREVRANGE leaderboard 0 2 WITHSCORES    # Top 3 players (descending score)
# → bob:2300, carol:1800, alice:1500

ZINCRBY leaderboard 500 "alice"          # Alice earns 500 points → 2000
ZRANK leaderboard "alice"                # Alice's rank (0-indexed, ascending)
ZREVRANK leaderboard "bob"               # Bob's rank from the top (0 = #1)
```

**Use Case:** Gaming leaderboards, trending posts (score = engagement), priority queues, rate limiting (score = timestamp).

### 14.6 HyperLogLog (Approximated Counting)

A probabilistic data structure that estimates the count of **unique items** using only **12 KB of memory**, regardless of how many items you count (even billions).

```bash
PFADD unique_visitors:2024-03-01 "user_1" "user_2" "user_3"
PFADD unique_visitors:2024-03-01 "user_1" "user_4"  # user_1 is a duplicate, not counted
PFCOUNT unique_visitors:2024-03-01                    # → ~4 (0.81% standard error)
```

**Use Case:** Counting unique visitors per day, unique search queries, unique IP addresses — when exact precision isn't needed and memory must be minimal.

### 14.7 Data Structure Cheat Sheet

| Structure | When to Use | Real Example |
| :--- | :--- | :--- |
| **String** | Simple cache, counters, JSON blobs | Cache API responses, session tokens |
| **Hash** | Structured objects with individual field access | User profiles, product details |
| **List** | Ordered sequences, recent items | Notifications, chat history, activity feed |
| **Set** | Unique membership, intersection/union | Online users, tags, mutual friends |
| **Sorted Set** | Ranked data, priority queues | Leaderboards, trending items, rate limiting |
| **HyperLogLog** | Count unique items (billions) in 12KB | Daily unique visitors, distinct searches |

## 15. Redis in Production — Architecture Patterns

### 15.1 Single Redis Instance (Development)

```
  App Server ──────▶ Redis (Single Instance)
```
*   **Pros:** Simple.
*   **Cons:** SPOF. If Redis dies, all your cache is gone and every request hits the DB.

### 15.2 Redis Sentinel (High Availability)

```
                    ┌──────────────┐
  App Server ──────▶│ Redis Master │
                    └──────┬───────┘
                           │ Replication
                    ┌──────┴───────┐
                    ▼              ▼
            ┌─────────────┐ ┌─────────────┐
            │ Redis Replica│ │ Redis Replica│
            └─────────────┘ └─────────────┘
                    
            ┌───────────────────────────┐
            │  Sentinel Processes (3x)  │
            │  Monitor health. If Master│
            │  dies, promote a Replica. │
            └───────────────────────────┘
```

*   **How it works:** Sentinel processes continuously monitor the Master. If the Master goes down, Sentinel automatically promotes a Replica to the new Master within seconds.
*   **Best For:** Production apps that need high availability but don't need horizontal write scaling.

### 15.3 Redis Cluster (Massive Scale)

```
        ┌──────────────────────────────────────┐
        │         REDIS CLUSTER                │
        │  Data is SHARDED across nodes        │
        ├──────────┬──────────┬────────────────┤
        │ Shard 1  │ Shard 2  │ Shard 3        │
        │ Keys A-F │ Keys G-N │ Keys O-Z       │
        │ (Master) │ (Master) │ (Master)       │
        │    +     │    +     │    +            │
        │ Replica  │ Replica  │ Replica         │
        └──────────┴──────────┴────────────────┘
```

*   **How it works:** Data is automatically sharded across multiple Redis nodes using hash slots (16,384 total). Each node is responsible for a subset of hash slots.
*   **Best For:** Systems needing more than one server's worth of RAM for their cache (e.g., 500GB+ of cached data).

## 16. Real-World Caching Scenarios

### Scenario 1: E-Commerce Product Page (Flipkart/Amazon)

**Problem:** Millions of users browse the same products daily. Each page load requires JOINing `products`, `categories`, `reviews`, and `sellers` tables — a 50ms query.

**Solution:**
| Layer | Implementation |
| :--- | :--- |
| **Browser Cache** | `Cache-Control: max-age=60` for static assets (CSS, images, JS) |
| **CDN** | Product images and thumbnails served from CloudFront edge servers |
| **Redis Cache** | Full product JSON cached with **Cache-Aside** pattern, TTL = 5 minutes |
| **Active Invalidation** | When seller updates price → `redis.del('product:456')` → next read fetches fresh data |
| **Thundering Herd Protection** | Mutex lock on popular products to prevent stampede on cache expiry |

---

### Scenario 2: Real-Time Gaming Leaderboard

**Problem:** A mobile game with 10 million players needs a live leaderboard showing the top 100 players, updated in real-time.

**Solution:**
```bash
# Player scores a kill, increment their score
ZINCRBY global_leaderboard 10 "player:abc123"

# Get Top 100 leaderboard — O(log N + 100), instant even with 10M entries
ZREVRANGE global_leaderboard 0 99 WITHSCORES

# "What's my rank?" — O(log N)
ZREVRANK global_leaderboard "player:abc123"
```
*   **Why Redis?** A SQL `SELECT * FROM scores ORDER BY score DESC LIMIT 100` on 10M rows is painfully slow (>1 second). Redis Sorted Set returns the same result in **<1ms**.

---

### Scenario 3: Rate Limiting API Requests

**Problem:** An API allows max 100 requests per minute per user. Prevent abuse.

**Solution — Sliding Window Counter with Redis:**
```javascript
async function isRateLimited(userId) {
  const key = `rate_limit:${userId}`;
  const current = await redis.incr(key);
  
  if (current === 1) {
    // First request — set the 60-second window
    await redis.expire(key, 60);
  }
  
  return current > 100;  // true = BLOCKED, false = ALLOWED
}
```

*   **Why Redis?** The `INCR` command is **atomic** (no race conditions even with thousands of concurrent requests). The `EXPIRE` automatically resets the counter after 60 seconds.

---

### Scenario 4: User Session Management

**Problem:** After login, the user's session data (user ID, role, permissions, cart) must be accessible from any of 50 app servers.

**Solution:**
```javascript
// On login: Store session in Redis with 30-minute TTL
const sessionId = crypto.randomUUID();
await redis.hset(`session:${sessionId}`, {
  userId: '123',
  role: 'admin',
  cart: JSON.stringify([101, 205, 308])
});
await redis.expire(`session:${sessionId}`, 1800); // 30 minutes

// On every subsequent request: Any server reads the session
const session = await redis.hgetall(`session:${sessionId}`);
```

*   **Why Redis?** Using Redis Hashes, any of 50 stateless app servers can instantly access the same session without sticky sessions or server-local storage.

## 17. Interview Questions & Problems with Solutions

### ❓ Q1: "Your cache has a 99% hit rate, but users complain about seeing stale prices after an admin updates a product. How do you fix it?"

**Answer:**
"We're currently using **TTL-Based expiration** (passive invalidation) with a 5-minute TTL. When an admin updates a price, the old price is served for up to 5 minutes.

**Fix:** Implement **Active Invalidation**. In the `updateProductPrice()` function, after writing to the DB, immediately call `redis.del('product:456')`. The next user request will cache-miss, fetch the fresh price from the DB, and repopulate the cache. Staleness drops from 5 minutes to milliseconds."

---

### ❓ Q2: "We use Redis as our cache. It just crashed. What happens?"

**Answer:**
"It depends on the caching pattern:
*   **Cache-Aside:** The app gracefully degrades. Every request becomes a cache miss and hits the DB directly. The system is **slower but still functional**. Once Redis recovers, it warms up naturally.
*   **Write-Behind:** This is **dangerous**. Any writes that were in Redis's memory but not yet flushed to the DB are **permanently lost**. This is why Write-Behind should only be used for non-critical data (analytics, counters).

**Prevention:** Use **Redis Sentinel** for automatic failover. If the master dies, a replica is promoted within seconds."

---

### ❓ Q3: "Design a cache for a system where the same key is requested 100,000 times per second and the TTL is 60 seconds. What's the biggest risk?"

**Answer:**
"The biggest risk is a **Cache Stampede (Thundering Herd)**. At the exact moment the TTL expires, all 100,000 concurrent requests simultaneously cache-miss and hammer the database.

**Solution:**
1.  **Mutex Lock:** Only one request is allowed to recompute the cache. Others wait and retry.
2.  **Early Refresh:** Refresh the cache at 80% of TTL (at 48 seconds) proactively, before it actually expires.
3.  **Jittered TTL:** `TTL = 60 + random(0, 10)` to stagger expirations and prevent synchronized misses."

---

### ❓ Q4: "When would you NOT use a cache?"

**Answer:**
1.  **Write-heavy, read-light workloads:** If data is written 1000x more often than it's read, the cache is constantly invalidated and provides no benefit. You're wasting RAM.
2.  **Highly dynamic, unique data:** If every request fetches unique, non-repeating data (e.g., real-time stock tick by tick), caching is useless because nothing is ever requested twice.
3.  **When strong consistency is non-negotiable:** Banking transactions must read the absolute latest balance from the DB. Even a 1-second stale cache could cause an overdraft. Use the DB directly.
4.  **Small datasets that fit in DB buffer:** If the entire table fits in the DB's own memory buffer pool (PostgreSQL `shared_buffers`), adding an external cache adds complexity without measurable speedup."

---

### ❓ Q5: "How would you choose between Redis and Memcached?"

**Answer:**

| Feature | Redis | Memcached |
| :--- | :--- | :--- |
| **Data Structures** | Strings, Hashes, Lists, Sets, Sorted Sets, Streams, HyperLogLog | Strings ONLY |
| **Persistence** | Optional (RDB + AOF) | ❌ None |
| **Replication** | ✅ Master-Replica, Sentinel, Cluster | ❌ None |
| **Threading** | Single-threaded (atomic by default) | Multi-threaded (faster for simple GET/SET at scale) |
| **Max Value Size** | 512 MB | 1 MB |
| **Use Case** | Feature-rich caching, leaderboards, sessions, pub/sub, rate limiting | Pure simple key-value caching at extreme throughput |

"I'd use **Redis** in 95% of cases because the rich data structures (Sorted Sets for leaderboards, Hashes for sessions, Pub/Sub for real-time events) provide far more value than Memcached's marginal multi-threaded performance advantage."
