## 1. What is an API Gateway?

### 1.1 The Definition

An **API Gateway** is a single entry point for all client requests into a microservices architecture. It sits between external clients and your internal services, handling **cross-cutting concerns** вҖ” authentication, rate limiting, routing, logging вҖ” so that individual microservices don't have to.

> **рҹ’Ў Intuition вҖ” The Hotel Concierge:**
> You walk into a 5-star hotel. You don't wander the hallways looking for the restaurant, the spa, and the laundry yourself.
> *   **Without an API Gateway:** You must know the exact door number (IP + port) of every service. Need food? Go to Room 412. Need clean clothes? Go to Room 107. Need a taxi? Call Extension 3321. If any room changes, you're lost.
> *   **With an API Gateway (The Concierge):** You walk to ONE desk. "I'd like dinner, laundry, and a taxi." The concierge checks your room key (authentication), verifies you haven't exceeded your daily service limit (rate limiting), and dispatches your requests to the correct departments (routing). You never need to know Room 412 exists.

## 2. API Gateway vs Load Balancer вҖ” The Critical Difference

This is one of the most frequently confused concepts in System Design interviews.

| Feature | Load Balancer | API Gateway |
| :--- | :--- | :--- |
| **Primary Job** | Distribute traffic evenly across identical servers | Route requests to the correct service + handle cross-cutting concerns |
| **Backend Servers** | All servers run the **same** application | Each backend is a **different** microservice |
| **Intelligence** | "Send this request to one of 10 identical User Service instances" | "This is a `/payments` request вҖ” route it to the Payment Service, not the User Service" |
| **Auth** | вқҢ No | вң… Validates JWT, API keys, OAuth tokens |
| **Rate Limiting** | вқҢ No (basic at best) | вң… Per-user, per-API-key, per-endpoint throttling |
| **Request Transformation** | вқҢ No | вң… Modify headers, rewrite URLs, merge responses |
| **Protocol Translation** | вқҢ No | вң… Accept REST from client, forward as gRPC to backend |
| **API Composition** | вқҢ No | вң… Aggregate responses from 3 services into 1 response |

## 4. Request Routing

The most fundamental job of the API Gateway: **look at the incoming request and decide which backend microservice should handle it.**

### 4.1 Path-Based Routing

```
Rule: Route based on the URL path

GET  /api/users/123      вҶ’ UserService
POST /api/orders          вҶ’ OrderService
POST /api/payments/charge вҶ’ PaymentService
GET  /api/products?q=shoe вҶ’ ProductService
```

### 4.2 Header-Based Routing

```
Rule: Route based on HTTP headers

Header "X-Client-Type: mobile"  вҶ’ MobileOptimizedService (lighter JSON responses)
Header "X-Client-Type: web"     вҶ’ WebService (full HTML + JSON)
Header "X-Api-Version: v2"      вҶ’ ServiceV2Pool
```

### 4.3 Method-Based Routing

```
Rule: Route based on HTTP method

GET  /api/orders   вҶ’ OrderReadService  (optimized for fast reads, read replicas)
POST /api/orders   вҶ’ OrderWriteService (optimized for writes, master DB)
```

### 4.4 Kong (API Gateway) Routing Configuration Example

```yaml
# Kong declarative config
services:
  - name: user-service
    url: http://user-service:3001
    routes:
      - name: user-route
        paths:
          - /api/users
        methods:
          - GET
          - POST

  - name: payment-service
    url: http://payment-service:3003
    routes:
      - name: payment-route
        paths:
          - /api/payments
        methods:
          - POST
```

## 5. Authentication & Authorization

### 5.1 Why Authenticate at the Gateway?

Without a gateway, **every single microservice** must independently validate the JWT token, check API keys, and verify permissions. That's duplicated code in 20+ services, and a single bug in one service's auth logic = a security hole.

With a gateway, auth happens **once** at the edge. Backend services trust that if a request arrived, it's already been authenticated.

### 5.3 Authentication Methods the Gateway Handles

| Method | How it Works | Use Case |
| :--- | :--- | :--- |
| **JWT (JSON Web Token)** | Client sends `Bearer <token>` in the Authorization header. Gateway verifies the cryptographic signature without contacting any other service. | Mobile apps, SPAs, user-facing APIs |
| **API Key** | Client sends `X-API-Key: abc123` in the header. Gateway looks up the key in a database/cache. | Third-party integrations, partner APIs, public APIs |
| **OAuth 2.0** | Gateway acts as the OAuth Resource Server. It validates access tokens against the Authorization Server. | Delegated access (e.g., "Sign in with Google") |
| **mTLS (Mutual TLS)** | Both client and server present certificates. Gateway verifies the client certificate. | Service-to-service communication, banking, high-security APIs |

## 6. Rate Limiting & Throttling

### 6.1 Why Rate Limit?

Without rate limiting:
*   A malicious bot sends 1 million requests/second to your API вҶ’ servers crash (DDoS).
*   A buggy client enters an infinite loop вҶ’ floods your API with duplicate requests.
*   A free-tier user consumes the same resources as a paying enterprise customer вҶ’ unfair.

### 6.2 Rate Limiting Strategies

| Strategy | How it Works | Example |
| :--- | :--- | :--- |
| **Fixed Window** | Count requests in fixed time windows (e.g., per minute). Reset at the start of each window. | "Max 100 requests per minute. At :00, counter resets to 0." |
| **Sliding Window** | Like Fixed Window but the window slides with each request, preventing bursts at window boundaries. | Smoother throttling, more accurate. |
| **Token Bucket** | A "bucket" holds N tokens. Each request costs 1 token. Tokens refill at a steady rate. When the bucket is empty, requests are rejected. | "Bucket holds 100 tokens, refills at 10 tokens/sec. Allows short bursts up to 100 but sustained rate is 10/sec." |
| **Leaky Bucket** | Requests enter a queue (bucket) that processes them at a fixed rate. If the queue is full, new requests are dropped. | Guarantees a constant, smooth output rate. |

### 6.3 Rate Limiting Granularity

| Level | How | Example |
| :--- | :--- | :--- |
| **Per IP** | Limit based on client IP address | `rate_limit: 100 req/min per IP` вҖ” stops DDoS |
| **Per User** | Limit based on authenticated user ID (from JWT) | `rate_limit: 1000 req/min per user` вҖ” stops buggy clients |
| **Per API Key** | Limit based on API key tier | Free tier: 100/min, Pro: 10,000/min, Enterprise: unlimited |
| **Per Endpoint** | Different limits for different endpoints | `GET /search`: 60/min. `POST /upload`: 5/min |

## 7. Request/Response Transformation

The API Gateway can **modify requests before forwarding** and **modify responses before returning** to the client.

### 7.1 Common Transformations

| Transformation | Example |
| :--- | :--- |
| **Header Injection** | Add `X-User-Id: 123` and `X-Request-Id: uuid` to every request forwarded to backend services |
| **Header Removal** | Strip sensitive internal headers like `X-Internal-Debug` from responses before sending to clients |
| **URL Rewriting** | Client requests `/api/v1/users/123`. Gateway rewrites to `/users/123` before forwarding (backend doesn't know about versioning) |
| **Protocol Translation** | Client sends REST (JSON over HTTP). Gateway translates to **gRPC** (Protocol Buffers over HTTP/2) for internal service communication |
| **Response Filtering** | Backend returns 50 fields. Mobile client only needs 5. Gateway strips unnecessary fields to reduce payload size |

```
     в”Ӯв—Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ӯ                           в”Ӯ
```

> **рҹ’Ў Why gRPC internally?** gRPC uses Protocol Buffers (binary serialization) instead of JSON (text). It's **2-10x faster** for inter-service communication. But mobile/web clients only speak REST/JSON, so the Gateway translates.

## 8. API Composition (Aggregation)

### 8.1 The Problem

A single page on your app needs data from **3 different microservices**. Without a Gateway, the client must make 3 separate API calls:

```
Mobile App в”Җв”Җв–¶ GET /api/users/123       вҶ’ User Service      (50ms)
Mobile App в”Җв”Җв–¶ GET /api/orders?user=123 вҶ’ Order Service     (80ms)
Mobile App в”Җв”Җв–¶ GET /api/wallet/123      вҶ’ Payment Service   (30ms)

Total: 3 round trips Г— network latency = SLOW on mobile (especially on 3G)
```

### 8.2 The Solution: Gateway Aggregation

The Gateway makes all 3 calls **internally in parallel** and returns a single merged response:

```
Mobile App в”Җв”Җв–¶ GET /api/dashboard/123   вҶ’ API Gateway

                API Gateway (internally):
                в”ңв”Җв”Җ GET /users/123     вҶ’ UserService      (50ms) в”Җв”җ
                в”ңв”Җв”Җ GET /orders?u=123  вҶ’ OrderService     (80ms) в”Җв”јв”Җв”Җ All 3 in parallel
                в””в”Җв”Җ GET /wallet/123    вҶ’ PaymentService   (30ms) в”Җв”ҳ
                                                                   в”Ӯ
                Merge responses:                                   в”Ӯ
                { user: {...}, orders: [...], wallet: {...} }  в—Җв”Җв”Җв”Җв”ҳ

Mobile App в—Җв”Җв”Җ 200 OK (single response, 80ms total instead of 160ms)
```

**Benefits:**
1.  Client makes **1 request** instead of 3. Huge win on slow mobile networks.
2.  Internal calls happen over the **fast datacenter network**, not the slow internet.
3.  The mobile app's code is simpler вҖ” one API call, one response.

> **вҡ пёҸ Danger:** Don't over-aggregate. If the Gateway becomes responsible for complex business logic (merging, filtering, transforming data from 10 services), it becomes a monolithic bottleneck. Keep aggregation simple.

## 9. Circuit Breaker Pattern

### 9.1 The Problem

Your API Gateway routes to the Payment Service. The Payment Service's database is down. Every request to Payment Service hangs for 30 seconds, then times out.

**Without a Circuit Breaker:** The Gateway keeps sending requests. Each one blocks for 30 seconds. Thread pool fills up. Soon, the Gateway can't handle requests for ANY service вҖ” User Service and Order Service are also affected even though they're perfectly healthy. **One dead service kills everything.**

### 9.2 How the Circuit Breaker Works

It's inspired by electrical circuit breakers that prevent a short circuit from burning down the house.

```
                    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
                    в”Ӯ          CIRCUIT BREAKER             в”Ӯ
                    в”Ӯ                                     в”Ӯ
                    в”Ӯ  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ    в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”җ
                    в”Ӯ  в”Ӯ CLOSED в”Ӯв”Җв”Җв”Җв–¶в”Ӯ  OPEN  в”Ӯв”Җв”Җв”Җв–¶в”ӮHALF-OPENв”Ӯ
                    в”Ӯ  в”Ӯ(normal)в”Ӯ    в”Ӯ(reject)в”Ӯ    в”Ӯ (test)  в”Ӯ
                    в”Ӯ  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ    в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ    в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                    в”Ӯ       в–І                          в”Ӯ
                    в”Ӯ       в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
                    в”Ӯ         (if test succeeds)
                    в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### 9.3 The 3 States

| State | Behavior | Transition |
| :--- | :--- | :--- |
| **CLOSED** (Normal) | All requests pass through to the backend service normally. The breaker counts failures. | If failures exceed threshold (e.g., 5 failures in 10 seconds) вҶ’ switch to **OPEN** |
| **OPEN** (Rejecting) | ALL requests are **immediately rejected** with HTTP 503 (Service Unavailable). Zero requests reach the backend. Instant response, no 30-second hang. | After a timeout period (e.g., 30 seconds) вҶ’ switch to **HALF-OPEN** |
| **HALF-OPEN** (Testing) | Allow **one test request** through to the backend. If it succeeds, the service has recovered. If it fails, it's still broken. | Success вҶ’ **CLOSED** (resume normal traffic). Failure вҶ’ **OPEN** (keep rejecting) |

### 9.4 Real-World Example

```
Time 0:00 - Circuit CLOSED. PaymentService responds normally. вң…
Time 0:15 - PaymentService DB crashes. Requests start failing. вқҢ
Time 0:20 - 5th consecutive failure. Circuit trips to OPEN. вҡЎ
Time 0:21 - All requests instantly return 503. No 30-second hangs. вң… (Gateway is protected)
Time 0:50 - 30 seconds passed. Circuit goes HALF-OPEN. рҹҹЎ
Time 0:51 - One test request sent to PaymentService. Still failing. вқҢ
Time 0:51 - Circuit goes back to OPEN. рҹ”ҙ
Time 1:20 - Another test. PaymentService DB recovered! вң…
Time 1:21 - Circuit goes CLOSED. Normal traffic resumes. вң…
```

> **рҹ’Ў Interview Tip:** The Circuit Breaker is part of the **Resilience Pattern** family. Pair it with **Retry with Exponential Backoff** (retry failed requests with increasing delays: 1s, 2s, 4s, 8s) and **Fallbacks** (return cached data or a default response when the circuit is open).

## 10. API Versioning

### 10.1 Why Version?

You have 500 partner companies using your `POST /api/orders` endpoint. You need to change the request format. If you just change it, all 500 partners' integrations break instantly.

**Solution:** Deploy the new version alongside the old one. Partners migrate on their own schedule.

### 10.2 Versioning Strategies (at the Gateway)

| Strategy | How the Client Specifies Version | Gateway Routing |
| :--- | :--- | :--- |
| **URL Path** | `GET /api/v1/users` vs `GET /api/v2/users` | `/api/v1/*` вҶ’ ServiceV1 pool. `/api/v2/*` вҶ’ ServiceV2 pool. |
| **Header** | `Accept-Version: v2` or `X-Api-Version: 2` | Route based on header value. Cleaner URLs. |
| **Query Parameter** | `GET /api/users?version=2` | Route based on query param. Simple but ugly. |

```
API Gateway Routing:

  /api/v1/users вҶ’ User Service v1 (legacy, deprecated)
  /api/v2/users вҶ’ User Service v2 (current)
  /api/v3/users вҶ’ User Service v3 (beta, internal only)
```

> **рҹ’Ў Industry Standard:** Use **URL Path versioning** (`/v1/`, `/v2/`). It's the most visible, debuggable, and widely used approach (used by Stripe, GitHub, Twitter APIs).

## 11. Canary Deployments & A/B Testing

### 11.1 Canary Deployments

**What it is:** Deploy a new version of a service to only **5% of traffic**. Monitor for errors. If clean, gradually increase to 100%.

```
API Gateway Routing:

  /api/users вҶ’ 95% traffic вҶ’ User Service v2.0 (stable)
  /api/users вҶ’  5% traffic вҶ’ User Service v2.1 (canary)

  If canary shows 0% error rate after 1 hour:
    Increase to 25%, then 50%, then 100%.
  If canary shows errors:
    Instantly route 100% back to v2.0. Zero user impact.
```

### 11.2 A/B Testing

**What it is:** Route different user segments to different service versions to test **which performs better** (higher conversion, more clicks).

```
API Gateway Rule:

  If user.country == "IN"  вҶ’ Route to Checkout Service (вӮ№ layout)
  If user.country == "US"  вҶ’ Route to Checkout Service ($ layout)
  If cookie "experiment_new_ui" == true вҶ’ Route to Checkout Service v3 (new design)
```

## 12. Logging, Monitoring & Observability

The API Gateway is the **perfect place** to instrument observability because ALL traffic flows through it.

### 12.1 What the Gateway Logs

| Data Point | Why | Example |
| :--- | :--- | :--- |

| Metric | Alert Threshold | Meaning |
| :--- | :--- | :--- |
| **Error Rate (5xx)** | > 1% | Backend services are failing |

| **Rate Limit Rejections** | > 100/min | Clients are being throttled вҖ” either legitimate growth or abuse |
| **Circuit Breaker Opens** | Any | A downstream service is down |

## 13. Cross-Cutting Concerns (CORS, Compression, Caching)

### 13.1 CORS (Cross-Origin Resource Sharing)

**Problem:** Browser security blocks your React app (`app.mysite.com`) from calling your API (`api.mysite.com`) because they're different origins.

**Solution:** The Gateway adds CORS headers to every response:

```http
Access-Control-Allow-Origin: https://app.mysite.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Max-Age: 86400
```

Handle CORS at the Gateway, not in every microservice.

### 13.2 Response Compression

The Gateway compresses responses with **gzip** or **Brotli** before sending to the client. A 500KB JSON response becomes 50KB. Massive bandwidth savings, especially for mobile users.

### 13.3 Response Caching

For read-heavy, rarely-changing endpoints (e.g., product catalog), the Gateway can cache the response:

```
First request:   Client вҶ’ Gateway вҶ’ Product Service (50ms) вҶ’ Cache response
Second request:  Client вҶ’ Gateway вҶ’ Return cached response (0.5ms) вҶҗ Service never contacted
```

## 16. Interview Questions & Problems with Solutions

### вқ“ Q1: "Do you need an API Gateway for a monolith?"

**Answer:**
"Technically no вҖ” a monolith is a single service, so there's no routing needed. But even with a monolith, placing **NGINX** in front as a reverse proxy gives you:
1.  SSL termination
2.  Rate limiting
3.  Static file serving
4.  Response compression
5.  A clean migration path вҖ” when you eventually split into microservices, NGINX is already in place.

So while it's not a full 'API Gateway,' using a reverse proxy in front of a monolith is industry standard and recommended."

---

### вқ“ Q2: "How is authentication different at the Gateway vs at the service?"

**Answer:**

| Layer | Responsibility | Example |
| :--- | :--- | :--- |
| **Gateway (AuthN)** | **Authentication** вҖ” *WHO are you?* Validate the JWT signature, check token expiration, extract `user_id`. | "This is User #123 and their token is valid." |
| **Service (AuthZ)** | **Authorization** вҖ” *WHAT can you do?* Check if User #123 has permission to access this specific resource. | "User #123 is an 'editor', so they can edit posts but cannot delete users." |

"The Gateway verifies identity centrally (AuthN). The service enforces fine-grained permissions locally (AuthZ), because the Gateway doesn't know each service's domain-specific permission rules."

---

### вқ“ Q3: "What happens if the API Gateway becomes a bottleneck?"

**Answer:**
"The Gateway is a potential SPOF and bottleneck because ALL traffic passes through it. Mitigations:
1.  **Horizontal Scaling:** Deploy multiple Gateway instances behind a Layer 4 Load Balancer (NLB).
2.  **Stateless Design:** The Gateway should be stateless вҖ” no in-memory sessions. Rate limit counters live in Redis, not the Gateway's memory.
3.  **Minimize Business Logic:** The Gateway should ONLY handle cross-cutting concerns (auth, routing, rate limiting). NEVER put business logic here.
4.  **Caching at the Edge:** Use CDN caching to prevent read-heavy requests from even reaching the Gateway."

---

### вқ“ Q4: "When would you NOT use an API Gateway?"

**Answer:**
1.  **Simple monolith with < 3 services:** Adding a Gateway to a 2-service system adds complexity with no benefit. NGINX as a reverse proxy is sufficient.
2.  **Internal service-to-service communication:** Services talking to each other should NOT route through the public API Gateway. Use an internal service mesh (Envoy sidecar) or direct service discovery (Consul/Kubernetes DNS) instead.
3.  **Ultra-low-latency systems:** Every additional network hop (Client вҶ’ Gateway вҶ’ Service) adds latency. For sub-1ms requirements, eliminate the Gateway and have clients connect directly."

---

### вқ“ Q5: "Your Payment Service is down. Users clicking 'Pay' wait 30 seconds, then get a timeout error. How do you fix this?"

**Answer:**
"Three layers of defense at the API Gateway:
1.  **Timeout:** Set a 5-second timeout on the Gateway вҶ’ Payment Service connection. Don't let users wait 30 seconds.
2.  **Circuit Breaker:** After 5 consecutive timeouts, trip the circuit to OPEN. Immediately return `503 Service Unavailable` with a friendly message instead of making users wait at all.
3.  **Fallback:** When the circuit is open, return a **graceful fallback response**: `{ 'status': 'payment_queued', 'message': 'Your payment is being processed. We will notify you.' }` Drop the payment request into a **Message Queue** and process it when the service recovers."

---

### вқ“ Q6: "Explain the Backend-for-Frontend (BFF) pattern."

**Answer:**
"Instead of one generic Gateway for ALL clients, you create **dedicated Gateway instances per client type**:

```
  Mobile App в”Җв”Җв–¶ Mobile BFF Gateway в”Җв”Җв–¶ Microservices
                 (returns lightweight JSON,
                  aggregates fewer services,
                  mobile-optimized)

  Web App    в”Җв”Җв–¶ Web BFF Gateway    в”Җв”Җв–¶ Microservices
                 (returns full data,
                  more fields, HTML-friendly)

  Partner API в”Җв”Җв–¶ Partner BFF Gateway в”Җв”Җв–¶ Microservices
                  (strict rate limits,
                   API key auth only,
                   minimal data exposure)
```

**Why?** A mobile app needs 5 fields. The web app needs 50. A generic Gateway forces ugly compromises. Separate BFFs let each client get exactly the API shape it needs. Used by Netflix, SoundCloud, and Spotify."
