## 1. What is System Design & Why Does It Matter?

### 1.1 What is System Design?

**System Design** is the process of defining the **architecture, components, modules, interfaces, and data flow** for a system to satisfy specified business and technical requirements.

> Think of it like building a city — you don't just start pouring concrete. You need zoning laws (architecture), roads (data flow), water/power plants (processing), and bridges (interfaces).

| Real-World Analogy | System Component | Why |
|---|---|---|
| Traffic Cop | **Load Balancer** | Distributes traffic evenly across servers |
| High-Speed Toll Lane | **Cache** | Speeds up access to frequent data |
| Filing Cabinets | **Database** | Persistent storage of structured data |
| Post Office | **Message Queue** | Asynchronous delivery of tasks |
| Local Warehouse | **CDN** | Store static files closer to users |

### 1.2 Why is System Design Required? (The Real Reasons)

#### For Interviews
- **Every major tech company** (Google, Amazon, Meta, Netflix) uses it to level candidates (Mid vs Senior vs Staff).
- It proves you can **design for scale**, not just write algorithmic code.
- It tests your ability to **manage trade-offs** (e.g., consistency vs latency).

#### For Real-World Software Development
| Without System Design Knowledge | With System Design Knowledge |
|---|---|
| Single server crashes → App goes offline | Load balancer + auto-scaling → Zero downtime |
| DB query takes 5 seconds under load | Redis cache + indexing → 5 milliseconds |
| Large video upload blocks user UI | Async Message Queue → User gets instant "processing" feedback |
| Users in Asia wait 3s for images from US servers | CDN integration → 50ms load time globally |

### 1.3 System Design in the Real World — Where It Hides

System Design is not just an interview topic; it is deeply embedded in everyday **Software Engineering workflows**. When building a new feature or product, teams follow the System Design lifecycle:

1. **Requirements Clarification:** Defining exactly what the business needs (e.g., "Users must be able to upload profile pictures").
2. **Capacity Planning (Estimations):** Calculating required storage and bandwidth. ("If 100M users upload a 2MB photo, we need 200TB of storage").
3. **High-Level Design (HLD):** Drawing the macro-architecture. (Mobile App → Load Balancer → API Server → Object Storage).
4. **Database Design:** Choosing the right data store (e.g., storing the image file in AWS S3, and the image metadata/URL in PostgreSQL).
5. **API Interface Design:** Defining the contract between the frontend and backend (e.g., `POST /v1/users/{user_id}/profile-picture`).
6. **Low-Level Design (LLD):** Writing the actual code structure, classes, and algorithms inside the API Server.

#### Real-World Scenarios Map

| Product Scenario | The Underlying System Design Solution |
|---|---|
| A viral tweet drops and 1 million people view it per second | **Caching (Redis)** — The tweet is pre-loaded in memory so the database isn't queried at all. |
| You add an item to an Amazon shopping cart on Black Friday | **Eventual Consistency (AP System)** — Amazon prioritizes availability over strict consistency to ensure the "Add to Cart" button never fails. |
| Netflix starts playing a 4K movie instantly | **CDNs (Content Delivery Networks)** — The movie file is cached on a custom server located geographically inside your local ISP. |
| You order an Uber and see cars moving on the map | **WebSockets + Geospatial Databases** — Continuous bi-directional connection filtering cars using Geohashes/QuadTrees. |
| You upload a massive 4K video to YouTube | **Message Queues (Kafka) + Worker Nodes** — Heavy compression work is pushed to the background so your UI doesn't freeze. |

> **Bottom line:** Every time an app feels "instant," "reliable," or "magical" at massive scale, you are experiencing good System Design.

### 2.2 The Problem-Solving Mindset for System Design

> **"There is no perfect architecture, only trade-offs."**

1. **Ask clarifying questions first.** Never assume you know the exact requirements.
2. **Focus on bottlenecks.** Don't try to build the final Netflix architecture day 1. Build a simple server, then ask: "What breaks when it hits 1 million users?"
3. **Master the estimations.** Be comfortable doing back-of-the-envelope math (RPS, storage per day, bandwidth).
4. **Know the trade-offs intimately.** Don't just mention a technology (e.g., Kafka). Know *why* you chose it over alternatives (e.g., RabbitMQ or SQS) and what you sacrifice by doing so.

## 3. Core Operational Metrics — Complete Guide

### 3.1 Scalability
Scalability is the ability to handle increased load without performance degradation.

| Type | How it works | Pros | Cons | Example |
|---|---|---|---|---|
| **Vertical (Scale Up)** | Buy a bigger server (CPU/RAM) | Very simple, no code changes | Hardware limit, Single Point of Failure | Upgrading AWS EC2 from t3.micro to m5.8xlarge |
| **Horizontal (Scale Out)** | Add more servers to a pool | Infinitely scalable, fault-tolerant | High complexity, requires Load Balancing | Adding 50 standard servers behind an ALB |

### 2.2 Latency vs. Throughput

- **Latency:** Time taken for a *single* request to be fulfilled (measured in milliseconds: ms). *(Analogy: How fast water travels through a pipe).*
- **Throughput:** Number of requests a system handles concurrently (measured in Requests Per Second: RPS). *(Analogy: How wide the pipe is).*

### 2.3 Availability vs. Reliability

- **Availability:** Percentage of time the system is operational.
  - 99.9% ("Three Nines") = ~8.7 hours downtime / year
  - 99.999% ("Five Nines") = ~5.26 minutes downtime / year
- **Reliability:** Probability that the system produces the *correct* output over a given time. (A system can be highly available but unreliable if it always returns errors).

## 4. The CAP Theorem & System Trade-offs

### 3.1 The Three Guarantees

In any distributed data store, you can only pick **TWO** of the following three:

1. **Consistency (C):** Every read receives the most recent write, or an error.
2. **Availability (A):** Every request receives a (non-error) response, without guarantee that it contains the most recent write.
3. **Partition Tolerance (P):** The system continues to operate despite network failures between nodes.

> **CRITICAL FACT:** Because networks WILL fail, **Partition Tolerance is mandatory**. Therefore, the real choice is between **CP** (Consistency) and **AP** (Availability).

### 3.2 CP vs AP — When to use what?

| Model | What it prioritizes | When to use | Example Databases |
|---|---|---|---|
| **CP** (Consistent) | System halts rather than returning stale data. | Financial transactions, inventory management. | HBase, MongoDB, Redis (Wait-for-sync) |
| **AP** (Available) | Replicas stay up, returning stale data if needed. | Social media feeds, Youtube views, likes. | Cassandra, DynamoDB |

## 5. Databases & Data Storage (SQL vs NoSQL)

### 4.1 SQL (Relational) vs NoSQL (Non-Relational)

| Feature | SQL (MySQL, PostgreSQL) | NoSQL (MongoDB, Cassandra) |
|---|---|---|
| **Structure** | Tables, rows, columns (Strict Schema) | Key-value, Document, Graph, Columnar (Flexible) |
| **Scaling** | Primarily Vertical (Hard to shard) | Horizontal (Built to shard easily) |
| **Properties** | ACID (Atomicity, Consistency, Isolation, Durability) | BASE (Basically Available, Soft state, Eventual consistency) |
| **Best For** | Money, complex relationships / JOINs | Massive read/write volume, rapid schema changes |

### 4.2 Types of NoSQL Databases

| Type | Best For | Examples |
|---|---|---|
| **Key-Value** | Caching, user sessions, fast O(1) lookups | Redis, DynamoDB, Memcached |
| **Document** | Product catalogs, CMS, varying schemas | MongoDB, Couchbase |
| **Column-Family** | High write throughput, time-series data, chat apps | Cassandra, HBase |
| **Graph** | Social networks, recommendation engines, fraud detection | Neo4j, Amazon Neptune |

### 4.3 Database Scaling Techniques

1. **Replication (Master-Slave):** 
   - Master handles *Writes*. Multiple Slaves handle *Reads*. 
   - Perfect for read-heavy applications (95% of web apps).
2. **Sharding (Partitioning):** 
   - Splitting a massive DB into smaller DBs. 
   - Example: Users A-M on DB1, N-Z on DB2. 
   - Highly complex; requires consistent hashing.

## 6. Essential Building Blocks

### 5.1 Load Balancers (LB)
Sits between clients and servers. Distributes incoming traffic.
- **Algorithms:** Round Robin, Least Connections, IP Hash.
- **Tools:** Nginx, HAProxy, AWS Application Load Balancer (ALB).

### 5.2 Caching
Storing frequently accessed data in RAM for sub-millisecond lookups.
- **Cache-Aside (Lazy Loading):** App queries DB, stores in Cache. 
- **Write-Through:** Data written to Cache and DB simultaneously (safe but slow writes).
- **Eviction Policies:** LRU (Least Recently Used), LFU, FIFO.
- **Tools:** Redis, Memcached.

### 5.3 Content Delivery Network (CDN)
Geographically distributed proxy servers caching static assets (Images, Videos, HTML/CSS/JS).
- If a user in London requests an image, it serves from a server in London, not from AWS Virginia.
- **Tools:** Cloudflare, Fastly, AWS CloudFront.

### 5.4 Message Queues
Asynchronous service-to-service communication.
- **Workflow:** Producer sends task → Queue holds it → Consumer processes it at its own pace.
- **Tools:** Apache Kafka, RabbitMQ, AWS SQS.

## 7. Architectural Paradigms: Monolith vs Microservices

| Feature | Monolithic | Microservices |
|---|---|---|
| **Definition** | All code bundled into one codebase and deployment. | Independent services communicating over network APIs. |
| **Pros** | Simple to trace, easy local testing, fast initial dev. | Scales independently, parallel teams, fault isolation. |
| **Cons** | Becomes massive "Ball of Mud", high blast radius on failure. | High network latency complexity, distributed tracing is hard. |
| **Use Case** | Early startups, MVP, small domain logic. | Massive scale (Uber, Netflix), 100+ engineering teams. |

## 8. Advanced System Architecture Patterns

### 7.1 Rate Limiting Algorithms
Prevents DoS attacks and manages API quotas.

| Algorithm | How it Works | Pros / Cons |
|---|---|---|
| **Token Bucket** | Bucket holds tokens. Refills at fixed rate. Request takes 1 token. | Smooths traffic. Very common (Amazon API Gateway). |
| **Fixed Window** | Counts requests in fixed time (e.g., 12:00 to 12:01). | Simple, but allows 2x traffic spikes at window edges. |
| **Sliding Window Log** | Stores timestamp of *every* request in Redis. | 100% accurate, but wildly memory inefficient. |
| **Sliding Window Counter** | Hybrid approach using weighted averages of previous minute. | Best balance of memory and accuracy (Cloudflare uses this). |

### 7.2 Consistent Hashing
**The Problem:** Normal hashing (`hash(key) % N`) breaks when a server is added or removed, invalidating almost all cached data.

**The Solution:** Map both Data and Servers to a circular ring (e.g., `0` to `360`). Hash the data -> land on ring -> walk clockwise to find the first server.
- **Virtual Nodes:** Add "fake" nodes for each physical server on the ring to distribute data evenly.

## 9. Communication Protocols

How does a client get real-time updates?

| Protocol | Strategy | Analogy | When to Use |
|---|---|---|---|
| **HTTP Polling** | Client asks every 5s: "Any updates?" | "Are we there yet?" | Simple dashboards. High overhead. |
| **Long Polling** | Server holds request open until data is ready. | Wait line. | Small scale chat apps. |
| **Server-Sent Events (SSE)** | Persistent 1-way connection (Server → Client). | Radio broadcast. | Stock tickers, live sports scores. |
| **WebSockets** | Persistent 2-way connection (Bi-directional). | Phone call. | Multiplayer gaming, Whatsapp, Collaborative editing. |

## 10. Mathematical Estimations for System Design

> In interviews, you must do "Back-of-the-envelope" math to justify architectural sizes.

### 9.1 Base Numbers to Memorize
- **Storage:** 1 Char = 1 Byte. Integer = 4-8 Bytes. Tweet = ~150 Bytes. Image = ~2MB. Video = ~50MB/min.
- **Scale:** 
  - 1 Million = 10⁶ (MB)
  - 1 Billion = 10⁹ (GB)
  - 1 Trillion = 10¹² (TB)

### 9.2 The "Latency Numbers Every Programmer Should Know"

| Operation | Time | Scaled (If L1 Cache = 1 sec) |
|---|---|---|
| L1 cache reference | 0.5 ns | 1 second |
| Main memory (RAM) reference | 100 ns | 3.3 minutes |
| Read 1 MB sequentially from SSD | 1,000,000 ns (1 ms) | 23 days |
| Disk seek (Magnetic HDD) | 10,000,000 ns (10 ms) | 7.6 months |
| Send packet CA->Netherlands->CA | 150,000,000 ns (150 ms) | 9.5 years |

> **Conclusion:** Avoid Disk (DB) and Network operations whenever possible. Use RAM (Redis).

## 11. Problem-Solving Frameworks (PEDALS)

Follow this 6-step framework for every System Design Interview.

| Step | What to do | Example |
|---|---|---|
| **P - Problem Clarification** | Ask questions, define scope, outline functional/non-functional reqs. | "Are we designing the feed or just posting? What is DAU?" |
| **E - Estimations** | DAU, RPS, Storage, Bandwidth. | "100M DAU * 2 reads/day = X RPS." |
| **D - Design High-Level** | Draw Client → LB → API → DB. | Get buy-in from interviewer on the macro flow. |
| **A - Architecture Deep Dive** | Data Schema, API Endpoints. | Define DB tables, partition keys, `POST /v1/tweet`. |
| **L - Limits & Bottlenecks** | Identify SPOFs and breaking points. | "If Justin Bieber tweets, our pulling mechanism will crash." |
| **S - Scale** | Fix bottlenecks with advanced tools. | "We will implement Redis Fanout (Push model) for caching." |

## 13. The System Design Interview Guide

### 💡 General Tips
1. **Drive the conversation.** You are the architect. The interviewer is the client.
2. **Defend your trade-offs.** "I chose Cassandra to handle the massive write-throughput of chat logs, even though it sacrifices strict consistency."
3. **Start Simple.** Build a working system for 10 users, then scale it to 10 Million.

### 🎯 Frequently Asked Interview Questions

#### Q1: Design a URL Shortener (e.g., TinyURL)
- **Focus:** API design, Base62 encoding, Key-Value DB (DynamoDB), heavy Read/Write ratio (Cache!).

#### Q2: Design Twitter / News Feed
- **Focus:** Push vs Pull models. Fan-out service using Redis for feed generation. How to handle celebrity spikes.

#### Q3: Design a Chat Application (WhatsApp)
- **Focus:** WebSockets, presence indicators, message ordering (Sequence IDs), write-heavy DBs (Cassandra).

#### Q4: Design a Rate Limiter
- **Focus:** Token Bucket algorithm, Redis + Lua scripts for atomic operations, distributed counters.

#### Q5: Design a Ride-Sharing App (Uber)
- **Focus:** Geospatial databases (QuadTrees / Geohash), WebSockets for live driver tracking, Dispatch algorithms.

## 14. Real-World Architecture Breakdowns

### 🔍 Scenario 1: Database Query Optimization

**Interviewer:** *"Your SQL query on a table takes 30 seconds. How do you speed it up?"*
**Answer — System Design Concepts:**
- Implement B-Tree Indexing = `O(log n)` lookups.
- Use a Redis Cache for frequent exact matches.
- Split the table horizontally (Sharding) if it's too large for indexing perfectly.

### 🔍 Scenario 2: Twitter/X Feed (The "Justin Bieber" Problem)

**The Problem:** Normal users have 500 followers. Celebrities have 100 Million.
**The Solution (Hybrid Fanout):**
- **Push Model:** For a normal user (500 followers), when they tweet, a background worker instantly pushes the tweet into the pre-computed Redis feeds of those 500 people. O(1) read latency.
- **Pull Model:** If Elon Musk tweets, pushing to 100M Redis instances would crash the system. Instead, his tweets are stored normally. When you open your feed, your app *pulls* Elon's tweets and merges them with your cached feed at run-time.

### 🔍 Scenario 3: Uber Geospatial Matching

**The Problem:** Finding drivers near a rider instantly. SQL `SELECT * WHERE lat > A AND long > B` is too slow on massive scale.
**The Solution (Geohashing):**
- Divide the world into a grid. Zooming in adds precision (`dr5x` = NY, `dr5x8` = Manhattan).
- Store drivers in Redis using the Geohash as a key. If a rider is in `dr5x8`, the system only queries drivers with the exact same prefix.

### 🔍 Scenario 4: Netflix Video Streaming

**The Problem:** Streaming heavy video globally with 0 buffering.
**The Solution (Open Connect CDNs):**
- Netflix does NOT stream video from AWS to your house.
- They install custom hardware (Open Connect Appliances) directly inside local ISPs (Comcast, Verizon).
- The movies are cached just a few miles from your router, bypassing the public internet entirely.

## 15. Full Overview of the Software Development Life Cycle (SDLC)

> **"System Design is just one phase of building software."** The SDLC is the overarching framework that tech companies use to build, maintain, and replace software.

The SDLC ensures that software is built systematically, minimizing risk, reducing costs, and guaranteeing that the final product actually solves the user's problem.

### The 7 Phases of the SDLC

#### Phase 1: Planning & Requirement Analysis ("What are we building?")
*   **Who is involved:** Product Managers, Business Analysts, Senior Engineers.
*   **What happens:** The team defines the problem. They gather requirements from clients, shareholders, and market research.
*   **Key Deliverable:** A PRD (Product Requirement Document) or SRS (Software Requirement Specification).
*   **Example:** "We need an app where users can order coffee ahead of time so they don't wait in line."

#### Phase 2: System Design ("How will we build it?")
*   **Who is involved:** Software Architects, Lead Engineers.
*   **What happens:** This is where everything in this guide comes into play! The team turns the PRD into technical blueprints.
*   **High-Level Design (HLD):** Defining the architecture (Load Balancers, Microservices, Databases).
*   **Low-Level Design (LLD):** Defining the database schema, API endpoints, and core classes.
*   **Key Deliverable:** Design Document (Technical Spec).

#### Phase 3: Implementation / Coding ("Let's build it.")
*   **Who is involved:** Frontend, Backend, and Mobile Developers.
*   **What happens:** The actual writing of the code based on the Design Documents. This is typically the longest phase. Code is usually version-controlled using Git (GitHub/GitLab) and broken down into sprints (Agile).
*   **Key Deliverable:** Source Code.

#### Phase 4: Testing ("Does it actually work?")
*   **Who is involved:** QA (Quality Assurance) Engineers, Automation Engineers.
*   **What happens:** The code is tested relentlessly to find bugs before the user does.
*   **Types of Testing:**
    *   *Unit Testing:* Testing individual functions.
    *   *Integration Testing:* Testing if microservices talk to each other correctly.
    *   *System/E2E Testing:* Testing the entire app flow from start to finish.
    *   *Load Testing:* Simulating 10,000 users to ensure the System Design holds up.
*   **Key Deliverable:** Bug Reports and a stable, "Release Candidate" build.

#### Phase 5: Deployment ("Ship it to the users.")
*   **Who is involved:** DevOps Engineers, SREs (Site Reliability Engineers).
*   **What happens:** The code is moved from the testing environment to the "Production" environment (live).
*   **Deployment Strategies:**
    *   *Blue-Green Deployment:* Deploy the new version (Green) alongside the old one (Blue). Switch traffic instantly. If it crashes, switch back.
    *   *Canary Release:* Roll the new feature out to just 5% of users. Monitor for crashes. If stable, roll out to 100%.
*   **Key Deliverable:** A live product accessible by the public.

#### Phase 6: Maintenance & Operations ("Keep it running.")
*   **Who is involved:** SREs, Customer Support, Core Engineering Team.
*   **What happens:** The product is live, but the work isn't done. The team monitors server health (using tools like Datadog or Prometheus), fixes newly discovered bugs, and releases security patches.
*   **Key Deliverable:** High "nines" of Availability (e.g., 99.99% Uptime).

#### Phase 7: Sunsetting / End-of-Life (Optional)
*   **What happens:** Eventually, software becomes obsolete (e.g., Internet Explorer). The service is gracefully shut down, and users are migrated to a newer system.

---

### Popular SDLC Methodologies

How companies actually run these 7 phases differs drastically:

| Methodology | How it works | When to use it |
| :--- | :--- | :--- |
| **Waterfall** | Linear. You finish Phase 1 completely before starting Phase 2. No going backward. | Mission-critical systems (NASA, pacemakers) where requirements never change. |
| **Agile (Scrum)** | Iterative. You do all 7 phases in 2-week "Sprints" for small features. Constant feedback. | 95% of modern tech companies. Startups, web apps, SaaS products. |
| **CI/CD (DevOps)** | "Continuous Integration / Continuous Deployment." Code is automatically tested and deployed the moment a developer hits "merge." | Elite tech companies (Netflix deploys code thousands of times a day). |

---

> _"System Design is the bridge between the Business Requirements in Phase 1 and the actual Code written in Phase 3. Master this, and you master the entire Software Development Lifecycle."_
