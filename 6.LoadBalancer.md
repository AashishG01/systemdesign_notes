# Part 1: Core Concepts

---

## 1. What is a Load Balancer?

### 1.1 The Definition

A **Load Balancer** is a device or software that sits between clients and a pool of backend servers and distributes incoming network traffic across those servers to ensure no single server is overwhelmed.

> **💡 Intuition — The Restaurant Host:**
> Imagine a restaurant with 10 tables (servers). Without a host, all customers walk in and crowd around Table #1 because it's the first one they see. Table #1 collapses, while Tables #2-10 sit empty.
>
> A **Load Balancer is the host** standing at the door. They check which tables are full, which are empty, and seat each new customer at the optimal table. No table is overwhelmed. If Table #5 is being cleaned (server down), the host simply stops seating people there.

### 1.2 The Core Responsibilities

| Responsibility | What it Does |
| :--- | :--- |
| **Traffic Distribution** | Spreads requests evenly across a pool of servers |
| **Health Monitoring** | Continuously checks if servers are alive and removes dead ones |
| **High Availability** | If a server crashes, traffic is seamlessly rerouted to healthy servers. Zero user impact |
| **SSL Termination** | Handles the expensive HTTPS encryption/decryption so backend servers don't have to |
| **Horizontal Scaling** | Enables adding/removing servers dynamically without downtime |

## 2. Why Load Balancers are Non-Negotiable

### 2.1 Without a Load Balancer

```
User A ──▶ Server 1 (IP: 192.168.1.1)  ← 100% of traffic. CPU at 100%. CRASH 💥
User B ──▶ Server 1
User C ──▶ Server 1
           Server 2 (IP: 192.168.1.2)  ← 0% of traffic. Sitting idle.
           Server 3 (IP: 192.168.1.3)  ← 0% of traffic. Sitting idle.
```

**Problems:**
1.  Users must know the IP address of a specific server. If that server dies, they get an error.
2.  One server gets all traffic while others sit idle — total waste of resources.
3.  You can't add or remove servers without telling every user to use a new IP.

### 2.2 With a Load Balancer

```
User A ──┐                          ┌──▶ Server 1 (33% traffic) ✅
User B ──┼──▶ Load Balancer ────────┼──▶ Server 2 (33% traffic) ✅
User C ──┘    (api.myapp.com)       └──▶ Server 3 (33% traffic) ✅
```

**Benefits:**
1.  Users only know one address: `api.myapp.com` (the Load Balancer's IP). They never know individual servers exist.
2.  Traffic is evenly distributed. No single server is overwhelmed.
3.  Add Server 4 at 3 AM? Just register it with the LB. Remove Server 2 for maintenance? Just deregister it. **Zero downtime. Zero user impact.**

## 3. Where Load Balancers Sit in the Architecture

Load Balancers typically appear at **3 critical points** in a modern system:

```
┌──────────┐
│  Users   │
└────┬─────┘
     │
     ▼
┌─────────────────────┐
│  LB #1: Edge LB     │ ← Between Users and Web/API Servers
│  (NGINX / AWS ALB)   │    Handles SSL, rate limiting, routing
└────────┬────────────┘
         │
    ┌────┼────┐
    ▼    ▼    ▼
  ┌───┐┌───┐┌───┐
  │API││API││API│  ← Web/API Server Pool
  │ 1 ││ 2 ││ 3 │
  └─┬─┘└─┬─┘└─┬─┘
    │    │    │
    └────┼────┘
         ▼
┌─────────────────────┐
│  LB #2: Internal LB  │ ← Between API Servers and Microservices
│  (Service Discovery)  │    Routes to correct microservice
└────────┬────────────┘
         │
    ┌────┼────┐
    ▼    ▼    ▼
  ┌───┐┌───┐┌───┐
  │Svc││Svc││Svc│  ← Microservice Pool (Payment, Auth, etc.)
  │ A ││ B ││ C │
  └───┘└───┘└───┘
         │
         ▼
┌─────────────────────┐
│  LB #3: Database LB  │ ← Between Services and Database Replicas
│  (Read Replica Pool)  │    Routes reads to replicas, writes to master
└─────────────────────┘
```

## 4. Layer 4 (L4) vs Layer 7 (L7) Load Balancing

This is the **most important distinction** in load balancing and a frequent interview question. The "Layer" refers to the **OSI Model Layer** at which the load balancer makes its routing decision.

### 4.1 Layer 4 Load Balancing (Transport Layer)

**What it inspects:** Only the TCP/UDP packet headers — the **source IP, destination IP, and port number**. It cannot see the HTTP payload (URL, headers, cookies).

```
Incoming Packet:
┌──────────────────────────────────────────────────────┐
│  Source: 72.14.204.1:5234                            │  ← L4 LB sees ONLY this
│  Dest:   api.myapp.com:443                           │
│  ──────────────────────────────────────────────────── │
│  GET /api/users HTTP/1.1                             │  ← L4 LB CANNOT see this
│  Host: api.myapp.com                                 │
│  Cookie: session=abc123                              │
└──────────────────────────────────────────────────────┘
```

**How it routes:** Purely based on IP + Port. It doesn't know if the request is `GET /images/cat.jpg` or `POST /api/payments`. It just forwards the raw TCP connection to a backend server.

**Pros:** Extremely fast (minimal processing), low CPU overhead, handles millions of connections.
**Cons:** Dumb routing. Cannot route based on URL path, HTTP method, cookies, or headers.

**Example:** **AWS Network Load Balancer (NLB)**, HAProxy (in TCP mode).

### 4.2 Layer 7 Load Balancing (Application Layer)

**What it inspects:** The **full HTTP request** — URL path, HTTP method, headers, cookies, query parameters, and even the request body.

```
Incoming Request:
┌──────────────────────────────────────────────────────┐
│  GET /api/users HTTP/1.1                             │
│  Host: api.myapp.com                                 │  ← L7 LB reads ALL of this
│  Cookie: session=abc123                              │
│  Authorization: Bearer eyJhbGc...                    │
└──────────────────────────────────────────────────────┘
```

**How it routes:** Based on content-aware rules.

```nginx
# NGINX L7 Routing Rules:
location /api/          → Route to API Server Pool
location /images/       → Route to Static File Server Pool
location /websocket/    → Route to WebSocket Server Pool

# Route by Cookie (A/B Testing):
if ($cookie_experiment == "new_ui") → Route to Canary Server Pool
```

**Pros:** Intelligent routing, A/B testing, canary deployments, path-based microservice routing.
**Cons:** Higher CPU overhead (must parse full HTTP request), slightly more latency.

**Example:** **NGINX**, **AWS Application Load Balancer (ALB)**, HAProxy (in HTTP mode), Envoy.

### 4.3 L4 vs L7 Comparison Table

| Feature | Layer 4 (Transport) | Layer 7 (Application) |
| :--- | :--- | :--- |
| **Inspects** | IP + Port only | Full HTTP request (URL, headers, cookies) |
| **Speed** | ⚡ Extremely fast | Fast (slightly slower due to parsing) |
| **Intelligence** | Dumb — no content-aware routing | Smart — path-based, header-based, cookie-based |
| **SSL Termination** | ❌ No (passes encrypted packets through) | ✅ Yes (decrypts, inspects, re-encrypts) |
| **Use Case** | Raw TCP/UDP traffic, gaming, IoT | HTTP APIs, microservice routing, A/B testing |
| **Product** | AWS NLB, HAProxy (TCP) | **NGINX**, AWS ALB, Envoy, HAProxy (HTTP) |

> **💡 Interview Answer:** "For most web applications, I use a **Layer 7 LB (NGINX or ALB)** because I need path-based routing to microservices and SSL termination. For ultra-high-throughput non-HTTP traffic like gaming or video streaming, I use a **Layer 4 LB (NLB)** for raw performance."

## 5. Hardware vs Software Load Balancers

| Aspect | Hardware LB | Software LB |
| :--- | :--- | :--- |
| **What it is** | A dedicated physical appliance (specialized hardware + custom OS) | A program running on commodity servers |
| **Examples** | F5 BIG-IP, Citrix ADC, A10 Networks | **NGINX**, HAProxy, Envoy, Traefik |
| **Cost** | $50,000 - $500,000+ per appliance | Free (open source) or cloud pricing |
| **Performance** | Extremely high (ASICs for packet processing) | Very high (software optimization on general-purpose CPUs) |
| **Flexibility** | Limited — vendor lock-in, firmware updates | Extremely flexible — scripting, custom rules, plugins |
| **Scaling** | Buy another $100K box | Spin up another VM in 30 seconds |
| **Who uses it** | Legacy banks, telecoms, governments | **Everyone else.** Netflix, Google, Amazon, every startup |

> **💡 Modern Reality:** Hardware LBs are dying. Software LBs running on cloud VMs dominate the industry. NGINX alone handles over 30% of all web traffic on the internet.

## 6. Load Balancing Algorithms — How Traffic is Distributed

The LB must decide: "I received Request #47,291. Which of my 10 servers should I send it to?" This decision is made by an **algorithm**.

### 6.1 Round Robin

**How it works:** Requests are sent to servers in sequential, circular order: 1, 2, 3, 1, 2, 3...

```
Request 1 → Server 1
Request 2 → Server 2
Request 3 → Server 3
Request 4 → Server 1  (wraps around)
Request 5 → Server 2
Request 6 → Server 3
```

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| Simplest possible algorithm | Ignores server capacity. A 4-CPU server gets the same traffic as a 32-CPU server |
| Zero state to maintain | Ignores request weight. A `GET /health` (1ms) and a `POST /upload-video` (10 min) are treated equally |

**Best For:** Homogeneous server fleets where all servers are identical and all requests are roughly equal.

### 6.2 Weighted Round Robin

**How it works:** Like Round Robin, but servers with higher capacity get proportionally more requests.

```
Server 1 (8 CPUs, weight=4):  Gets 4 out of every 7 requests
Server 2 (4 CPUs, weight=2):  Gets 2 out of every 7 requests
Server 3 (2 CPUs, weight=1):  Gets 1 out of every 7 requests
```

**Best For:** Heterogeneous server fleets where some servers are more powerful than others.

### 6.3 Least Connections

**How it works:** The LB tracks how many active connections each server currently has and routes the next request to the server with the **fewest active connections**.

```
Server 1: 47 active connections
Server 2: 12 active connections  ← next request goes HERE
Server 3: 85 active connections
```

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| Adapts to real-time server load | Requires the LB to track state (connection count per server) |
| Handles slow requests well (slow server accumulates connections, gets less new traffic) | Slightly more CPU overhead on the LB |

**Best For:** Systems with variable request processing times (some requests take 5ms, others take 5 seconds).

### 6.4 Least Response Time

**How it works:** Routes to the server with the **lowest average response time AND fewest active connections**.

**Best For:** When you want the fastest possible user experience. The LB actively measures which server is performing best right now.

### 6.5 IP Hash

**How it works:** Takes the client's IP address, runs it through a hash function, and the hash deterministically maps to a specific server. The same IP always goes to the same server.

```
hash("72.14.204.1") % 3 = 1  → Server 1 (always)
hash("98.132.41.9") % 3 = 0  → Server 0 (always)
hash("45.76.31.8")  % 3 = 2  → Server 2 (always)
```

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| Primitive "sticky sessions" — same user always hits the same server | If a server is removed, ALL hash mappings change. Many users get reshuffled to different servers, breaking their sessions |
| Good for stateful protocols | Doesn't balance load if IPs aren't evenly distributed |

**Best For:** When you need basic session affinity without cookies. **But Consistent Hashing (Section 7) is far superior.**

### 6.6 Random

**How it works:** Randomly picks a server for each request. Statistically, over time, it distributes requests roughly evenly.

**Best For:** Surprisingly effective for large server pools (>50 servers) where perfect distribution isn't critical.

### 6.7 Algorithm Comparison Table

| Algorithm | Load Awareness | State Needed | Best For |
| :--- | :--- | :--- | :--- |
| **Round Robin** | ❌ None | None | Identical servers, uniform requests |
| **Weighted Round Robin** | ❌ None (pre-configured) | Weights | Mixed server sizes |
| **Least Connections** | ✅ Connection count | Per-server counter | Variable-duration requests |
| **Least Response Time** | ✅ Response time + connections | Per-server metrics | Latency-sensitive apps |
| **IP Hash** | ❌ None | None | Basic sticky sessions |
| **Random** | ❌ None | None | Large pools, simplicity |

## 7. Consistent Hashing — The Special Algorithm

### 7.1 The Problem with Regular Hashing

With regular IP Hash, if you have 3 servers and add a 4th:

```
BEFORE (3 servers):
hash(request) % 3 = server_index

AFTER (4 servers):
hash(request) % 4 = server_index  ← EVERY mapping changes!
```

When the number of servers changes, the modulo result changes for almost **every request**. If servers were caching user-specific data, those caches are now on the wrong servers. **Cache hit rate drops to near zero.**

### 7.2 How Consistent Hashing Works

Imagine a circular ring (0 to 360 degrees). Both **servers** and **request keys** are hashed onto this ring.

```
                        0° / 360°
                          │
                    Server A (hash=30°)
                   ╱      │
                 ╱        │
               ╱          │
            ╱             │
   270° ───╱──────────────┼──────────── 90°
           │              │            ╲
           │              │              ╲
           │    Server C  │               Server B
           │   (hash=200°)│              (hash=120°)
           │              │
                180°
```

**Rule:** A request is routed to the **first server found clockwise** from its hash position on the ring.

```
Request X (hash=50°) → walks clockwise → hits Server B (120°) ✅
Request Y (hash=150°) → walks clockwise → hits Server C (200°) ✅
Request Z (hash=250°) → walks clockwise → hits Server A (30°, wrapping around) ✅
```

### 7.3 Why it's Better: Adding/Removing Servers

**Adding Server D at position 160°:**
*   Only requests between 120° and 160° are remapped (from Server C to Server D).
*   **All other requests stay on their original servers.** Cache disruption is minimal.

**Removing Server B (120°):**
*   Only requests that were going to B (30° - 120°) are now routed to the next clockwise server (Server C).
*   **All other requests are unaffected.**

> **💡 Key Benefit:** When adding or removing 1 server from N servers, only **~1/N** of requests are remapped, not **all** requests like with regular hashing.

### 7.4 Virtual Nodes (Fixing Uneven Distribution)

With only 3 servers on the ring, one server might "own" 60% of the ring space (handling 60% of traffic). **Virtual Nodes** fix this by placing multiple hash points per server:

```
Server A → hash positions: 30°, 110°, 250°, 330°  (4 virtual nodes)
Server B → hash positions: 60°, 140°, 210°, 290°  (4 virtual nodes)
Server C → hash positions: 90°, 170°, 240°, 350°  (4 virtual nodes)
```

With 100-200 virtual nodes per server, load distribution becomes nearly perfect.

> **💡 Where Consistent Hashing is Used:** Amazon DynamoDB, Apache Cassandra, Memcached, CDN routing, Discord, Akamai. Anywhere servers are added/removed dynamically.

## 8. Health Checks — Detecting Dead Servers

### 8.1 What is a Health Check?

The LB continuously sends small "Are you alive?" requests to each backend server at regular intervals. If a server fails to respond, the LB marks it as **unhealthy** and stops routing traffic to it.

### 8.2 Types of Health Checks

| Type | How it Works | What it Catches |
| :--- | :--- | :--- |
| **TCP Health Check** | LB tries to open a TCP connection to the server's port. Success = healthy. | Server crashed, port not listening, network is down |
| **HTTP Health Check** | LB sends `GET /health` and checks for HTTP 200 OK response. | App crashes, deployment failures, bad code pushed |
| **Deep Health Check** | LB sends `GET /health/deep` which also checks DB connectivity, Redis connectivity, disk space | Database down, Redis unreachable, disk full |

### 8.3 Health Check Configuration

```nginx
# NGINX health check example
upstream backend_pool {
    server 10.0.1.1:8080;
    server 10.0.1.2:8080;
    server 10.0.1.3:8080;
}

server {
    location /api/ {
        proxy_pass http://backend_pool;
        
        # Health check settings
        health_check interval=10s    # Check every 10 seconds
                     fails=3         # Mark unhealthy after 3 consecutive failures
                     passes=2;       # Mark healthy again after 2 consecutive successes
    }
}
```

### 8.4 Health Check Flow

```
                    ┌──────────────┐
                    │ Load Balancer│
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
          ┌───────┐   ┌───────┐   ┌───────┐
  Every   │ Srv 1 │   │ Srv 2 │   │ Srv 3 │
  10 sec: │ GET   │   │ GET   │   │ GET   │
          │/health│   │/health│   │/health│
          │       │   │       │   │       │
  Result: │ 200 ✅│   │ 500 ❌│   │ 200 ✅│
          └───────┘   └───────┘   └───────┘
                           │
                     3 failures in
                      a row = REMOVED
                      from pool
```

> **💡 Interview Insight:** A "deep" health check that verifies DB + Redis + Disk is dangerous if the DB is temporarily slow. All servers fail the deep check simultaneously, and the LB removes ALL servers (total outage). **Solution:** Use a shallow `/health` (app is alive) for the LB and a separate `/health/deep` for monitoring/alerting only.

## 9. Sticky Sessions (Session Affinity)

### 9.1 What Are Sticky Sessions?

**Sticky Sessions** force the LB to route all requests from the **same client** to the **same backend server** for the duration of their session.

### 9.2 Why Would You Need Them?

If your backend server is **stateful** (stores session data in local memory), switching servers mid-session loses the user's state (shopping cart, form data, login session).

### 9.3 How They Work

```
First Request:
  User A ──▶ Load Balancer ──▶ Server 2
                │
                └─ LB sets cookie: "SERVERID=srv2"

All Subsequent Requests:
  User A (cookie: SERVERID=srv2) ──▶ Load Balancer ──▶ Server 2 (always)
```

### 9.4 Why Sticky Sessions Are an Anti-Pattern

| Problem | Explanation |
| :--- | :--- |
| **Uneven Load** | Popular users are "stuck" to one server. That server gets overloaded while others are idle. |
| **Scaling Failure** | If you add Server 4, existing sticky users don't redistribute. The new server sits empty. |
| **Failover Failure** | If Server 2 crashes, ALL users stuck to it lose their session data and must re-login. |
| **Horizontal Scaling Broken** | The entire point of horizontal scaling is that ANY server can handle ANY request. Sticky sessions break this. |

> **💡 The Correct Solution:** Make your backend **stateless**. Store sessions in **Redis** (external, shared across all servers). Then sticky sessions become unnecessary. Any server can handle any user.

## 10. SSL/TLS Termination

### 10.1 What is SSL Termination?

HTTPS traffic is encrypted. Someone must decrypt it. **SSL Termination** is when the Load Balancer handles the encryption/decryption, so backend servers receive plain HTTP.

```
WITH SSL Termination at the LB:
  User ──[HTTPS/Encrypted]──▶ Load Balancer ──[HTTP/Plain]──▶ Backend Server
                                    │
                             Decrypts here
                             (CPU-intensive)

WITHOUT SSL Termination (End-to-End):
  User ──[HTTPS/Encrypted]──▶ Load Balancer ──[HTTPS/Encrypted]──▶ Backend Server
                                                                      │
                                                               Decrypts here
                                                               (every server does it)
```

### 10.2 Why Terminate at the LB?

| Benefit | Explanation |
| :--- | :--- |
| **Offloads CPU** | SSL/TLS decryption is CPU-intensive. Offloading it to the LB frees backend servers to focus on business logic. |
| **Centralized Certificate Management** | Manage SSL certificates in ONE place (the LB) instead of on every backend server. |
| **Simpler Backend** | Backend servers handle plain HTTP — simpler code, easier debugging. |
| **Inspection** | The L7 LB can now read the decrypted HTTP request to make content-based routing decisions. |

> **⚠️ Security Note:** Traffic between the LB and backend servers is unencrypted (HTTP). If the internal network is untrusted (e.g., shared cloud), use **SSL re-encryption** (LB decrypts, then re-encrypts with an internal certificate before forwarding).

## 11. High Availability for the Load Balancer Itself

### 11.1 The Problem: The LB is a SPOF

If you have just one Load Balancer and it crashes, ALL traffic stops. The LB itself becomes a Single Point of Failure.

### 11.2 The Solution: Active-Passive (Failover)

```
                    ┌──────────────────┐
    Traffic ───────▶│ LB #1 (ACTIVE)   │──────▶ Server Pool
                    └────────┬─────────┘
                             │ Heartbeat every 1 second
                    ┌────────▼─────────┐
                    │ LB #2 (PASSIVE)  │   ← Idle. Waiting. Monitoring LB #1.
                    │ (Standby)        │      Shares the same Virtual IP (VIP).
                    └──────────────────┘

    If LB #1 dies:
                    ┌──────────────────┐
                    │ LB #1 💀 DEAD     │
                    └──────────────────┘
                    ┌──────────────────┐
    Traffic ───────▶│ LB #2 (ACTIVE)   │──────▶ Server Pool
                    │ Takes over VIP   │      Failover in < 5 seconds
                    └──────────────────┘
```

**How it works:**
1.  Two LBs share a **Virtual IP (VIP)** — the IP that DNS points to.
2.  LB #1 is active. LB #2 sends a heartbeat ping to LB #1 every second.
3.  If LB #2 detects 3 missed heartbeats, it assumes LB #1 is dead.
4.  LB #2 announces itself as the new owner of the VIP (via ARP/GARP).
5.  Traffic seamlessly flows to LB #2. **Users experience zero downtime.**

**Technology:** VRRP (Virtual Router Redundancy Protocol), **Keepalived** (Linux), AWS ELB (managed, automatically HA).

### 11.3 Active-Active

Both LBs handle traffic simultaneously. DNS returns both IPs, or a hardware/cloud layer distributes between them. Double the capacity and no wasted standby resources.

## 12. Global Server Load Balancing (GSLB)

### 12.1 What is GSLB?

GSLB routes users to the **geographically closest data center** using DNS-based load balancing.

```
User in Tokyo ──▶ DNS ──▶ "api.myapp.com = 103.21.x.x (Tokyo DC)" ──▶ Tokyo Servers
User in Paris ──▶ DNS ──▶ "api.myapp.com = 185.72.x.x (London DC)" ──▶ London Servers
User in NYC   ──▶ DNS ──▶ "api.myapp.com = 54.231.x.x (Virginia DC)" ──▶ Virginia Servers
```

### 12.2 How GSLB Makes Routing Decisions

| Factor | Description |
| :--- | :--- |
| **Geographic Proximity** | Route to the closest datacenter to minimize latency (speed of light through fiber) |
| **Health** | If the Tokyo DC is completely down, route Tokyo users to the next closest DC (Singapore) |
| **Load** | If Virginia DC is at 95% capacity, route some East Coast users to the Ohio DC instead |
| **Compliance** | EU users' data MUST stay in EU data centers (GDPR). Route EU users exclusively to the London DC |

**Examples:** AWS Route 53, Cloudflare DNS, Akamai GTM.

## 13. Real-World Load Balancing Scenarios

### Scenario 1: Netflix Video Streaming

**Challenge:** 200+ million users worldwide watching video simultaneously. Each stream needs sustained, high-throughput data delivery.

| Layer | LB Technology | Why |
| :--- | :--- | :--- |
| **GSLB** | AWS Route 53 (DNS) | Routes users to the nearest AWS region (US-East, EU-West, AP-Southeast) |
| **Edge LB** | AWS ALB (L7) | Routes `/api/` to the API fleet and `/video/` to the streaming fleet based on URL path |
| **Internal LB** | Eureka + Ribbon (Netflix OSS) | Client-side LB for inter-microservice communication. Each service discovers others via Eureka registry and load-balances client-side |

### Scenario 2: E-Commerce Flash Sale (Amazon Prime Day)

**Challenge:** Traffic spikes 100x in 5 minutes. 300 million items sold in 48 hours.

| Technique | Implementation |
| :--- | :--- |
| **Pre-scaling** | Spin up 10x more servers 1 hour before the sale starts. Register them with the LB |
| **L7 Path Routing** | `/api/deals` → dedicated "Deals" server pool (massively scaled). `/api/profile` → normal pool (low traffic today) |
| **Least Connections** | Use Least Connections algorithm because deal-page requests are heavy (DB queries, inventory checks) and take variable time |
| **Auto-scaling** | AWS ALB integrates with Auto Scaling Groups. If CPU > 70% for 3 minutes, automatically add 10 more servers |

### Scenario 3: Gaming Server (Fortnite/PUBG)

**Challenge:** Real-time multiplayer game. Ultra-low latency is critical (<<50ms). Players must stay connected to the SAME game server for the entire match.

| Technique | Implementation |
| :--- | :--- |
| **Layer 4 LB** | Use NLB (Layer 4) for raw UDP/TCP performance — no HTTP parsing overhead |
| **GSLB** | Route players to the geographically closest datacenter to minimize ping |
| **Consistent Hashing** | Once a player is assigned to a match server, consistent hashing ensures reconnections go to the same server |
| **Sticky Sessions** | Necessary here — the game state IS on the server. If the server changes, the player loses the match. This is one of the rare valid use cases for sticky sessions |

## 14. Interview Questions & Problems with Solutions

### ❓ Q1: "What happens if the Load Balancer itself goes down?"

**Answer:**
"A single LB is a SPOF. In production, we always deploy an **Active-Passive pair** using Keepalived/VRRP. Both LBs share a Virtual IP (VIP). The Passive LB sends heartbeats to the Active every second. If the Active dies (~3 missed heartbeats), the Passive takes over the VIP via a GARP broadcast. Failover happens in under 5 seconds. Users don't notice.

Cloud providers solve this automatically — AWS ALB/NLB are managed services that are inherently highly available across multiple Availability Zones."

---

### ❓ Q2: "Layer 4 or Layer 7? Which would you use for a REST API?"

**Answer:**
"**Layer 7 (NGINX or AWS ALB).** I need to:
1.  Inspect the URL path to route `/api/users` to the User Service and `/api/payments` to the Payment Service.
2.  Perform SSL termination centrally.
3.  Read the `Authorization` header for authentication decisions.
4.  Implement rate limiting based on API key.

None of this is possible with a Layer 4 LB, which can only see IP addresses and ports."

---

### ❓ Q3: "We have 100 servers. We're adding 10 more. How do we ensure minimal cache disruption?"

**Answer:**
"Use **Consistent Hashing**. With regular hash-based routing (`hash % N`), changing N from 100 to 110 reshuffles nearly ALL requests to different servers, destroying cache hits.

With Consistent Hashing, each server occupies a position on a virtual ring. Adding 10 servers only remaps ~10/110 (~9%) of requests. The remaining 91% continue hitting the same server with warm caches. Using **virtual nodes** (100+ per server) ensures the 9% remapping is evenly distributed."

---

### ❓ Q4: "A user is complaining that their shopping cart keeps disappearing. What's happening?"

**Answer:**
"Classic stateful architecture bug. The cart is stored in the **server's local memory**. The LB is using Round Robin, so each request goes to a different server:
- Request 1 → Server A (add to cart, stored in A's RAM)
- Request 2 → Server B (view cart — empty! B has no idea about A's RAM)

**Fix:** Make the backend **stateless**. Move the cart to a shared **Redis** instance. Now Servers A, B, and C all read/write the cart from the same Redis. The LB can route to any server and the cart persists."

---

### ❓ Q5: "How would you handle a sudden 50x traffic spike to a single API endpoint?"

**Answer:**
"A multi-layered approach:
1.  **L7 Path-Based Routing:** Route the hot endpoint (`/api/viral-post`) to a **dedicated, isolated server pool** so it doesn't starve other endpoints.
2.  **Auto-Scaling:** Configure the LB's auto-scaling group to add servers when CPU > 60% for 2 minutes.
3.  **Rate Limiting at the LB:** NGINX `limit_req` module caps each IP to 10 requests/second to prevent abuse.
4.  **Caching:** If the endpoint returns the same data for every user, put a **Redis cache** with a 10-second TTL. 99% of traffic is served from cache, not the backend.
5.  **CDN:** If it's a static resource, push it to the CDN edge. The LB and backend servers never even see the traffic."

---

### ❓ Q6: "Explain the difference between a Reverse Proxy and a Load Balancer."

**Answer:**

| Feature | Reverse Proxy | Load Balancer |
| :--- | :--- | :--- |
| **Primary Purpose** | Sits in front of servers to provide security, caching, and SSL termination | Distributes traffic across multiple servers |
| **# of Backend Servers** | Can work with just 1 backend server | Requires 2+ servers to be meaningful |
| **Key Features** | Caching, compression, SSL termination, request rewriting, security (hiding internal IPs) | Traffic distribution, health checks, failover, scaling |
| **Overlap** | ✅ Often acts as a load balancer too | ✅ Often acts as a reverse proxy too |

"In practice, **NGINX is both** a reverse proxy and a load balancer simultaneously. The terms overlap heavily. If an interviewer insists on a distinction: a Reverse Proxy is about **shielding the backend**, while a Load Balancer is about **distributing traffic**."
