## 1. What is a Distributed System?

### 1.1 The Definition
A **Distributed System** is a collection of independent computers (called **nodes**) that appears to the outside world as a single, unified system. The nodes communicate by passing messages over a network to achieve a common goal.

> **💡 Intuition — The Restaurant Kitchen:**
> *   A **Non-Distributed System** is a single chef doing everything: chopping vegetables, grilling meat, plating, and washing dishes. If the chef is sick, the restaurant shuts down.
> *   A **Distributed System** is a brigade — a sous-chef, a grill master, a pastry chef, a dishwasher. They yell "Order up!" to coordinate. If the pastry chef takes a break, the main courses keep flowing.

### 1.2 The 3 Core Characteristics
Why does a "team of computers" behave fundamentally differently from a single powerful computer?

| Characteristic | What it Means | Real Consequence |
| :--- | :--- | :--- |
| **Concurrency** | Multiple nodes execute tasks at the exact same time. | You can process 1,000 user requests simultaneously across 10 servers, not one after another. |
| **No Global Clock** | You cannot rely on wall-clock time to order events (network delays make timestamps unreliable). | You must use algorithms like **Lamport Timestamps** or **Vector Clocks** to establish "what happened before what." |
| **Independent Failures** | Node A can crash while Nodes B and C keep running perfectly. | The system must be designed to tolerate partial failures gracefully (this is why we need Replication). |

### 1.3 Real-World Scenario: What Happens When You Google Something?

You type "Best coffee shops near me" into Google. Your request doesn't travel to "one giant Google computer." Here is the distributed reality:

```
  Your Browser
       │
       ▼
┌──────────────┐
│ Load Balancer │ → Routes to the closest regional data center
└──────┬───────┘
       ▼
┌──────────────┐     ┌─────────────────┐     ┌──────────────┐
│  Web Server  │────▶│ Location Service │     │  Ads Service │
│  (Orchestrator)│────▶│ (Where are you?) │     │ (Sponsored)  │
│              │────▶│ Search Index     │     │              │
└──────┬───────┘     └─────────────────┘     └──────────────┘
       │
       │  All 3 microservices return data in parallel
       ▼
  Final HTML Page (delivered in ~200ms)
```

**Key Insight:** Google doesn't have 1 service. It has *thousands* of independent microservices — Search, Ads, Location, Maps, YouTube — all communicating over an internal network. This is a distributed system.

## 2. Single Node vs Multi-Node Architecture

Every system on Earth starts as a single node. You only distribute when physics forces you to.

### 2.1 Single Node Architecture

**What it is:** Everything — the web application, the database, the cache, the background workers — runs on one physical server or virtual machine (VM).

**How to scale it:** **Vertical Scaling (Scaling Up)** — buy a bigger server with more RAM and a faster CPU.

```
┌──────────────────────────────────┐
│         SINGLE SERVER            │
│                                  │
│  ┌──────────┐  ┌──────────────┐  │
│  │ Web App  │  │   Database   │  │
│  │ (Node.js)│  │ (PostgreSQL) │  │
│  └──────────┘  └──────────────┘  │
│                                  │
│  ┌──────────┐  ┌──────────────┐  │
│  │  Cache   │  │   Workers    │  │
│  │ (Redis)  │  │ (Background) │  │
│  └──────────┘  └──────────────┘  │
│                                  │
│         16 GB RAM, 4-Core CPU    │
└──────────────────────────────────┘
```

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| **Simplicity:** One server to deploy, monitor, and debug. `ssh` into it and you see everything. | **Single Point of Failure (SPOF):** If the motherboard fries, your entire business goes offline instantly. |
| **Data Consistency:** ACID transactions are trivial because the app and DB share the same hard drive. | **Hardware Ceiling:** You can only put so much RAM on a motherboard. Physics has a limit. |
| **Zero Network Latency:** The app talks to the DB via local memory, not over internet cables. | **Downtime for Upgrades:** To swap the CPU, you must physically shut the machine down. |

### 2.2 Multi-Node Architecture (Distributed System)

**What it is:** The application is split across tens, hundreds, or thousands of servers connected by a network.

**How to scale it:** **Horizontal Scaling (Scaling Out)** — buy 10 more cheap, identical servers and put a Load Balancer in front.

```
                    ┌──────────────┐
    Users ─────────▶│ Load Balancer│
                    └──────┬───────┘
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Server 1 │ │ Server 2 │ │ Server 3 │
        │ (App)    │ │ (App)    │ │ (App)    │
        └──────────┘ └──────────┘ └──────────┘
              │            │            │
              └────────────┼────────────┘
                           ▼
                    ┌──────────────┐
                    │   Database   │
                    │ (Shared DB)  │
                    └──────────────┘
```

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| **Limitless Scale:** Need 10x traffic capacity? Spin up 10x more cheap servers. | **Extreme Complexity:** Deployments require Docker + Kubernetes. Debugging needs distributed tracing (Jaeger/Zipkin). |
| **High Availability:** Server #4 catches fire? The Load Balancer instantly reroutes to the other 99 servers. Zero downtime. | **Network Failures:** You must now worry about packet loss, network partitions, and latency *between your own servers*. |
| **Geographic Distribution:** Place nodes in New York, London, and Tokyo for globally fast response times. | **Data Inconsistency:** Syncing databases across 3 continents is a massive CS problem (See: CAP Theorem). |

## 3. Monolith vs Microservices Architecture

"Nodes" refer to the physical hardware. "Architecture" refers to how the **codebase** is organized.

### 3.1 Monolithic Architecture

**What it is:** The entire application code — Authentication, Payment, Reviews, Search — is compiled into a **single codebase** and deployed as a single unit (one `.jar`, one Docker image).

> **💡 Analogy:** A monolith is a Swiss Army Knife. Everything in one tool. Convenient, but if the screwdriver breaks, the whole knife is useless.

**Real-World Examples:** Early Twitter, early Netflix, early Amazon (all started as monoliths).

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| **Simple Deployment:** Deploy one file. One CI/CD pipeline. One server. | **Slower Development at Scale:** 100 engineers pushing code to one repo = daily merge conflicts and broken builds. |
| **Zero Network Overhead:** `paymentService.charge(user)` is an in-memory function call (nanoseconds). | **Coupled Failures:** A memory leak in the "Reviews" module crashes the *entire* app — meaning Payments also goes down. |
| **Easy Local Testing:** `npm start` boots the whole app on your laptop. | **Locked Tech Stack:** Chose Java 5 years ago? You can't suddenly write the new AI feature in Python. |

### 3.2 Microservices Architecture

**What it is:** The application is split into many small, independent services. Each service does **exactly one thing** (e.g., `PaymentService`, `UserService`, `ReviewService`). They communicate over the network via REST APIs or gRPC.

> **💡 Analogy:** Microservices are a team of specialists with separate toolboxes. The electrician can upgrade independently without affecting the plumber.

**Real-World Examples:** Netflix (~1000 microservices), Amazon, Uber, Spotify.

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| **Independent Scaling:** Black Friday? Scale `PaymentService` to 50 instances. Leave `ReviewService` at 3. | **Network Latency Everywhere:** `paymentService.charge(user)` is now an HTTP call (milliseconds, not nanoseconds), and it can fail. |
| **Fault Isolation:** `ReviewService` crashes? Users can still checkout and pay. | **Data Consistency Nightmare:** Each service owns its own DB. No foreign keys across services. |
| **Tech Agnosticism:** AI team uses Python, Backend uses Go, Frontend uses Node.js. | **Debugging Hell:** A single bug might span 5 services. You need Distributed Tracing (Jaeger). |

### 3.3 When to Use Which?

| Situation | Use Monolith | Use Microservices |
| :--- | :--- | :--- |
| Team size < 10 engineers | ✅ | ❌ Overkill |
| MVP / Startup / Prototype | ✅ | ❌ |
| Team size > 50 engineers | ❌ | ✅ |
| Individual features need independent scaling | ❌ | ✅ |
| You need 99.99% uptime | ❌ | ✅ |

> **💡 The Golden Rule:** Start with a Monolith. Evolve to Microservices *only when the pain of the monolith becomes unbearable.* Amazon, Netflix, and Twitter all started as monoliths and migrated as they grew.

## 4. N-Tier Architecture (1-Tier, 2-Tier, 3-Tier)

"Tiers" refer to the **logical separation** of concerns between the Presentation, Processing, and Data layers. This is about *how responsibilities are divided*, not how many physical servers you have.

### 4.1 1-Tier Architecture
*   **What it is:** UI + Business Logic + Database all sit on the **same machine**.
*   **Example:** Microsoft Word saving `.docx` files to your local hard drive. A standalone desktop game.
*   **Verdict:** Zero scalability. Zero multi-user capability. Fine for consumer desktop apps, terrible for the web.

### 4.2 2-Tier Architecture (Client-Server)
*   **What it is:** The Client (Tier 1) contains the UI + some business logic. It talks **directly** to a Database Server (Tier 2) over a network.
*   **Example:** An internal company tool installed on every employee's laptop, directly querying a centralized Oracle database.
*   **The Fatal Flaw:** If you fix a bug in the business logic, you must push a forced update to every single laptop. Also, security is terrible — the client has direct access to the DB.

### 4.3 3-Tier Architecture (The Web Standard) ⭐
*   **What it is:** The gold standard of modern web development.

```
┌─────────────────┐    HTTP     ┌─────────────────┐    SQL      ┌─────────────────┐
│   TIER 1        │ ──────────▶ │   TIER 2        │ ──────────▶ │   TIER 3        │
│   Presentation  │             │   Application   │             │   Data          │
│   (React/Vue    │             │   (Node.js/Java  │             │   (PostgreSQL/  │
│    in browser)  │ ◀────────── │    API Server)   │ ◀────────── │    MongoDB)     │
└─────────────────┘   JSON      └─────────────────┘   Results   └─────────────────┘
```

*   **Why 3-Tier Wins:**
    1.  **Security:** The browser *never* touches the database directly. All requests go through the API (Tier 2), which validates, authenticates, and sanitizes everything.
    2.  **Easy Updates:** Fix a bug in the backend? Deploy to Tier 2. Users don't need to update anything.
    3.  **Independent Scaling:** Your frontend is a static React build on a CDN. Your API scales horizontally behind a Load Balancer. Your DB scales vertically.

### 4.4 N-Tier (Multi-Tier) Architecture
*   **What it is:** An extension of 3-Tier where the Application Tier is further split into many sub-tiers for extreme scale and separation.
*   **Example:** Client (Tier 1) → CDN (Tier 2) → API Gateway (Tier 3) → Auth Service (Tier 4) → Business Logic (Tier 5) → Cache (Tier 6) → Database (Tier 7).

## 5. The End-to-End Request Lifecycle

To truly master System Design, you must understand the **exact journey a single HTTP request takes** from the moment a user presses Enter to the moment they see the page.

### The Journey: "User types `www.amazon.com` and hits Enter"

```
Step 1: DNS          Step 2: CDN          Step 3: LB           Step 4: Gateway
Browser ──────▶ DNS Server ──────▶ CDN Edge ──────▶ Load Balancer ──────▶ API Gateway
"What IP?"     "205.251.242.103"  "Here's the     "Send to         "Authenticated?
                                   logo + CSS"     Server #7"       Rate limit OK?"
```

| Step | What Happens | Time |
| :--- | :--- | :--- |
| **1. DNS Resolution** | Browser asks DNS: *"What is the IP address for amazon.com?"* DNS replies: `205.251.242.103`. | ~50ms |
| **2. CDN Hit (Static)** | Before reaching Amazon's datacenter, the request hits a nearby CDN Edge Server. It instantly returns static files — the Amazon logo, CSS stylesheets, React JS bundles. | ~20ms |
| **3. Load Balancer** | For dynamic data (the personalized shopping cart), the request travels to the datacenter and hits the **Load Balancer**. | ~5ms |
| **4. API Gateway** | The LB routes to an **API Gateway**. The Gateway checks authentication (is this user logged in?), applies rate limiting (is this user spamming?), and routes to the correct microservice. | ~10ms |
| **5. Microservice** | The `HomepageService` receives the request. It needs the user's past purchases to show personalized recommendations. | ~1ms |
| **6. Cache Check** | The service asks **Redis Cache**: *"Do you have data for User #123?"* If **YES** (Cache Hit) → return instantly. If **NO** (Cache Miss) → proceed to step 7. | ~0.5ms |
| **7. Database Query** | Cache Miss: The service queries **PostgreSQL**. The DB reads from disk, finds the user profile, and returns it. The service *immediately saves this result in Cache for next time*. | ~50ms |
| **8. Response** | The microservice builds the JSON, sends it back through the API Gateway → Load Balancer → Internet → Browser. React renders the page. | ~15ms |
| | **Total Round Trip** | **~150ms** |

> **💡 Interview Power Move:** If an interviewer asks you to "Design X," start by drawing this exact lifecycle diagram on the whiteboard. It immediately shows you understand how real systems work.

## 6. Vertical vs Horizontal Scaling

When a single server reaches its limit, you have exactly two directions to scale: **Up** or **Out**.

### 6.1 Vertical Scaling (Scaling Up)

**What it is:** Making your *existing* server bigger and more powerful.

*   **Example:** Upgrading your database server from 16GB RAM → 256GB RAM, or swapping a 4-core CPU for a 64-core CPU.
*   **Pros:** Zero code changes required. No architectural redesign. Just more hardware.
*   **Cons:**
    1.  **Hardware Ceiling:** There is a physical limit. You can't put 10TB of RAM on a motherboard that supports 2TB.
    2.  **Downtime Required:** You must shut the server off to swap hardware (unless you use cloud VMs, which can resize with a brief restart).
    3.  **Cost Curve:** Going from 16GB → 32GB RAM is cheap. Going from 512GB → 1TB is astronomically expensive. The price grows exponentially, not linearly.

### 6.2 Horizontal Scaling (Scaling Out)

**What it is:** Adding *more* servers to handle the load collectively.

*   **Example:** Instead of one 256GB server, you deploy ten 32GB servers behind a Load Balancer. Total capacity: 320GB of distributed RAM.
*   **Pros:** Theoretically infinite scalability. Just keep adding cheap commodity servers.
*   **Cons:**
    1.  **Code Complexity:** Your application must be *stateless* (Section 7) to work correctly across multiple servers.
    2.  **Data Distribution:** If your DB is horizontally scaled, you now deal with Sharding, cross-shard queries, and eventual consistency.
    3.  **Operational Overhead:** You need Docker, Kubernetes, Load Balancers, Service Discovery, Health Checks.

### 6.3 The Comparison Table

| Aspect | Vertical (Scale Up) | Horizontal (Scale Out) |
| :--- | :--- | :--- |
| **Cost** | Exponential (diminishing returns) | Linear (each new server costs the same) |
| **Downtime** | Required for hardware swap | Zero downtime (add servers to the pool live) |
| **Complexity** | None | High (distributed system problems) |
| **Limit** | Physics (biggest server on Earth) | Infinite (keep adding servers) |
| **Best For** | Databases (scale vertically as long as possible) | App/Web Servers (scale horizontally from Day 1) |

> **💡 The Golden Rule:** Scale your **App Servers horizontally** from Day 1. Scale your **Relational Database vertically** for as long as humanly possible, because horizontally sharding a SQL database is an engineering nightmare.

## 7. Stateful vs Stateless Services

When you horizontally scale your app servers, you immediately run into the **Session Problem**.

### 7.1 Stateful Services (The Problem)

**What it is:** The server stores "state" (data about the client) in its own **local memory or disk** between requests.

> **💡 The Disaster Scenario:**
> 1. You log into **Server A**. Server A saves `session_id = abc123` in its local RAM.
> 2. You click "Add to Cart." The **Load Balancer** routes this request to **Server B** (round robin).
> 3. Server B checks its local RAM — it has no idea who you are. **"Please log in again."** 😡
>
> **Verdict:** Stateful architecture is **fundamentally incompatible** with horizontal scaling.

### 7.2 Stateless Services (The Solution) ⭐

**What it is:** Every HTTP request from the client contains **ALL the information** the server needs. The server stores **zero** session data locally. It forgets you exist after every response.

> **💡 How it works:**
> 1. You log in. The server creates a **JWT (JSON Web Token)** containing `{ user_id: 123, role: "admin" }`, cryptographically signs it, and sends it back to the browser.
> 2. The server **immediately forgets you exist**.
> 3. On your next request, you send the JWT in the `Authorization` header. The Load Balancer sends you to **Server B**. Server B reads the JWT, verifies the cryptographic signature, and knows *exactly* who you are. **No session lookup needed.**
>
> **Verdict:** Stateless architecture is **mandatory** for horizontal scaling. Any of your 100 servers can handle any user's request perfectly.

### 7.3 But Where Does the State Go?

If you need persistent state (like a shopping cart), **do NOT store it on the web server**. Store it in a **centralized external datastore**.

```
                    ┌──────────┐
              ┌────▶│ Server A │────┐
              │     └──────────┘    │
┌──────────┐  │     ┌──────────┐    │     ┌───────────────┐
│  Client  │──┼────▶│ Server B │────┼────▶│  Redis Cache  │
└──────────┘  │     └──────────┘    │     │ (Session/Cart │
              │     ┌──────────┐    │     │  data lives   │
              └────▶│ Server C │────┘     │  HERE)        │
                    └──────────┘          └───────────────┘
```

Servers A, B, C are all **stateless**. They all ask Redis for the cart data. Problem solved.

## 8. Latency vs Throughput

These are the two most important — and most frequently confused — metrics in System Design. They are **related but NOT the same**.

### 8.1 Latency

**Definition:** The time it takes for a **single request** to travel from sender → server → back to sender.
**Unit:** Milliseconds (ms)
**Analogy:** How fast a *single car* travels from City A to City B.

*   **Low Latency is critical for:** Real-time multiplayer gaming (Call of Duty), High-frequency trading, Video calls (Zoom).
*   **How to reduce latency:**
    *   Move servers closer to users → **CDNs, Edge Computing**
    *   Avoid disk reads → **In-memory caching (Redis)**
    *   Optimize algorithms → O(N) to O(log N) or O(1)
    *   Use faster protocols → **UDP** instead of TCP (for gaming)

### 8.2 Throughput

**Definition:** The **volume of work** a system can process in a given time window.
**Unit:** Requests Per Second (RPS), Queries Per Second (QPS), MB/s
**Analogy:** How many *total cars* can travel from City A to City B in one hour. A 10-lane highway at 40mph has enormous throughput even though each car is slow.

*   **High Throughput is critical for:** Video uploading (YouTube), Big Data analytics, Batch processing (nightly bank reconciliation).
*   **How to increase throughput:**
    *   Add more servers → **Horizontal scaling**
    *   Process asynchronously → **Message Queues (Kafka)**
    *   Increase parallelism → **Connection pools, multi-threading**

### 8.3 The Trade-Off: The Ferrari vs The Bus

If you need to move 1,000 people from New York to Boston:

| Strategy | Latency | Throughput |
| :--- | :--- | :--- |
| **The Ferrari** — Put 1 person in a 200mph car | ✅ Extremely Low (1 hour) | ❌ Terrible (1 person/trip) |
| **The Bus** — Put all 1,000 on a 30mph bus | ❌ Extremely High (6 hours) | ✅ Amazing (1,000 people/trip) |

> **💡 Interview Tip:** When asked to "Design System X," immediately ask the interviewer: *"What is more important — latency or throughput?"*
> *   **Stock Trading App** → Sacrifice throughput for guaranteed 10ms latency.
> *   **YouTube Video Upload** → Sacrifice latency (10 min to process is fine) for massive throughput (500 hours uploaded/minute).

## 9. Identifying Database Bottlenecks

Your horizontally scaled, stateless, load-balanced web tier is meaningless if the **database is choking**. Databases are almost always the bottleneck.

### 9.1 The 4 Types of Database Bottlenecks

| Bottleneck | Symptom | Root Cause | Solution |
| :--- | :--- | :--- | :--- |
| **Read-Heavy** | High CPU, slow `SELECT` queries | Millions of people viewing a viral tweet simultaneously | Add a **Cache** (Redis). If insufficient, add **Read-Replicas**. |
| **Write-Heavy** | High disk I/O, table locking, slow `INSERT`s | Thousands of IoT sensors logging data every second | Buffer writes in a **Message Queue** (Kafka). Use NoSQL (Cassandra) optimized for append-only writes. |
| **Storage** | "Out of Disk" errors, exponentially slower queries | B-Tree indexes too large for RAM, physical disk full | **Shard** the table across multiple servers (Users A-M → Server 1, N-Z → Server 2). |
| **Connection Pool** | "Too many clients" / "Connection Refused" errors | 100 App Servers opening 10,000 TCP connections to one DB | Use a **Connection Pooler** (PgBouncer for PostgreSQL). |

> **💡 Diagnostic Framework:** When the DB is slow, don't guess. Run `EXPLAIN ANALYZE` on the slow query. Check if it's doing a Seq Scan (missing index), a slow JOIN (missing index on FK), or locking (concurrent write contention).

## 10. The 6 Pillars of Optimization

When a system is slow or failing under load, architects reach for these six foundational tools. Master these, and you can solve 90% of System Design interview questions.

### 10.1 Cache Reduces DB Load

**The Problem:** 100,000 people open the Amazon homepage simultaneously, firing 100,000 `SELECT * FROM products WHERE featured = true` queries. The PostgreSQL server melts.

**The Solution:** Put a **Cache** (Redis) in front of the database. A cache stores data in blazing-fast RAM instead of a slow hard disk.

```
┌──────────┐    "Do you have it?"    ┌─────────┐
│ App      │ ─────────────────────▶  │  Redis  │
│ Server   │ ◀─── Cache HIT (0.5ms)  │  Cache  │  ← 99% of requests stop here
│          │                         └─────────┘
│          │    Cache MISS (1%)           │
│          │ ─────────────────────▶  ┌─────────┐
│          │ ◀─── DB Response (50ms) │  Postgres│  ← Only 1% reach the DB
└──────────┘   (save to cache)       └─────────┘
```

**Result:** DB load drops by 99%. Queries go from 50ms → 0.5ms.

**Cache Strategies:**
*   **Cache-Aside (Lazy Loading):** App checks cache first. On miss, queries DB, then writes result to cache. Most common pattern.
*   **Write-Through:** Every write goes to cache AND DB simultaneously. Ensures cache is always up-to-date but slower writes.
*   **TTL (Time-To-Live):** Cache entries expire after N seconds, forcing a fresh DB read. Prevents serving stale data.

---

### 10.2 Load Balancer Distributes Traffic

**The Problem:** You have 10 identical web servers, but all users are hitting Server #1. It crashes from exhaustion while Servers #2-10 sit completely idle.

**The Solution:** Place a **Load Balancer** (NGINX, HAProxy, AWS ALB) between users and servers. Users only ever talk to the Load Balancer.

**Load Balancing Algorithms:**

| Algorithm | How it Works | Best For |
| :--- | :--- | :--- |
| **Round Robin** | Sends requests to servers in sequential order: 1, 2, 3, 1, 2, 3... | Equal-power servers with uniform request cost. |
| **Least Connections** | Sends to whichever server currently has the fewest active connections. | When some requests are fast (GET) and some are slow (file upload). |
| **IP Hash** | Hashes the user's IP address to deterministically route them to the same server every time. | When you need sticky sessions (stateful apps — avoid this if possible). |
| **Weighted Round Robin** | Some servers get more traffic. Server A (8 CPU) gets 4x the traffic of Server B (2 CPU). | Heterogeneous server fleet. |

**Health Checks:** The LB continuously pings each server (e.g., `GET /health` every 10 seconds). If Server #5 stops responding, the LB instantly removes it from the pool. Zero downtime for users.

---

### 10.3 Async Processing Reduces Blocking

**The Problem:** A user uploads a 10GB 4K video. The Web Server starts compressing it (10 minutes). During this time, the server is **frozen** — unable to handle any other users. This is **Synchronous Blocking**.

**The Solution:** Decouple heavy work using a **Message Queue** (Kafka, RabbitMQ) and **Background Worker Nodes**.

```
┌──────────┐  1. Upload video   ┌────────────┐
│  User    │ ──────────────────▶│  Web Server│
└──────────┘                    │            │──▶ 2. Store video in S3
    ▲                           │            │──▶ 3. Drop message in Kafka:
    │                           │            │      "Video #123 needs processing"
    │  4. Instantly respond:    │            │
    │  "Upload successful!      └────────────┘
    │   Processing your video."
    │                           ┌────────────┐
    │                           │   Kafka    │
    │                           │   Queue    │
    │                           └─────┬──────┘
    │                                 │
    │                           ┌─────▼──────┐
    │  6. Notify via WebSocket  │   Worker   │ ← 5. Picks up message,
    └───────────────────────────│   Server   │    compresses video for
                                │            │    10 minutes in background
                                └────────────┘
```

**Result:** The user-facing API responds in milliseconds. Heavy computation happens asynchronously in the background. The API never blocks.

---

### 10.4 Indexing Makes Reads Faster

**The Problem:** 100 million rows in your `users` table. `SELECT * FROM users WHERE email = 'john@doe.com'` scans ALL 100 million rows (O(N)). Takes 4 seconds.

**The Solution:** Create a **B-Tree Index** on the `email` column.

```sql
CREATE INDEX idx_users_email ON users(email);
```

**How it works:** The database builds a balanced tree (B-Tree) that sorts all 100 million emails alphabetically and stores pointers to the physical disk location of each row.

**Result:** The query now traverses the tree in O(log N) time. Finding 1 row among 100 million takes ~27 steps instead of 100,000,000 steps. **Query drops from 4 seconds → 3 milliseconds.**

> **⚡ Trade-off:** Indexes make READs faster but WRITEs slower (every INSERT/UPDATE must also update the B-Tree). Only index columns you frequently `WHERE`, `JOIN ON`, or `ORDER BY`.

### 10.5 Replication Improves Availability

**The Problem:** Your single database server is in Houston. An earthquake destroys the data center. Your app is offline AND your data is permanently lost.

**The Solution:** **Database Replication** (Master → Replica architecture).

```
                    ┌────────────────────┐
    Writes ────────▶│   MASTER (Primary) │
                    └────────┬───────────┘
                             │ Automatic Replication
                    ┌────────┴───────────┐
                    ▼                    ▼
            ┌──────────────┐    ┌──────────────┐
 Reads ────▶│  REPLICA #1  │    │  REPLICA #2  │◀──── Reads
            │ (New York)   │    │ (London)     │
            └──────────────┘    └──────────────┘
```

**Benefits:**
1.  **High Availability:** Master crashes → Replica #1 promotes itself to the new Master automatically. Zero downtime.
2.  **Read Scalability:** Route all `SELECT` queries to Replicas. Master focuses exclusively on writes. Massive read capacity increase.
3.  **Geographic Performance:** Replicas in different regions serve nearby users faster.

**Does NOT solve:** Storage limits (every server holds the FULL dataset).

---

### 10.6 Sharding Improves Scalability

**The Problem:** 50 billion rows of IoT sensor data. A single 15TB hard drive physically cannot hold it all. The B-Tree index exceeds RAM capacity.

**The Solution:** **Database Sharding** (Horizontal Partitioning) — slice the table across multiple independent database servers.

```
        ┌─────────────────────────┐
        │     SHARD ROUTER        │ ← Determines which shard based on shard key
        └──────┬──────┬──────┬────┘
               │      │      │
        ┌──────┘      │      └──────┐
        ▼             ▼              ▼
  ┌───────────┐ ┌───────────┐ ┌───────────┐
  │  SHARD 1  │ │  SHARD 2  │ │  SHARD 3  │
  │ Users A-H │ │ Users I-P │ │ Users Q-Z │
  │  (5TB)    │ │  (5TB)    │ │  (5TB)    │
  └───────────┘ └───────────┘ └───────────┘
```

**Benefits:** Infinite storage and write scalability. Each shard only handles its portion.

> **⚡ Trade-off:** Sharding is the **most complex** concept in system design. Cross-shard queries (e.g., "Find all friends of User A across all shards") require scatter-gather operations. Always try Caching + Replication BEFORE resorting to Sharding.

## 11. Intuition & Problem Solving Frameworks

### 11.1 "The Bottleneck Hunter" Framework

When asked to scale a system, **never jump to microservices or sharding immediately**. Instead, mentally start with a single server and let it break. Ask: *"What component cried first?"*

| What Cried? | Symptom | Reach For This Tool |
| :--- | :--- | :--- |
| **CPU** | Heavy computation (image processing, AI inference) | **Horizontal scaling** — add more app servers |
| **RAM** | Too many concurrent connections, in-memory data structures overflow | **Redis/Memcached** — offload to external cache |
| **Disk I/O (Read)** | Slow SELECT, high read latency | **Cache** (Redis) + **Read Replicas** |
| **Disk I/O (Write)** | Slow INSERT, table locks, write contention | **Message Queue** (Kafka) for async writes |
| **Network** | Pushing GBs of images/video, choking the router | **CDN** + **S3** for static asset offloading |
| **Load Distribution** | One server at 100% CPU, others at 0% | **Load Balancer** (NGINX) |
| **Storage** | Disk full, indexes exceed RAM | **Sharding** (last resort) |

### 11.2 The "When to Distribute" Checklist

Before splitting a monolith into microservices, honestly ask:

- [ ] Does my team have **> 20 engineers** constantly stepping on each other's code?
- [ ] Is a single feature (e.g., Video Transcoding) consuming **90% of server resources**, starving other features?
- [ ] Do I need **99.99% uptime** where even a 5-minute reboot costs thousands?
- [ ] Do different parts need **different tech stacks** (Python for AI, Go for API)?

*If you checked < 2 boxes, stay with the monolith. Don't over-engineer.*

## 12. Interview Questions & Scenarios

### ❓ Q1: "Our single-node database is running out of disk space. Scale vertically or horizontally?"
*   **Bad Answer:** "Shard it horizontally for infinite storage."
*   **Good Answer:** "It depends on growth projections. Sharding breaks ACID guarantees across partitions and adds massive complexity. If we just need 2x space right now, accept 15 minutes of scheduled downtime and vertically scale with a larger disk. If we expect 100x growth, plan for sharding — but try archiving old data and adding read replicas first."

### ❓ Q2: "Design a system where latency matters far more than throughput."
*   **Answer:** "In-memory everything. Use Redis/Memcached as the primary datastore instead of disk-based SQL. Place servers at the network edge via CDNs. Use UDP over TCP where possible. Keep servers underutilized (~50% CPU) because latency spikes exponentially as servers approach 100% utilization due to request queueing."

### ❓ Q3: "Multiplayer video game — Latency or Throughput? How?"
*   **Answer:** "Latency is king. A 300ms ping means you're dead before you see the enemy. Throughput per user is actually low — tiny packets of XYZ coordinates, not 4K images.
*   **Protocol:** Use **UDP** (fire packets blindly, no acknowledgement) instead of TCP (waits for confirmation per packet). If a position update from 3 seconds ago is lost, you don't care —  you only want the latest one."

### ❓ Q4: "Why doesn't Amazon use one $500K supercomputer?"
*   **Answer:**
    1.  **SPOF:** Even a $500K machine can have its power cord unplugged. You need **Replication** for availability.
    2.  **Physics:** Speed of light through fiber from Texas to Sydney = 150ms minimum. Users need **geographically distributed** nodes.
    3.  **Hardware Ceiling:** No single CPU exists that can handle millions of Prime Day transactions/sec. You must **horizontally scale** and **shard**.

### ❓ Q5: "A 3rd-party bank API takes 30 seconds to respond. How do you prevent your servers from freezing?"
*   **Answer:** "**Async Processing.** When the user clicks 'Pay':
    1.  Web Server drops a `PaymentPending` event into a **Kafka Queue**.
    2.  Immediately returns `HTTP 202 Accepted` — *"Processing your payment."*
    3.  A background **Worker Node** picks up the event, calls the slow bank API, waits 30 seconds, and updates the DB.
    4.  Frontend gets notified via **WebSocket** or **Long Polling** when the payment completes."

### ❓ Q6: "Stateful or Stateless? Which is better for a web app?"
*   **Answer:** "**Stateless, always.** In a stateful architecture, if the Load Balancer sends you to a different server, you lose your session. In stateless, every request carries its own auth token (JWT), so any of 100 servers can handle any user's request. State that *must* persist (shopping carts) goes into an external store like **Redis**, not the web server's memory."

## 13. Applying the Fundamentals — Design Problems

Now that you know the 6 Pillars, here are 3 classic design problems solved using **only** the fundamentals from this guide.

### 🏗️ Problem 1: Design a Global Image Hosting Site (like Imgur)

**Challenge:** Users worldwide upload millions of heavy images. Storing them in a DB kills it. Users in Tokyo loading images from a New York server get terrible latency.

**Architecture (Using Fundamentals):**

| Layer | Tool Used | Why |
| :--- | :--- | :--- |
| App Servers | **Horizontal scaling** + **Load Balancer** | Handle millions of concurrent uploads |
| Image Storage | **Amazon S3** + **Global CDN** | Never store binary files in a relational DB. CDN serves images from edge servers near the user in 5ms |
| Metadata DB | **PostgreSQL** with **Master-Replica** setup | Store `image_id, uploader_id, upload_date, likes`. Replicas handle all read-heavy "Top Images" queries |
| Upload Processing | **Kafka Queue** + **Workers** | Thumbnail generation, compression happen asynchronously. API responds instantly |

---

### 🏗️ Problem 2: Design a Viral Ticket Buying System (like Ticketmaster)

**Challenge:** Taylor Swift tickets on sale. 500,000 people click "Buy" at the same millisecond. 500K concurrent writes will crash any SQL database.

**Architecture (Using Fundamentals):**

| Layer | Tool Used | Why |
| :--- | :--- | :--- |
| Browsing Page | **Redis Cache** | The "Tickets Available" page is cached. 99% of browsing traffic never hits the DB. TTL: 5 seconds |
| Buy Requests | **Kafka Message Queue** | 500K "Buy" requests are queued, not sent directly to DB |
| Processing | **Worker Pool** (controlled speed) | Workers process exactly 1,000 ticket purchases/sec from the queue. DB never overwhelmed |
| Inventory | **PostgreSQL** with **Pessimistic Locking** | `SELECT ... FOR UPDATE` prevents double-selling the same ticket |

---

### 🏗️ Problem 3: Design a Massive URL Shortener (like Bitly)

**Challenge:** Generate `bit.ly/xyz` for long URLs. 10 million clicks/day — extreme read-heavy traffic.

**Architecture (Using Fundamentals):**

| Layer | Tool Used | Why |
| :--- | :--- | :--- |
| Read Path | **Redis Cache** | 99% of clicks hit the same popular links. Cache serves them in 0.3ms |
| Cache Miss | **PostgreSQL** with **B-Tree Index** on `short_url` column | The 1% of cache misses are resolved in O(log N) — milliseconds |
| Throughput | **50 Horizontal Servers** behind **Load Balancer** | Redirect logic is trivial — needs minimal CPU. Scale out cheaply for massive RPS |
| Write Path | **PostgreSQL** (single master) | Write volume is low (creating short URLs is rare compared to clicking them). Single master is sufficient |
