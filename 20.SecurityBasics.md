## 1. Authentication vs Authorization

| | Authentication (AuthN) | Authorization (AuthZ) |
| :--- | :--- | :--- |
| **Question** | **"WHO are you?"** | **"WHAT are you allowed to do?"** |
| **Analogy** | Showing your ID at the door | Your ID says "VIP" so you can enter the backstage |
| **Process** | Login with username + password, or OAuth, or SSO | Check user's role/permissions against the resource |
| **When** | Before anything else | After authentication |
| **Example** | Login → "You are user_123" | "User_123 has role: ADMIN → can delete posts" |

```
Request flow:

  1. User sends: POST /login { email, password }
  2. Server: "Credentials valid. You are user_123." (Authentication ✅)
  3. Server returns: JWT token
  
  4. User sends: DELETE /api/posts/456  (with JWT token)
  5. Server checks JWT: "Token valid. user_123." (Authentication ✅)
  6. Server checks permissions: "Is user_123 allowed to delete post 456?" 
     → user_123 is the author → (Authorization ✅) → Delete post.
     → user_123 is NOT the author → 403 Forbidden (Authorization ❌)
```

## 2. OAuth 2.0 — The Industry Standard

### 2.1 What Problem Does OAuth Solve?

You want to log into App X using your Google account. **Without OAuth**, you'd give App X your Google password — terrible! OAuth lets App X verify your identity through Google **without ever seeing your password**.

### 2.2 OAuth 2.0 Authorization Code Flow

```
┌──────┐                ┌────────────┐                 ┌──────────┐
│ User │                │  App (X)   │                 │  Google  │
│      │                │  (Client)  │                 │ (Auth    │
│      │                │            │                 │  Server) │
└──┬───┘                └─────┬──────┘                 └────┬─────┘
   │  1. Click "Login          │                            │
   │     with Google"          │                            │
   │──────────────────────────▶│                            │
   │                           │  2. Redirect to Google     │
   │                           │     with client_id +       │
   │                           │     redirect_uri           │
   │◀──────────────────────────│───────────────────────────▶│
   │                           │                            │
   │  3. Google asks: "Allow   │                            │
   │     App X to access       │                            │
   │     your profile?"        │                            │
   │◀───────────────────────────────────────────────────────│
   │                           │                            │
   │  4. User clicks "Allow"   │                            │
   │───────────────────────────────────────────────────────▶│
   │                           │                            │
   │  5. Google redirects back │  6. Auth Code              │
   │     to App X with         │◀───────────────────────────│
   │     auth_code             │                            │
   │                           │  7. App X exchanges        │
   │                           │     auth_code + secret     │
   │                           │     for ACCESS TOKEN       │
   │                           │───────────────────────────▶│
   │                           │                            │
   │                           │  8. Access Token returned  │
   │                           │◀───────────────────────────│
   │                           │                            │
   │                           │  9. Use Access Token to    │
   │                           │     call Google APIs       │
   │                           │───────────────────────────▶│
   │                           │  10. User profile data     │
   │  11. Logged in! ✅        │◀───────────────────────────│
   │◀──────────────────────────│                            │
```

### 2.3 OAuth Key Concepts

| Term | What It Is |
| :--- | :--- |
| **Client** | The application requesting access (App X) |
| **Resource Owner** | The user who owns the data |
| **Authorization Server** | The server issuing tokens (Google) |
| **Resource Server** | The API hosting the user's data (Google APIs) |
| **Access Token** | Short-lived token to access APIs (expires in 1 hour) |
| **Refresh Token** | Long-lived token to get new access tokens (expires in 30 days) |
| **Scope** | What permissions are requested (`profile`, `email`, `read:repos`) |

## 3. JWT (JSON Web Tokens)

### 3.1 What Is a JWT?

A **self-contained token** that holds user information (claims). The server can verify it without querying a database — the signature proves it wasn't tampered with.

### 3.2 JWT Structure

```
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjoxMjN9.SflKxw...
├───── HEADER ──────┤├────── PAYLOAD ───────┤├─ SIGNATURE ─┤

HEADER (base64):
  { "alg": "HS256", "typ": "JWT" }

PAYLOAD (base64):
  { "user_id": 123, "role": "admin", "exp": 1709312460 }

SIGNATURE:
  HMAC-SHA256(base64(header) + "." + base64(payload), SECRET_KEY)
```

### 3.3 JWT vs Session-Based Auth

| Feature | JWT (Token-based) | Session (Cookie-based) |
| :--- | :--- | :--- |
| **Storage** | Client stores the token (localStorage / cookie) | Server stores session in Redis/DB |
| **Stateless** | ✅ Server verifies signature only (no DB lookup) | ❌ Server must lookup session ID in Redis |
| **Scalability** | ✅ Any server can verify (no shared state) | ⚠️ Need centralized session store (Redis) |
| **Revocation** | ❌ Hard! Can't invalidate a JWT before it expires | ✅ Easy — delete session from Redis |
| **Size** | Larger (contains all claims) | Small (just a session ID) |
| **Best for** | APIs, microservices, mobile apps | Traditional web apps, need instant revocation |

### 3.4 JWT Security Best Practices

```
✅ DO:
  → Set short expiry (15 min access token + refresh token pattern)
  → Use HTTPS only (JWT in plain HTTP = stolen credentials)
  → Store in httpOnly cookie (not localStorage — XSS vulnerable)
  → Validate signature on EVERY request
  → Include `exp` (expiry) and `iat` (issued at) claims

❌ DON'T:
  → Store sensitive data in payload (it's base64 encoded, NOT encrypted)
  → Use JWT for sessions if you need instant logout/revocation
  → Use algorithm "none" (critical vulnerability — always validate alg)
```

## 4. API Security Patterns

### 4.1 API Keys

```
GET /api/weather?city=Delhi
Header: X-API-Key: sk_live_abc123def456

Use for: Server-to-server communication, developer API access.
NOT for: User authentication (API keys don't represent a user).
```

### 4.2 Rate Limiting (see Guide 10)

```
Limit: 100 requests per minute per API key.
→ Protects from abuse, DDoS, and runaway scripts.
```

### 4.3 Input Validation

```javascript
// ❌ DANGEROUS — trusts user input
const userId = req.params.id;
db.query(`SELECT * FROM users WHERE id = ${userId}`);
// User sends: ?id=1; DROP TABLE users; → SQL INJECTION 💀

// ✅ SAFE — parameterized query
db.query('SELECT * FROM users WHERE id = $1', [userId]);
// User sends: ?id=1; DROP TABLE users; → treated as literal string ✅
```

### 4.4 CORS (Cross-Origin Resource Sharing)

```
Your API: api.example.com
Your frontend: app.example.com
Attacker site: evil.com

WITHOUT CORS:
  evil.com can make API requests to api.example.com using a 
  logged-in user's browser cookies → steal data.

WITH CORS:
  api.example.com returns:
  Access-Control-Allow-Origin: app.example.com
  
  Browser sees: evil.com ≠ app.example.com → BLOCKS the request. ✅
```

## 5. HTTPS & TLS

### Why HTTPS is Non-Negotiable

```
HTTP (plain text):
  User sends: POST /login { "password": "mysecret123" }
  
  Anyone on the network (Wi-Fi, ISP, hacker) can READ this:
  └── "password": "mysecret123" ← VISIBLE! 💀

HTTPS (encrypted):
  User sends: POST /login { encrypted blob }
  
  Anyone on the network sees:
  └── xK8$#mQ&2z... ← UNREADABLE without the encryption key ✅
```

### mTLS (Mutual TLS) for Microservices

```
Normal TLS: Client verifies server's certificate. (One-way trust.)
mTLS: Client AND server verify each other's certificates. (Two-way trust.)

Use in microservices:
  Service A → mTLS → Service B
  
  Service B: "Is Service A authorized to call me? Certificate valid?" ✅
  Prevents: Rogue services from calling internal APIs.
  Used by: Kubernetes service mesh (Istio), banking systems.
```

## 6. Common Attack Vectors & Defenses

| Attack | How It Works | Defense |
| :--- | :--- | :--- |
| **SQL Injection** | `id=1; DROP TABLE users` in query parameter | Parameterized queries (never concat user input) |
| **XSS (Cross-Site Scripting)** | Attacker injects `<script>` in a comment/post → steals cookies | Escape HTML output. CSP headers. httpOnly cookies. |
| **CSRF (Cross-Site Request Forgery)** | evil.com makes a POST to your API using victim's logged-in session | CSRF tokens. SameSite cookie attribute. |
| **DDoS** | Millions of requests from botnets → overwhelm your servers | CDN (Cloudflare), WAF, rate limiting, auto-scaling |
| **Brute Force (Login)** | Try 10,000 passwords per second | Rate limit login attempts. CAPTCHA after 5 failures. Account lockout. |
| **Man-in-the-Middle** | Attacker intercepts HTTP traffic between client and server | HTTPS (TLS). HSTS header (force HTTPS). |

## 7. Security in System Design

### Where Security Fits in Your Architecture

```
┌──────────────────────────────────────────────────────────────────┐
│                   SECURITY LAYERS                                │
│                                                                  │
│  Layer 1: EDGE (CDN / WAF)                                      │
│  ├── DDoS protection (Cloudflare absorbs volumetric attacks)    │
│  ├── WAF rules (block SQL injection, XSS patterns)              │
│  ├── Bot detection                                               │
│                                                                  │
│  Layer 2: API GATEWAY                                            │
│  ├── Authentication (validate JWT / API key)                     │
│  ├── Rate limiting (per user / per IP)                           │
│  ├── Input validation & sanitization                             │
│                                                                  │
│  Layer 3: APPLICATION                                            │
│  ├── Authorization (role-based access control — RBAC)           │
│  ├── Business logic validation                                   │
│  ├── Parameterized queries (prevent SQL injection)               │
│                                                                  │
│  Layer 4: DATA                                                   │
│  ├── Encryption at rest (AES-256 for stored data)               │
│  ├── Encryption in transit (TLS for all network communication)  │
│  ├── PII masking in logs (never log passwords, card numbers)    │
│  ├── Principle of least privilege (DB user has minimum perms)   │
└──────────────────────────────────────────────────────────────────┘
```

## 8. Interview Questions

### ❓ Q1: "How would you design authentication for a microservices system?"

**Answer:** "Use an **API Gateway** as the single authentication point. The gateway validates the JWT on every request. If valid, it forwards the request to internal services with a trusted header (`X-User-ID: 123`). Internal services trust the gateway — no re-authentication. For service-to-service calls (no user context), use **mTLS** — each service has a certificate, and only authorized services can communicate."

### ❓ Q2: "JWT or Sessions? When would you choose each?"

**Answer:** "JWT for APIs, mobile apps, and microservices — because it's **stateless** (no shared session store needed). Sessions for traditional web apps where you need **instant revocation** (logout, account ban, password change). In practice: short-lived JWT (15 min) + refresh token (30 days in httpOnly cookie). On logout, blacklist the refresh token in Redis."

### ❓ Q3: "A user reports they were charged twice. How do you prevent this?"

**Answer:** "This is a **duplicate request** problem. Fix with **idempotency keys**:
1. Frontend generates a unique `Idempotency-Key: order_789_pay_abc` per payment.
2. Backend checks: 'Was this key already processed?' → If yes, return the cached result.
3. Backend processes payment + stores the key → result pair in Redis (TTL: 24 hours).
4. Even if the user clicks 'Pay' 10 times, only 1 payment goes through."

### ❓ Q4: "How do you store user passwords securely?"

**Answer:** "**Never store plain text.** Use **bcrypt** (work factor 12):
1. User creates password: `'mysecret123'`
2. Server generates salt + hashes: `bcrypt.hash('mysecret123', 12)` → `$2b$12$LJ3...` 
3. Store only the hash in DB. Original password is irrecoverable.
4. On login: `bcrypt.compare('mysecret123', storedHash)` → true/false.
Why bcrypt? It's deliberately SLOW (100ms per hash). Brute-forcing 10M passwords takes years, not seconds (unlike MD5/SHA which are sub-microsecond)."
