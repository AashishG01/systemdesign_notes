## 1. DNS — Domain Name System

### 1.1 What is DNS?

DNS is the **phonebook of the internet**. It translates human-readable domain names (`google.com`) to IP addresses (`142.250.190.78`) that computers understand.

### 1.2 DNS Resolution — Step by Step

```
User types: www.example.com

Step 1: Browser Cache
  → "Have I visited this domain in the last 5 minutes?"
  → If YES → use cached IP. Done. ✅

Step 2: OS Cache
  → Check /etc/hosts file and OS DNS cache.

Step 3: Recursive Resolver (ISP's DNS Server)
  → ISP's DNS server checks its cache.
  → If not cached, begins recursive lookup:

Step 4: Root Name Server (13 root servers worldwide)
  → "I don't know example.com, but .com is handled by these TLD servers"

Step 5: TLD Name Server (.com server)
  → "I don't know example.com, but its authoritative NS is ns1.example.com"

Step 6: Authoritative Name Server (ns1.example.com)
  → "example.com → 93.184.216.34" ✅

Step 7: Response flows back to browser. Cached with TTL (e.g., 300 seconds).
```

### 1.3 DNS Record Types

| Record | Purpose | Example |
| :--- | :--- | :--- |
| **A** | Domain → IPv4 address | `example.com → 93.184.216.34` |
| **AAAA** | Domain → IPv6 address | `example.com → 2606:2800:220:1:248:...` |
| **CNAME** | Domain → another domain (alias) | `www.example.com → example.com` |
| **MX** | Mail server for the domain | `example.com → mail.example.com` |
| **NS** | Authoritative name server | `example.com → ns1.cloudflare.com` |
| **TXT** | Text data (SPF, DKIM, verification) | `"v=spf1 include:_spf.google.com"` |

### 1.4 DNS in System Design

| Concept | Use in System Design |
| :--- | :--- |
| **DNS Load Balancing** | Return multiple A records. Client picks one (round-robin DNS). |
| **GSLB** | GeoDNS returns different IPs based on user's location (US user → US server). |
| **DNS Failover** | Health-check servers. If primary is down, DNS returns backup server's IP. |
| **Low TTL** | Set TTL to 60 seconds for fast failover (DNS updates propagate in 1 minute). |
| **CDN DNS** | CDN uses Anycast DNS to route to the nearest edge server. |

## 2. TCP vs UDP

### 2.1 TCP (Transmission Control Protocol)

```
TCP = "Registered Mail"
  → Guaranteed delivery (acknowledgements)
  → Ordered delivery (packets arrive in sequence)
  → Error checking (corrupted packets are retransmitted)
  → Connection-oriented (3-way handshake before data transfer)

3-Way Handshake:
  Client → SYN →           Server     "Can I connect?"
  Client ← SYN-ACK ←       Server     "Yes, and can I connect back?"
  Client → ACK →           Server     "Yes. Connection established."
  Client ←→ DATA ←→        Server     (bidirectional data transfer)
```

### 2.2 UDP (User Datagram Protocol)

```
UDP = "Throwing a paper airplane"
  → No guaranteed delivery (packets may be lost)
  → No ordering (packets may arrive out of order)
  → No connection setup (no handshake — just send)
  → FAST (minimal overhead)
```

### 2.3 TCP vs UDP Comparison

| Feature | TCP | UDP |
| :--- | :--- | :--- |
| **Reliability** | ✅ Guaranteed delivery | ❌ "Fire and forget" |
| **Ordering** | ✅ In-order delivery | ❌ No ordering |
| **Speed** | Slower (overhead from ACKs, retransmission) | Faster (no overhead) |
| **Connection** | Connection-oriented (3-way handshake) | Connectionless |
| **Use Cases** | HTTP, HTTPS, SSH, FTP, email, database connections | Video streaming, VoIP, online gaming, DNS lookups, live broadcast |

> **💡 Rule of Thumb:** If losing a packet is unacceptable (web pages, files, API calls) → **TCP**. If speed matters more than perfection (live video, gaming where 1 dropped frame is invisible) → **UDP**.

## 3. HTTP/1.1 vs HTTP/2 vs HTTP/3

| Feature | HTTP/1.1 | HTTP/2 | HTTP/3 |
| :--- | :--- | :--- | :--- |
| **Year** | 1997 | 2015 | 2022 |
| **Transport** | TCP | TCP | **QUIC (UDP-based)** |
| **Connections** | 1 request per connection (pipelining broken in practice) | Multiplexed — many requests on 1 connection | Multiplexed + no head-of-line blocking |
| **Header compression** | ❌ None | ✅ HPACK | ✅ QPACK |
| **Server Push** | ❌ | ✅ Server can push resources before client asks | ✅ |
| **Head-of-Line Blocking** | ❌ Severe (1 slow response blocks all) | ⚠️ At TCP level | ✅ Solved (QUIC streams are independent) |

```
HTTP/1.1 (6 TCP connections per domain):
  Connection 1: GET /page.html      → wait → response → GET /style.css → wait → response
  Connection 2: GET /script.js      → wait → response → GET /image1.jpg ...
  ❌ Each connection handles 1 request at a time. 6 max parallel.

HTTP/2 (1 TCP connection, multiplexed):
  Connection 1: Stream 1: GET /page.html    ──▶ response
                Stream 2: GET /style.css    ──▶ response    (all in parallel!)
                Stream 3: GET /script.js    ──▶ response
                Stream 4: GET /image1.jpg   ──▶ response
  ✅ All requests on 1 connection. No blocking. Much faster.

HTTP/3 (QUIC / UDP):
  Same multiplexing as HTTP/2, but if Stream 3 packet is lost,
  only Stream 3 is retransmitted. Streams 1, 2, 4 continue unaffected.
  ✅ Solves TCP's head-of-line blocking problem entirely.
```

## 4. WebSockets, SSE, Long Polling

### Comparison

| Technology | Direction | Connection | Best For |
| :--- | :--- | :--- | :--- |
| **Polling** | Client → Server (repeated) | New connection each time | Simple, low-frequency updates |
| **Long Polling** | Client → Server (held open) | Held until data available | Chat (before WebSockets were common) |
| **Server-Sent Events (SSE)** | Server → Client (one-way) | Persistent HTTP connection | Live feeds, dashboards, notifications |
| **WebSockets** | Bidirectional | Persistent TCP connection | Real-time chat, gaming, collaboration |

```
POLLING (wasteful):
  Client: "Any updates?"     Server: "No."
  Client: "Any updates?"     Server: "No."
  Client: "Any updates?"     Server: "Yes! Here's the data."
  ❌ 2 wasted requests.

LONG POLLING (better):
  Client: "Any updates?" → Server holds connection open...
  ... (30 seconds pass) ...
  Server: "Yes! Here's the data." → Client immediately reconnects.
  ✅ No wasted requests. But connection recreation overhead.

SSE (server push):
  Client: "Subscribe to updates."
  Server → Push: "Update 1"
  Server → Push: "Update 2"
  Server → Push: "Update 3"
  ✅ One-way, server pushes when ready. Built-in reconnection.

WEBSOCKET (full duplex):
  Client ←→ Server: "I can send AND receive at any time."
  ✅ Bidirectional. Real-time. Persistent connection.
  ⚠️ Stateful. Harder to scale (need Redis Pub/Sub for multi-server).
```

## 5. TLS/SSL & HTTPS

### 5.1 What is HTTPS?

**HTTPS = HTTP + TLS** (Transport Layer Security). TLS encrypts data between client and server so no one in between can read or tamper with it.

### 5.2 TLS Handshake (Simplified)

```
1. Client Hello: "I support TLS 1.3. Here are my supported cipher suites."
2. Server Hello: "Let's use TLS 1.3 with AES-256. Here's my certificate."
3. Client verifies: "Certificate signed by trusted CA? Domain matches? Not expired?"
4. Key Exchange: Client and server agree on a shared secret key (Diffie-Hellman).
5. Encrypted communication begins. All data is encrypted with symmetric key (AES).
```

### 5.3 Why It Matters in System Design

| Concept | Impact |
| :--- | :--- |
| **SSL Termination** | Decrypt HTTPS at the reverse proxy / load balancer. Backend servers use plain HTTP (faster, simpler). |
| **mTLS (mutual TLS)** | Both client AND server present certificates. Used for service-to-service authentication in microservices. |
| **Certificate Management** | Use Let's Encrypt for free auto-renewing certs. Or AWS ACM for managed certificates. |

## 6. IP Addressing, Ports, and Sockets

### Quick Reference

| Concept | What It Is | Example |
| :--- | :--- | :--- |
| **IP Address** | Unique identifier for a machine on a network | `192.168.1.5` (private), `13.32.0.1` (public) |
| **Port** | Identifies a specific process/service on a machine | `:80` (HTTP), `:443` (HTTPS), `:5432` (PostgreSQL) |
| **Socket** | IP:Port combination — a unique network endpoint | `192.168.1.5:3000` |
| **Private IP** | Only visible within a local network (NAT) | `10.0.0.x`, `192.168.x.x`, `172.16.x.x` |
| **Public IP** | Visible on the internet | `34.102.136.1` (your server's IP) |

### Well-Known Ports

| Port | Service |
| :--- | :--- |
| 80 | HTTP |
| 443 | HTTPS |
| 22 | SSH |
| 3306 | MySQL |
| 5432 | PostgreSQL |
| 6379 | Redis |
| 27017 | MongoDB |
| 9092 | Kafka |

## 7. Interview Questions

### ❓ Q1: "Walk me through what happens when you type google.com in a browser."

**Answer:**
```
1. DNS Resolution: Browser → OS cache → ISP DNS → Root → .com TLD → Google's NS → IP: 142.250.x.x
2. TCP Connection: 3-way handshake (SYN → SYN-ACK → ACK) to port 443
3. TLS Handshake: Certificate verification + key exchange → encrypted channel
4. HTTP Request: GET / HTTP/2 with headers (Host, User-Agent, Accept)
5. Server Processing: Load balancer → app server → generates HTML
6. HTTP Response: 200 OK with HTML body, Cache-Control headers
7. Browser Rendering: Parse HTML → Fetch CSS/JS (parallel via HTTP/2) → Build DOM → Paint
```

### ❓ Q2: "Why is UDP used for video streaming instead of TCP?"

**Answer:** "In live video, a lost packet (1 frame out of 30 per second) is invisible to the viewer. But TCP would stall the entire stream to retransmit that packet (head-of-line blocking). UDP drops it and moves on — the viewer sees smooth video with an imperceptible glitch. Latency matters more than perfection in live streaming."

### ❓ Q3: "When would you use WebSockets vs SSE vs Polling?"

**Answer:** "Polling for updates every 30+ seconds (weather app). SSE for server-initiated one-way pushes (live score feed, stock ticker). WebSockets for bidirectional real-time (chat, multiplayer gaming, collaborative editing). SSE is simplest — use it unless you need bidirectional communication."
