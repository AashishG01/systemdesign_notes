# 1. Why People Go Blank in Interviews

You go blank because you lack **3 mental muscles**:

| Missing Muscle | What it Means | How to Build It |
| :--- | :--- | :--- |
| **Bottleneck Thinking** | You can't instantly answer "Where will this system break?" | For every feature you build, ask: "What if 10K users hit this at once?" |
| **Scaling Instinct** | You don't reflexively think about caching, queues, and sharding | For every DB query you write, ask: "What if this table has 100M rows?" |
| **Trade-off Awareness** | You can't articulate WHY you chose X over Y | Always say: "I chose X because..., the alternative was Y, which has the downside of..." |

### The Mindset Shift

```
❌ WRONG MINDSET:  "What is the perfect design for a URL shortener?"
                    (You try to recall a memorized diagram. Go blank.)

✅ RIGHT MINDSET:  "What are the REQUIREMENTS? What are the BOTTLENECKS?
                    What BREAKS at scale? What TRADE-OFFS am I making?"
                    (You think step by step. Never blank.)
```

> **You are not designing a system. You are HUNTING bottlenecks and MAKING trade-offs.** That's all system design is.

# 2. The 7-Step Universal Design Framework

Use this framework for **ANY** design problem. It works for every question.

```
┌─────────────────────────────────────────────────────────────────────┐
│                  THE 7-STEP FRAMEWORK                               │
│                                                                     │
│  Step 1: CLARIFY REQUIREMENTS (2-3 min)                            │
│  ├── Functional: What must the system DO?                          │
│  ├── Non-Functional: Scale, latency, availability, consistency     │
│  └── Constraints: Read-heavy? Write-heavy? Real-time?              │
│                                                                     │
│  Step 2: ESTIMATE SCALE (1-2 min)                                  │
│  ├── Users: DAU / MAU                                              │
│  ├── Traffic: Requests per second (RPS)                            │
│  ├── Storage: Size × Quantity × Retention                          │
│  └── Bandwidth: Payload × RPS                                      │
│                                                                     │
│  Step 3: DEFINE API (2 min)                                        │
│  ├── Key endpoints (REST)                                          │
│  └── Request/Response format                                        │
│                                                                     │
│  Step 4: DATA MODEL (2-3 min)                                      │
│  ├── Entities, relationships                                       │
│  ├── SQL vs NoSQL decision                                         │
│  └── Key indexes                                                    │
│                                                                     │
│  Step 5: HIGH-LEVEL DESIGN (5-7 min)                               │
│  ├── Draw the architecture diagram                                 │
│  ├── Box = Service. Arrow = Data flow.                             │
│  └── Client → LB → API → Cache → DB                               │
│                                                                     │
│  Step 6: DEEP DIVE (5-10 min)                                      │
│  ├── Pick 1-2 hard components and go deep                          │
│  ├── "Where does it break?" analysis                               │
│  └── Show trade-offs and alternatives                              │
│                                                                     │
│  Step 7: BOTTLENECKS & SCALING (3-5 min)                           │
│  ├── What if traffic spikes 100x?                                  │
│  ├── What if the DB crashes?                                       │
│  ├── Single points of failure?                                     │
│  └── Monitoring & alerting                                         │
└─────────────────────────────────────────────────────────────────────┘
```

# 3. The Bottleneck Hunting Checklist

**After drawing ANY design, ask these 6 questions:**

| # | Question | If the Answer is YES | Fix |
| :--- | :--- | :--- | :--- |
| 1 | Is a single server handling all traffic? | SPOF + capacity limit | Add **Load Balancer** + horizontal scaling |
| 2 | Is every read hitting the database? | DB CPU saturated, slow queries | Add **Redis Cache** (cache-aside pattern) |
| 3 | Is a slow task blocking the API response? | High latency, bad UX | Move to **Message Queue** (async processing) |
| 4 | Is one DB table growing infinitely? | Query degradation, storage limit | Add **Sharding** or archive old data |
| 5 | Is there a single database for reads and writes? | Write contention slows reads | Add **Read Replicas** (CQRS if extreme) |
| 6 | Is there no fallback if a service dies? | Total outage when one service fails | Add **Circuit Breaker** + fallback responses |

# 4. The Scaling Instinct Cheat Sheet

Whenever you hear a buzzword in a design problem, your brain should immediately jump to the solution:

| When You Hear... | Immediately Think... |
| :--- | :--- |
| "Read-heavy" | → Cache (Redis), Read Replicas, CDN |
| "Write-heavy" | → Message Queue (Kafka), batch writes, sharding |
| "Low latency" | → Cache, CDN, in-memory data, pre-computation |
| "High availability" | → Replication, multi-region, failover, no SPOF |
| "Search / full-text" | → Elasticsearch (not SQL LIKE) |
| "Real-time" | → WebSockets, Pub/Sub, Server-Sent Events |
| "Millions of users" | → Horizontal scaling, sharding, CDN, async processing |
| "Long-running task" | → Message Queue + Workers (don't block the API) |
| "Unique constraint at scale" | → Bloom Filter (probabilistic), distributed ID generation |
| "Ordered events" | → Kafka with partition key |
| "Counting / analytics" | → Pre-aggregation, materialized views, HyperLogLog |
| "Global users" | → CDN, GSLB, multi-region deployment |
| "File/image/video" | → Object storage (S3), CDN, separate from app server |

# 5. Problem 1: URL Shortener (Bitly)

## Requirements
- Given a long URL, generate a short URL (e.g., `bit.ly/abc123`)
- Redirecting the short URL takes you to the original long URL
- Links expire after a configurable time (optional)
- Analytics: track click count per link

## Scale Estimation
| Metric | Value |
| :--- | :--- |
| New URLs/day | 100M |
| Redirects/day | 1B (10:1 read:write ratio) |
| Redirect RPS | ~12,000 |
| URL storage | ~500 bytes × 100M = 50GB/day |

## API Design
```
POST /api/shorten   { long_url, expiry }  → { short_url }
GET  /:shortCode    → 301 Redirect to long_url
GET  /api/stats/:shortCode  → { clicks, created_at }
```

## Data Model
```sql
Table: urls
  id          BIGINT PRIMARY KEY
  short_code  VARCHAR(7) UNIQUE INDEX
  long_url    TEXT
  created_at  TIMESTAMP
  expires_at  TIMESTAMP
  click_count INT DEFAULT 0
```

## Architecture
```
Client → LB → API Server → Redis Cache → PostgreSQL
                               │
                  Cache short_code → long_url
                  (99% of reads from cache)
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Short code collision** | Two URLs get the same 7-char code | Use base62 encoding of auto-increment ID. No collision possible. Or: generate random, check DB, retry. |
| **Read-heavy DB** | 12K redirects/sec hitting DB | **Redis cache**: `short_code → long_url`. Cache-aside. 99% cache hit rate. DB almost never touched for reads. |
| **Single DB** | 50GB/day = 18TB/year | Shard by short_code hash. Or use NoSQL (DynamoDB) for horizontal scaling. |
| **Analytics writes** | 12K `UPDATE click_count` per second saturates DB | **Kafka** + batch aggregation. Publish click events to Kafka. A consumer batch-updates counts every 10 seconds. |
| **Hot links** | A viral link gets 1M clicks/sec. Single Redis key bottleneck. | Cache in CDN (301 with `Cache-Control: max-age=300`). 99.9% of hits never reach your server. |

# 6. Problem 2: Real-Time Chat System (WhatsApp)

## Requirements
- 1-to-1 messaging + group chats (up to 256 members)
- Real-time delivery (< 100ms latency)
- Offline message delivery (queue + deliver when user comes online)
- Read receipts (sent, delivered, read)
- Message history storage

## Architecture
```
┌────────┐  WebSocket  ┌──────────────┐         ┌───────────────┐
│ Client │◄───────────▶│  Chat Server │────────▶│ Message Queue │
└────────┘              │  (stateful   │         │   (Kafka)     │
                        │   WS conn)   │         └───────┬───────┘
                        └──────────────┘                 │
                               │                   ┌─────▼──────┐
                        ┌──────▼──────┐            │ Notification│
                        │   Redis     │            │  Service    │
                        │ (user→server│            │ (push for   │
                        │  mapping)   │            │  offline)   │
                        └─────────────┘            └────────────┘
                               │
                        ┌──────▼──────┐
                        │ Cassandra   │
                        │ (messages)  │
                        └─────────────┘
```

## Message Flow
```
Alice sends "Hello" to Bob:

1. Alice → WebSocket → Chat Server A
2. Chat Server A looks up Bob's connection: Redis says "Bob is on Chat Server C"
3. Chat Server A → Kafka topic "messages" → Chat Server C
4. Chat Server C → WebSocket → Bob
5. Message stored in Cassandra (async, via Kafka consumer)

If Bob is OFFLINE:
3. Chat Server A → Kafka → Notification Service → Push notification to Bob's phone
4. Message stored in Cassandra. When Bob comes online, Chat Server fetches undelivered messages.
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Millions of WebSocket connections** | Each chat server holds connections in memory (~50K sockets/server) | Horizontally scale to 200 chat servers. Use Redis to map `user_id → server_id` |
| **Group messages** | 256-member group = 1 message fanned out to 255 recipients across N servers | Kafka fan-out: publish once, each recipient's chat server consumes independently |
| **Message ordering** | Messages arriving out of order ruins conversation | Kafka partition key = `conversation_id`. All messages for a conversation go to the same partition → ordered |
| **Hot conversations** | A celebrity group with 256 members where every member sends 10 msg/sec | Rate limit messages per user per conversation. Batch read receipts (buffer for 2 sec, send one batch ACK) |
| **DB writes** | Millions of messages/sec persisted to DB | **Cassandra** (write-optimized, horizontally scalable). Partition key = `conversation_id`, clustering key = `timestamp`. |

# 7. Problem 3: File Upload & Storage Service (Google Drive)

## Requirements
- Upload/download files (up to 10GB)
- File metadata (name, size, type, owner)
- Sharing (public link, private, specific users)
- Version history

## Architecture
```
Client → API Gateway → Upload Service → S3 (file storage)
                              │
                        PostgreSQL (metadata)
                              │
                        Redis (pre-signed URL cache)
```

## Upload Flow
```
1. Client requests upload URL:   POST /api/files/upload-url { filename, size }
2. Server generates pre-signed S3 URL (valid for 15 min)
3. Client uploads directly to S3 using pre-signed URL (bypasses your server entirely!)
4. S3 triggers Lambda/webhook → Update metadata in PostgreSQL
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Large files through your server** | A 10GB upload through your API server = memory exhaustion, blocked threads | **Pre-signed S3 URLs**: client uploads directly to S3. Your server never touches the file bytes. |
| **Simultaneous uploads** | 10K users uploading simultaneously saturates network bandwidth | Pre-signed URLs solve this — uploads go to S3's massive infrastructure, not your server. |
| **DB for file metadata** | Millions of files × millions of users | Shard metadata DB by `user_id`. Each user's files are on the same shard. |
| **Hot downloads** | A publicly shared file goes viral — 1M downloads/hour | **CDN** (CloudFront). Cache the file at edge locations. S3 serves it once → CDN serves the rest. |
| **Version explosion** | User saves a 1GB file 100 times = 100GB stored | **Delta storage**: only store the diff between versions, not the full file each time. |

# 8. Problem 4: Notification Service (Push/Email/SMS)

## Requirements
- Send push notifications, emails, and SMS
- Template-based messages
- Priority levels (urgent vs marketing)
- Rate limiting (don't spam users)
- Delivery tracking (sent, delivered, read, failed)

## Architecture
```
Any Service → Kafka ("notification-events") → Notification Service
                                                      │
                              ┌────────────────────────┼────────────────┐
                              ▼                        ▼                ▼
                        ┌──────────┐           ┌──────────┐     ┌──────────┐
                        │  Push    │           │  Email   │     │   SMS    │
                        │  Worker  │           │  Worker  │     │  Worker  │
                        │ (FCM/    │           │ (SES/    │     │ (Twilio/ │
                        │  APNS)   │           │ SendGrid)│     │ SNS)     │
                        └──────────┘           └──────────┘     └──────────┘
                              │                        │                │
                              └────────────────────────┼────────────────┘
                                                       ▼
                                              ┌──────────────┐
                                              │ Delivery Log │
                                              │ (Cassandra)  │
                                              └──────────────┘
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Spike: 10M notifications at once** | "Flash sale starts now!" to all users | **Kafka queue** absorbs the burst. Workers process at a controlled rate (50K/sec). Users receive notifications over 3 minutes — acceptable. |
| **Third-party API limits** | FCM allows 500 msg/sec. You need to send 10M. | **Rate-limited workers** with Token Bucket. Queue absorbs the burst, workers drain at the provider's allowed rate. |
| **User spammed** | 20 marketing emails in 1 hour → user unsubscribes | Per-user notification rate limit in Redis: max 3 marketing notifications/day. Priority system: urgent > transactional > marketing. |
| **Delivery failures** | SMS to invalid number, email bounces | **DLQ + Retry**: Retry 3 times with exponential backoff. After 3 failures → DLQ. Mark user's channel as invalid. |
| **Template rendering** | 10M personalized emails (each with user's name, order details) | Pre-render templates in workers, not in the API server. Parallelize across 100 workers. |

# 9. Problem 5: Rate Limiter

## Requirements
- Limit API requests per user/IP/API-key
- Support multiple time windows (per-second, per-minute, per-day)
- Distributed (works across 50 API servers)
- Low latency (< 1ms overhead per request)

## Architecture
```
Client → API Gateway → Rate Limiter Middleware → Backend Service
                              │
                       ┌──────▼──────┐
                       │    Redis    │ ← Centralized counters
                       │   Cluster   │   (INCR + EXPIRE)
                       └─────────────┘
```

## Algorithm Choice: Token Bucket
```
Why: Allows bursts (real user behavior is bursty) + enforces average rate.

Redis Lua Script (atomic):
  1. Get current tokens and last refill time
  2. Calculate tokens to add (elapsed × refill_rate)
  3. If tokens >= 1: consume 1, return ALLOWED
  4. Else: return REJECTED
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Redis latency** | Every API request → Redis round-trip (+1ms) | Redis Cluster with replicas. Use **local in-memory cache** for first N requests — sync with Redis every 1 second. |
| **Redis failure** | Redis down = no rate limiting or block everything? | **Fail open** (allow all) for most APIs. **Fail closed** (block all) for security-critical APIs. Local fallback counter. |
| **Distributed IP rotation** | Attacker uses 10K IPs to bypass per-IP limits | Layer defenses: per-IP + per-user + behavioral analysis + CAPTCHA + WAF. |
| **Hot key** | One API key making 1M req/sec → single Redis key bottleneck | Shard the counter: `key:1`, `key:2`, ..., `key:10`. Sum all shards. |

# 10. Problem 6: Search System (Elasticsearch)

## Requirements
- Full-text search across millions of documents (products, articles, users)
- Autocomplete / typeahead
- Filters (price range, category, date)
- Ranking (relevance, popularity)
- < 200ms response time

## Architecture
```
Client → API Gateway → Search Service → Elasticsearch Cluster
                                              │
                                    ┌─────────┼─────────┐
                                    ▼         ▼         ▼
                                 Shard 1   Shard 2   Shard 3
                                 (A-H)     (I-P)     (Q-Z)

Write Path:
  Product Service → Kafka ("product-updates") → ES Indexer → Elasticsearch
  (DB is source of truth. ES is a READ REPLICA for search.)
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Search on SQL DB** | `SELECT * FROM products WHERE name LIKE '%shoe%'` → full table scan, 30 seconds on 100M rows | **Elasticsearch** — inverted index, full-text search in < 50ms on 100M documents |
| **Stale search results** | Product updated in PostgreSQL, but ES still shows old data | **Kafka pipeline**: PostgreSQL CDC → Kafka → ES Indexer. Latency: ~1 second. Acceptable for search. |
| **Autocomplete latency** | User types each character, fetching suggestions each time | **Edge n-gram index** in ES + **Redis cache** for top 1000 most popular prefixes. Response < 30ms. |
| **Hot queries** | "iPhone" is searched 100K times/min | **Redis cache** for popular queries with 30-second TTL. 95% of searches are repeated queries. |
| **Index size** | 500M products × 1KB = 500GB index | Shard across N nodes. ES handles this natively. Add nodes as data grows. |

# 11. Problem 7: Blog Platform (Medium)

## Requirements
- Write/edit/publish articles (rich text + images)
- Follow authors, personalized feed
- Likes, comments, bookmarks
- Tags and categories
- SEO-friendly URLs

## Architecture
```
Client → CDN (cached article HTML) → LB → API Servers → Redis → PostgreSQL
                                                                      │
                                                                  S3 (images)
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Viral article** | 1 article gets 10M views in 1 hour | **CDN caching** (cache rendered article page for 60s). After 1st request, CDN serves 9,999,999 requests. Server sees ~1 request/minute. |
| **Feed generation** | "Show me articles from 500 authors I follow, ranked by relevance" = expensive DB query | **Pre-computed feed** (Fan-out on write): When Author X publishes, push article_id to every follower's feed in Redis. Feed is a Redis Sorted Set. |
| **Like counter** | 100K users "like" an article simultaneously → 100K UPDATE queries | **Redis INCR** for real-time count + batch-write to PostgreSQL every 30 seconds. |
| **Image hosting** | Articles with 10MB images served from your API server | **S3 + CDN**. Upload images to S3, serve via CloudFront. App server never touches image bytes. |
| **Comment storms** | Controversial article gets 50K comments → rendering all of them crashes the page | **Paginated comments** (20 per page) + **cursor-based pagination** (not offset-based — offset is slow for deep pages). |

# 12. Problem 8: Expense Tracker (Splitwise)

## Requirements
- Create groups, add expenses
- Split bills equally, by percentage, or exact amounts
- Calculate "who owes whom"
- Settle debts (mark as paid)
- Transaction history

## Data Model
```sql
Table: groups       (id, name, created_by)
Table: members      (group_id, user_id)
Table: expenses     (id, group_id, paid_by, amount, description, split_type, created_at)
Table: expense_splits (expense_id, user_id, amount_owed)
Table: settlements  (id, group_id, from_user, to_user, amount, settled_at)
```

## The Core Algorithm: Simplify Debts
```
Raw debts:
  Alice owes Bob $20
  Bob owes Charlie $15
  Alice owes Charlie $10

Simplified (minimum transactions):
  Step 1: Calculate net balance
    Alice:   -$30 (owes $30 total)
    Bob:     +$5  (owed $20, owes $15 → net +$5)
    Charlie: +$25 (owed $25 total)
    
  Step 2: Match debtors with creditors
    Alice pays Charlie $25
    Alice pays Bob $5
    
  Result: 2 transactions instead of 3
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Concurrent expense creation** | Two users add an expense at the same time → inconsistent balances | **Database transactions** (SERIALIZABLE isolation) on the balance calculation. Or: optimistic locking with version column. |
| **Complex group queries** | "Calculate who owes whom in a 50-person group with 500 expenses" | **Pre-compute balances** on every expense change. Store a `balances` table with `(group_id, user_id, net_balance)`. Recalculate on writes, read instantly. |
| **Currency conversion** | Multi-currency group (some in USD, some in EUR) | Store all amounts in a **base currency** (smallest unit, e.g., cents in USD). Convert at input time using exchange rate API. |

# 13. Problem 9: Paste Service (Pastebin)

## Requirements
- Create text pastes (up to 10MB)
- Generate unique short links
- Set expiry (1 hour, 1 day, 1 week, never)
- View paste content via short link
- Optional: syntax highlighting, private pastes

## Architecture
```
Client → LB → API → S3 (paste content) + PostgreSQL (metadata) + Redis (hot pastes)
```

## Key Design Decisions
| Decision | Choice | Why |
| :--- | :--- | :--- |
| Store paste content in DB? | ❌ No. Use **S3**. | Pastes can be 10MB. Storing 10M × 10MB = 100TB in PostgreSQL is insane. S3 is built for this. |
| Generate short link how? | Base62 encode a unique ID | Auto-increment ID → Base62(id) = `abc123`. Short, unique, no collisions. |
| Handle expiry? | TTL column + background cleanup job | Cron job runs hourly: `DELETE FROM pastes WHERE expires_at < NOW()`. Also delete S3 objects. |

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Viral paste** | A code snippet goes viral — 1M views in 1 hour | **CDN** (immutable content is perfect for CDN caching). Set `Cache-Control: public, max-age=86400`. |
| **Large paste reads** | 10MB paste served 10K times = 100GB bandwidth | **CDN + S3**. Your server never touches the paste content. |
| **Expired paste cleanup** | 1 billion expired pastes clogging the DB | **S3 lifecycle policies** auto-delete objects after expiry. DB cleanup via batched DELETE queries. |

# 14. Problem 10: Social Media Feed (Instagram/Twitter)

## Requirements
- Post content (text, images, videos)
- Follow/unfollow users
- Personalized news feed
- Like, comment, share
- Real-time feed updates

## The Core Problem: Feed Generation ⭐

This is the hardest problem in social media design.

### Approach 1: Fan-Out on Read (Pull Model)
```
When User opens Feed:
  1. Fetch list of people they follow (500 users)
  2. Fetch recent posts from each (500 queries!)
  3. Merge, rank, return top 50
  
  ❌ Problem: 500 DB queries every time someone opens the feed. Extremely slow.
```

### Approach 2: Fan-Out on Write (Push Model) ⭐
```
When User X posts:
  1. Fetch all followers of X (10,000 users)
  2. Push X's post_id into each follower's feed (Redis Sorted Set)
  3. When a follower opens their feed, just read their pre-built feed from Redis
  
  ✅ Feed read = 1 Redis query. Instant.
  ❌ Problem: Celebrity with 50M followers → 50M Redis writes per post. Expensive.
```

### Approach 3: Hybrid (Industry Standard) ⭐⭐
```
Normal users (< 10K followers): Fan-Out on Write (push to followers' feeds)
Celebrities (> 10K followers):  Fan-Out on Read  (merge at read time)

Why: Celebrity posts are recent and few. Fetching 20 celeb posts at read time is fast.
     Normal user posts need to be pre-pushed because there are millions of them.
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Celebrity posts** | 50M fan-out writes per post | Hybrid model: celebrities use Fan-Out on Read |
| **Feed storage** | Pre-built feeds for 500M users in Redis = massive memory | Store only last 500 post_ids per user. Older? Query DB. Use Redis memory-optimized data structures. |
| **Image/video serving** | 1TB of images uploaded daily | **S3 + CDN**. Never serve media from your app servers. |
| **Like counter** | Viral post: 10M likes → 10M DB writes | **Redis INCR** (real-time approximation) + batch write to DB every 30 seconds |
| **Real-time updates** | Follower should see new post without refreshing | **WebSocket** or **Server-Sent Events**. Push new post_ids to online followers' open connections. |

# 15. Problem 11: E-Commerce System (Amazon)

## Requirements
- Product catalog (search, browse, filter)
- Shopping cart
- Order placement + payment
- Inventory management
- Order tracking

## The Core Problem: Inventory Race Condition

```
PROBLEM:
  Product: "Limited Edition Shoe" — Only 1 left in stock.
  
  User A: "Add to cart" → Reads stock = 1 ✅
  User B: "Add to cart" → Reads stock = 1 ✅ (simultaneously!)
  User A: "Buy" → stock = stock - 1 → stock = 0 ✅
  User B: "Buy" → stock = stock - 1 → stock = -1 ❌ OVERSOLD!

SOLUTION: Pessimistic Locking
  SELECT stock FROM products WHERE id = 123 FOR UPDATE;  -- Acquires row-level lock
  -- Only one transaction can proceed. The other waits.
  UPDATE products SET stock = stock - 1 WHERE id = 123 AND stock > 0;
  -- The AND stock > 0 is a safety net.
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Flash sale** | 100K users buy 1 product simultaneously | **Queue orders**: put buy requests into Kafka. Process sequentially. First-come-first-served with inventory checks. |
| **Cart abandonment** | User adds item, never buys. Stock reserved forever. | **TTL on cart reservations** (15 minutes). After TTL, release reserved stock back to inventory. |
| **Search on product catalog** | `LIKE '%shoe%'` on 100M products | **Elasticsearch** for full-text search + filters. SQL DB is source of truth. |
| **Payment failures** | Card declined after order placed | **Saga Pattern**: Order → Payment (failed) → Compensating event: cancel order, release inventory |
| **Order tracking** | Millions of status updates per hour from delivery partners | **Kafka** event stream: `OrderShipped`, `OrderOutForDelivery`, `OrderDelivered`. Consumers update status asynchronously. |

# 16. Problem 12: Parking Lot System

## Requirements
- Multi-floor parking lot with different spot sizes (compact, regular, large)
- Entry/exit gates with ticket generation
- Track which spots are available on each floor
- Calculate parking fee at exit
- Real-time availability display

## Data Model
```sql
Table: parking_spots   (id, floor, spot_number, size, is_occupied)
Table: tickets         (id, vehicle_plate, spot_id, entry_time, exit_time, fee)
```

## Core Algorithm: Spot Assignment
```
Vehicle arrives at gate:
  1. Determine vehicle size (compact/regular/large)
  2. Find nearest available spot of matching size (or larger)
     Query: WHERE size >= vehicle_size AND is_occupied = false
            ORDER BY floor ASC, spot_number ASC LIMIT 1
  3. Mark spot as occupied (atomic update)
  4. Generate ticket with entry_time

Vehicle exits:
  1. Scan ticket
  2. Calculate fee = (exit_time - entry_time) × rate_per_hour
  3. Mark spot as available
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Two cars assigned same spot** | Concurrent entry → both read spot as available | **SELECT ... FOR UPDATE** (pessimistic lock) or **Redis atomic decrement** of available count per floor. |
| **Gate congestion** | 100 cars arriving simultaneously at peak hour | Pre-compute available spots in **Redis** (decrement on entry, increment on exit). No DB query at the gate — Redis responds in < 1ms. |
| **Multi-lot scaling** | Managing 500 parking lots across a city | Each lot has its own Redis instance for availability. Central PostgreSQL for analytics and billing. |

## Requirements

### Q3: "Your API works fine during the day but crashes every night at exactly 2 AM. What's happening?"

**Structured Answer:**

```
My first instinct: Something SCHEDULED is running at 2 AM.

Check 1: CRON JOBS
  → Is there a nightly batch job? (report generation, data export, cleanup)
  → If it runs a massive query: SELECT * FROM orders WHERE date > 'last_year'
     on a 100M-row table → locks the table → API queries timeout → crash.
  → Fix: Run batch jobs on a READ REPLICA, not the primary DB.

Check 2: BACKUP JOBS
  → Is a database backup running at 2 AM? 
  → pg_dump on a large DB locks tables and consumes all I/O.
  → Fix: Use streaming replication. Backup from the replica, not the primary.

Check 3: LOG ROTATION / DISK FULL
  → Daily logs fill up the disk. At 2 AM disk hits 100% → app can't write.
  → Fix: Log rotation with retention policy. Or ship logs to ELK stack.

Check 4: MEMORY LEAK
  → App gradually leaks memory. By 2 AM (after 24 hours), OOM killer strikes.
  → Fix: Profile memory usage. Set process restart policy (PM2 --max-memory-restart).
```

### Q4: "Users are reporting 'Your session has expired' randomly — but their session shouldn't have expired. What's happening?"

**Structured Answer:**

```
MOST LIKELY CAUSE: Stateful servers behind a Load Balancer.

  Request 1 → Server A: User logs in. Session stored in Server A's memory.
  Request 2 → Server B: LB routes here. Server B has no session → "Expired!"

  Fix: Move sessions from in-memory to REDIS (shared across all servers).
  
SECOND POSSIBILITY: TTL too aggressive.
  → Redis session TTL is 5 minutes. User is idle for 6 minutes → expired.
  → Fix: Extend TTL to 30 min. Refresh TTL on every request (sliding expiration).

THIRD POSSIBILITY: Multiple deployments.
  → You deployed new code. Server A restarted → in-memory sessions lost.
  → Fix: Again, external session storage (Redis). Deployment never kills sessions.
```

### Q6: "Your database handles 10,000 reads/sec fine. Product just tripled the user base. How do you handle 30,000 reads/sec without sharding?"

**Structured Answer:**

```
Solution 1: CACHE (biggest wins first)
  → Add Redis cache-aside pattern.
  → If 80% of reads are for the same 20% of data (Pareto principle),
     cache hit rate = 80%. Effective DB load drops to 6,000 reads/sec. ✅

Solution 2: READ REPLICAS
  → Add 2 read replicas. Route all SELECT queries to replicas.
  → Primary handles writes only. Each replica handles 10K reads/sec.
  → Total capacity: 30K reads/sec. ✅

Solution 3: QUERY OPTIMIZATION
  → Run EXPLAIN ANALYZE. Find missing indexes, N+1 queries, full table scans.
  → A single missing index can cause 100x slowdown.
  → Optimize before scaling hardware.

Solution 4: CONNECTION POOLING
  → If you have 20 app servers each opening 100 DB connections = 2000 connections.
  → PostgreSQL degrades above ~500 connections.
  → Add PgBouncer (connection pooler) between app and DB.
```

### Q8: "SQL or NoSQL for a social media application? Justify your choice AND explain what you're sacrificing."

**Structured Answer:**

I'd use BOTH. Here's the breakdown:

  Why: Write-heavy (millions of posts/sec). Cassandra scales writes linearly.

  Trade-off: No JOINs. Can't do "SELECT posts JOIN users WHERE...".

SEARCH → Elasticsearch
  Why: Full-text search on posts and users.
  Trade-off: Eventual consistency (search results lag 1-2 seconds behind reality).
             Acceptable for search.

CACHE → Redis
  Why: Feed cache, session storage, trending topics counter.
  Trade-off: Data loss risk on restart (mitigate with AOF persistence).

### Q10: "Should I use WebSockets or polling for a real-time dashboard?"

**Structured Answer:**

```
DEPENDS on the update frequency:

Updates every 5+ seconds → POLLING (simple, stateless)
  → Client sends GET /api/dashboard every 5 seconds.
  → Pro: Stateless, works through any proxy/firewall, trivial to implement.
  → Con: Up to 5-second delay. Wastes bandwidth when no data has changed.
  → Fix: Use LONG POLLING (client waits for response — server responds 
         only when data changes. Connection held open for 30 sec max.)

Updates every < 1 second → WEBSOCKETS
  → Persistent bidirectional connection. Server pushes instantly.
  → Pro: True real-time (< 50ms). No wasted requests.
  → Con: Stateful connection. Harder to scale (must track which user is 
         connected to which server). Breaks through some corporate firewalls.

Updates to many clients (same data) → SERVER-SENT EVENTS (SSE)
  → One-directional: server → client. Simpler than WebSockets.
  → Pro: Built-in browser reconnection. HTTP-based (no firewall issues).
  → Con: One-directional only. Max 6 connections per domain in HTTP/1.1.

MY CHOICE for a dashboard: SSE or Long Polling.
  → Dashboard data changes every few seconds, not milliseconds.
  → SSE is simpler than WebSockets and sufficient for this use case.
```

### Q11: "Your payment service is down. Users keep clicking 'Pay'. What happens and how do you handle it?"

**Answer:**

```
WITHOUT protection:
  → Each click creates a new request to Payment Service.
  → Request hangs for 30 seconds (timeout), then fails.
  → User gets frustrated, clicks 5 more times.
  → 5 payment attempts queued. Service comes back. ALL 5 payments go through.
  → User charged 5 times. Nightmare.

WITH protection (4 layers):

1. IDEMPOTENCY KEY (prevents double-charging):
   → Frontend generates a unique key per payment: "order_789_pay_abc123"
   → Backend: If this key was already processed → return previous result.
   → Even 10 retries only produce 1 charge. ✅

2. BUTTON DEBOUNCING (prevents UI spam):
   → After user clicks "Pay", disable the button for 5 seconds.
   → Show a spinner. Prevent double-clicks at the source.

3. CIRCUIT BREAKER (prevents cascading failure):
   → After 5 consecutive failures to Payment Service, circuit opens.
   → Immediately return: "Payment is temporarily unavailable. Try again in 2 min."
   → No more 30-second hangs.

4. ASYNC QUEUE (graceful degradation):
   → Instead of failing, drop the payment into a Kafka queue.
   → Return: "Your payment is being processed. We'll notify you."
   → Worker retries when Payment Service recovers.
```

### Q12: "Your database is on fire — CPU at 100%, queries timing out. Users are seeing errors. Walk me through the next 30 minutes."

  → Enable "maintenance mode" for non-critical features.

MINUTE 10-30: ROOT CAUSE FIX
  → Add the missing index (CREATE INDEX CONCURRENTLY — no downtime).
  → Optimize the offending query.
  → If it was a traffic spike — add Redis cache for the hot query.
  → If it was a batch job — move it to run on the replica during off-hours.

POST-INCIDENT: PREVENT RECURRENCE
  → Set up slow query logging (log queries > 1 second).
  → Add alerting: "DB CPU > 80% for 2 minutes → PagerDuty alert."
  → Post-mortem document for the team.

## Category 7: ⚖️ CAP Theorem & Consistency

---

### Q14: "Your e-commerce site shows 'In Stock' but when the user clicks Buy, it says 'Out of Stock'. What caused this and how do you fix it?"

**Answer:**

```
CAUSE: Eventual Consistency.

  The product page is served from a READ REPLICA or CACHE that hasn't 
  received the latest inventory update yet.
  
  Timeline:
    T0: Last unit sold. Primary DB: stock = 0
    T1: Replica still has: stock = 1 (replication lag = 500ms)
    T2: User sees "In Stock" (read from replica) → clicks Buy
    T3: Checkout queries PRIMARY → stock = 0 → "Out of Stock" ❌

FIX (3 options):

  1. STRONG CONSISTENCY AT CHECKOUT:
     → Product page can use eventual consistency (cache/replica). Fine.
     → But the "Buy" button MUST query the PRIMARY DB:
       SELECT stock FROM products WHERE id = 123 FOR UPDATE;
     → Shows accurate stock at the moment of purchase.

  2. OPTIMISTIC UI + APOLOGY:
     → Show "In Stock" from cache (fast).
     → At checkout, validate against primary. If out of stock:
       "Sorry, this item just sold out. We'll notify you when it's back."
     → This is what Amazon does. Rare enough to be acceptable.

  3. STOCK RESERVATION:
     → When user adds to cart, RESERVE 1 unit in Redis (TTL = 10 min).
     → Decrement available count in Redis immediately.
     → Other users see updated count. No surprise at checkout.
     → If user abandons cart, TTL expires, stock is released.
```

---

### Q15: "When would you choose AP over CP? Give a real example."

**Answer:**

```
CHOOSE AP (Availability over Consistency) when:
  → "Stale data for a few seconds is OK, but downtime is NOT."

EXAMPLE: Shopping Cart (Amazon)

  User adds item to cart on Phone → Replicated to all data centers.
  Network partition between US-East and US-West data centers.

  CP approach: Block ALL cart operations until partition heals.
    → User can't add items to cart. Loses the sale. ❌

  AP approach: Both data centers accept cart writes independently.
    → User in US-East adds "Shoes". User in US-West adds "Shirt" (same account).
    → When partition heals, MERGE both carts: Shoes + Shirt.
    → Conflict resolution: "last writer wins" or "union merge".
    → User might see a briefly inconsistent cart, but NEVER sees downtime. ✅

  Amazon uses AP for carts (DynamoDB is AP by default).
  Amazon uses CP for inventory/payment (can't oversell or double-charge).
```

### Q16: "Explain the PACELC theorem in one sentence, then give an example."

**Answer:**

```
ONE SENTENCE: "During a Partition, choose Availability or Consistency. 
              Else (no partition), choose Latency or Consistency."

EXAMPLE:
  DynamoDB (PA/EL):
    → During partition → stays Available (serves stale data)
    → No partition → optimizes for Low Latency (async replication)
    → Result: FAST reads, but you might read slightly stale data.

  Google Spanner (PC/EC):
    → During partition → stays Consistent (blocks reads until sync)
    → No partition → STILL prioritizes Consistency (synchronous replication)
    → Result: Globally consistent reads, but higher latency (~10ms).
```

### Q19: "When should you use WebSockets vs SSE vs Long Polling?"

### Q20: "What's the difference between a Forward Proxy and a Reverse Proxy?"

**Answer:**

```
FORWARD PROXY (client-side):
  → Sits in front of CLIENTS. Hides client's identity from server.
  → Client KNOWS it's using a proxy (configured in browser/OS).
  → Use: VPN, corporate content filtering, bypass geo-restrictions.
  → Example: Company blocks facebook.com via forward proxy.

REVERSE PROXY (server-side):
  → Sits in front of SERVERS. Hides server's identity from client.
  → Client does NOT know a proxy exists (transparent).
  → Use: Load balancing, SSL termination, caching, security, DDoS protection.
  → Example: NGINX, Cloudflare, AWS ALB.

KEY INSIGHT:
  Every Load Balancer is a Reverse Proxy.
  But a Reverse Proxy can work with EVEN 1 backend server 
  (SSL termination, caching, security — still useful).
```

## Category 9: 🔑 Distributed IDs & Data Partitioning

### Q21: "How would you generate unique order IDs across 10 microservices, each running 50 instances?"

**Answer:**

```
OPTION 1: Twitter Snowflake (BEST for this scale) ⭐
  → 64-bit ID: [41-bit timestamp][10-bit machine_id][12-bit sequence]
  → Each of 500 instances gets a unique machine_id (0-1023).
  → Each instance generates 4,096 IDs per millisecond independently.
  → IDs are time-sortable. No coordination needed after setup.
  → Total: 500 × 4,096 = 2 million unique IDs per millisecond. ✅

OPTION 2: UUID v4 (simplest but wasteful)
  → Each instance generates random 128-bit UUIDs.
  → No coordination at all. But 36-char IDs waste storage,
    and random UUIDs hurt B-tree index performance.

OPTION 3: ULID (if you want string IDs that sort chronologically)
  → 26-char string: timestamp prefix + randomness.
  → Lexicographically sortable. No coordination.

MY CHOICE: Snowflake.
  → 64-bit (compact, fits in BIGINT).
  → Time-sortable (great for DB indexed, range queries by date).
  → High throughput (millions/sec).
  → Trade-off: need to assign machine_ids at startup (Zookeeper or config).
```

### Q22: "Your users table has 500 million rows and queries are slow. How do you shard it?"

**Answer:**

```
Step 1: CHOOSE A PARTITION KEY
  → user_id (hash-based). Why?
    → Most queries are per-user: "get user profile", "get user settings".
    → All data for one user lives on one shard → no cross-shard joins.

Step 2: CHOOSE NUMBER OF SHARDS
  → 500M rows / 50M per shard = 10 shards.
  → shard_id = hash(user_id) % 10

Step 3: USE CONSISTENT HASHING (not modulo)
  → If I later need 12 shards, modulo rehashes EVERYTHING.
  → Consistent hashing with virtual nodes: only ~10% of keys migrate.

Step 4: HANDLE CROSS-SHARD QUERIES
  → "Find all users named 'Aashish'" → touches ALL 10 shards (scatter-gather).
  → Fix: Build a search index (Elasticsearch) that has user_id + name.
    → Search ES → get user_ids → query the correct shard by user_id.

Step 5: HANDLE HOTSPOTS
  → Celebrity user_id gets 100x more reads.
  → Cache their data in Redis. If extremely hot, add a dedicated cache layer.

WHAT I DID NOT DO (and why):
  → Range-based sharding by user_id (1-50M on shard 1, etc.)
    → Bad because new signups all hit the last shard. Uneven writes.
  → Sharding by country
    → Bad because India has 60% of users → one shard is 6x bigger.
```

### Q23: "Why is UUID v4 bad for database primary keys at scale?"

**Answer:**

```
B-tree indexes store keys in SORTED ORDER.

UUID v4 is RANDOM: 
  f47ac10b → Insert at page 50
  a3b8d1b6 → Insert at page 12
  c5dae3d8 → Insert at page 37
  
  Every insert goes to a DIFFERENT page → constant PAGE SPLITS.
  Pages in cache are evicted because you're touching random locations.
  Write throughput degrades 2-5x compared to sequential IDs.

Snowflake/ULID IDs are TIME-SORTED:
  1709312400001 → Insert at the END of the index
  1709312400002 → Insert at the END
  1709312400003 → Insert at the END

  Sequential appends → no page splits → great cache locality.
  10x better index write performance.

RULE: If you need uniqueness + ordering → Snowflake or ULID.
      If you ONLY need uniqueness and don't care about DB performance → UUID v4.
```

## Category 10: 🔬 Bloom Filters & Probabilistic Data Structures

---

### Q24: "How would you check if a username is taken WITHOUT querying the database every time?"

**Answer:**

```
NAIVE APPROACH:
  User types "coolname123" → SELECT FROM users WHERE username = 'coolname123'
  10M signups/day → 10M DB queries just for availability checks. DB overloaded.

BLOOM FILTER APPROACH:
  1. On server start, load all existing usernames into a Bloom Filter (~1.14 MB for 10M names).
  2. User types "coolname123":
     → Check Bloom Filter:
       → "DEFINITELY NOT TAKEN" → Show ✅ instantly. No DB query.
       → "MAYBE TAKEN" → Query DB to confirm.
  3. If username is new, add it to the Bloom Filter + DB.

  RESULT:
  → 95% of availability checks answered by Bloom Filter (in-memory, <1ms).
  → Only 5% hit the DB (the "maybe" cases + actual duplicates).
  → DB queries reduced by 95%. ✅

  FALSE POSITIVE IMPACT:
  → Bloom filter says "taken" but it's actually available → user picks another name.
  → Annoying but not dangerous. Acceptable trade-off for 95% fewer DB queries.
```

---

### Q25: "What's a Bloom Filter and when would you use one in a system design?"

**Answer:**

```
WHAT: A space-efficient bit array that answers:
  → "DEFINITELY NOT in set" (100% accurate)
  → "PROBABLY in set" (may have false positives, never false negatives)

WHEN TO USE (3 patterns):
  1. AVOIDING EXPENSIVE LOOKUPS:
     → "Does this key exist in the DB?" Check Bloom Filter first.
       If NO → skip DB query entirely. (Cassandra uses this for every read.)
  
  2. DEDUPLICATION AT SCALE:
     → "Have I already crawled this URL?" (Google's web crawler)
     → HashSet for 10 billion URLs = 500 GB. Bloom Filter = 1.13 GB.
  
  3. SPAM / ABUSE DETECTION:
     → "Is this email in the blacklist?" Bloom Filter check in <1ms
       vs. querying a 100M-row database.

MEMORY COMPARISON:
  10 million items, 1% false positive rate:
  → HashSet: ~500 MB
  → Bloom Filter: ~1.14 MB (430x smaller!)
```

### Q29: "How would you design authentication for a microservices system?"

**Answer:**

```
ARCHITECTURE:

  Client → API Gateway (single auth point) → Internal Services

  FLOW:
  1. User logs in → API Gateway validates credentials → Returns JWT 
     (access token: 15 min, refresh token: 30 days in httpOnly cookie)
  
  2. Subsequent requests:
     Client sends JWT in Authorization header.
     API Gateway validates JWT signature (no DB lookup — stateless ✅).
     If valid → forwards request to internal service with header:
       X-User-ID: 123
       X-User-Role: admin
  
  3. Internal services TRUST the gateway. No re-authentication.
     They only check authorization: "Can role 'admin' delete this resource?"

  SERVICE-TO-SERVICE AUTH:
  → Internal services use mTLS (mutual TLS).
  → Each service has a certificate. Only authorized services can communicate.
  → Prevents rogue services from calling internal APIs.

  TOKEN REVOCATION (the hard part):
  → JWT can't be invalidated before expiry (stateless = no server-side state).
  → Fix: Keep access tokens SHORT (15 min).
  → On logout: blacklist the refresh token in Redis (small set, fast lookup).
  → On next access token expiry, user can't get a new one → effectively logged out.
```

# 21. Problem 14: CDN / Edge Network (Cloudflare)

## Requirements
- Serve cached content from edge servers closest to the user
- 200+ Points of Presence (PoPs) worldwide
- Support caching for static assets (JS, CSS, images) and dynamic content (short TTL)
- Cache invalidation (purge API, TTL-based, versioned URLs)
- DDoS protection at the edge
- SSL termination at edge (free HTTPS)

## Scale Estimation
| Metric | Value |
| :--- | :--- |
| Global requests/sec | 50 million |
| Cache hit rate target | > 95% |
| Edge PoPs | 200+ cities |
| Avg cached response | 50KB |
| Purge propagation time | < 5 seconds globally |

## Architecture
```
User → DNS (Anycast) → Nearest Edge PoP → Cache Check
                                             │
                              ┌───────────────┼───────────────┐
                              ▼                               ▼
                        Cache HIT ✅                    Cache MISS ❌
                        Return instantly                      │
                        (< 10ms)                              ▼
                                                     Origin Server
                                                     (customer's server)
                                                          │
                                                     Cache response
                                                     at edge + return
                                                     to user

Purge Flow:
  Customer calls: POST /api/purge { url: "/product/123" }
  → Central Control Plane → Fan-out to ALL 200 PoPs
  → Each PoP deletes cached entry → < 5 sec globally
```

## Key Design Decisions
| Decision | Choice | Why |
| :--- | :--- | :--- |
| DNS routing | **Anycast** | Same IP announced from all PoPs. BGP routes user to nearest. No GeoDNS complexity. |
| Cache storage at edge | **In-memory (LRU) + SSD** | Hot content in RAM (sub-ms). Warm content on SSD.  Cold content → origin fetch. |
| Cache invalidation | **TTL + Purge API + Versioned URLs** | TTL for automatic expiry. Purge for emergencies. Versioned URLs for static assets. |
| SSL | **Terminate at edge** | Edge handles HTTPS → plain HTTP to origin. Reduces origin CPU. Free SSL via Let's Encrypt. |

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Cache stampede** | TTL expires, 10K users request same resource simultaneously → 10K origin requests | **Lock at the edge**: first request fetches from origin. Others wait for cache to be populated. Only 1 origin request. |
| **Cache storage at PoP** | Popular PoP has 10TB of cached content → SSD full | **LRU eviction**. Least recently accessed content evicted first. Hot content stays. Cold content re-fetched on demand. |
| **Purge propagation** | Customer purges URL but edge in Tokyo still shows stale content for 30 seconds | **Pub/Sub fan-out**: Central control plane publishes purge to Kafka/Redis Pub/Sub. Each PoP subscribes. Propagation < 5 seconds. |
| **DDoS at the edge** | Attacker sends 100 Gbps traffic to one PoP | **Anycast absorbs** — traffic automatically spreads across ALL PoPs (100 Gbps / 200 PoPs = 500 Mbps each — trivial). |
| **TLS handshake overhead** | Each new connection = TLS handshake (~50ms) | **TLS session resumption** + **HTTP/3 QUIC** (0-RTT connection establishment). |

# 22. Problem 15: Authentication & SSO Service (Auth0/Okta)

## Requirements
- User registration and login (email/password)
- OAuth 2.0 / OpenID Connect ("Login with Google/GitHub")
- Single Sign-On (SSO) across multiple applications
- Multi-Factor Authentication (MFA — OTP, TOTP, push notification)
- Token management (JWT access + refresh tokens)
- Role-Based Access Control (RBAC)
- Rate limiting on login attempts (brute-force protection)

## Architecture
```
┌──────────┐         ┌──────────────┐         ┌──────────────┐
│  Client  │────────▶│  API Gateway │────────▶│  Auth Service│
│ (App/Web)│         │  (rate limit │         │              │
└──────────┘         │   + CORS)    │         │  ┌─────────┐ │
                     └──────────────┘         │  │ User DB │ │
                                              │  │(Postgres)│ │
                                              │  └─────────┘ │
                                              │              │
                                              │  ┌─────────┐ │
                                              │  │  Redis   │ │
                                              │  │(sessions,│ │
                                              │  │ tokens,  │ │
                                              │  │ rate lim)│ │
                                              │  └─────────┘ │
                                              └──────┬───────┘
                                                     │
                                              ┌──────▼───────┐
                                              │  External    │
                                              │  OAuth       │
                                              │  (Google,    │
                                              │   GitHub)    │
                                              └──────────────┘

Token Flow:
  Login → Auth Service validates → Issues JWT (15 min) + Refresh Token (30 days)
  API call → API Gateway validates JWT signature (no DB call!) → Forward
  Token expired → Client uses Refresh Token → Get new JWT
  Logout → Blacklist refresh token in Redis → Next JWT expires naturally (15 min)
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Brute force attacks** | Attacker tries 100K passwords/sec on one account | **Rate limiting**: 5 attempts per account per 15 min. Then CAPTCHA. Then account lockout for 30 min. Store counters in Redis. |
| **JWT revocation** | User changes password. Old JWT still valid for 15 min. | Keep JWT lifetime SHORT (15 min). On password change, blacklist ALL refresh tokens in Redis. Old JWTs expire naturally. |
| **Credential stuffing** | Attacker uses leaked password database from another site | **MFA** (mandatory for high-risk accounts). **Breached password check** (Bloom filter against HaveIBeenPwned list). **IP reputation** (block known bad IPs). |
| **SSO across 20 apps** | User logs into App A. Expects to be logged into App B too. | **Centralized session**: Auth Service stores session. All apps check with Auth Service. JWT carries `session_id`. If session is valid → auto-login. |
| **Token storage on client** | JWT stored in localStorage → stolen via XSS attack | Store JWT in **httpOnly cookie** (JS can't access). Set `Secure` and `SameSite=Strict` flags. Never localStorage. |
| **Password storage** | DB leaked. Passwords visible? | **bcrypt** (work factor 12). Even with the hash, brute-forcing takes years. Never store plain text or MD5. |

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Log volume spike** | Deployment bug causes 100x error logging → 1TB/hour | **Kafka buffer** absorbs the spike. Add **sampling** — log 10% of DEBUG, 100% of ERROR. Rate-limit log volume per service. |
| **Elasticsearch storage** | 10TB/day × 30 days = 300TB. ES cluster is expensive. | **Tiered storage**: hot data (7 days) on SSD. Warm data (30 days) on HDD. Delete after 30 days. Or use **Loki** (10x cheaper, log labels instead of full-text indexing). |
| **Search latency** | Searching 300TB of logs takes 30 seconds | **Index design**: shard by date (1 index per day). Search queries specify date range → only query relevant shards. Add filters (service, level) to narrow scope. |
| **Alert fatigue** | 200 alerts/day → engineers ignore all of them | Every alert MUST be **actionable**. If it doesn't change behavior → delete it. Every alert has a **runbook**. Use alert deduplication and grouping. |
| **Trace storage** | Storing every trace for 500 services = massive data | **Sample traces**: store 1% of successful traces, 100% of error traces. Enough for debugging without storing everything. |
| **Collector failure** | FluentBit on a node crashes → logs from that node lost | **Disk buffer**: FluentBit writes to local disk before sending. On recovery, replays buffered logs. No data loss. |

# 24. Problem 17: Global DNS Service (Route 53 / Cloudflare DNS)

## Requirements
- Resolve domain names to IP addresses
- < 50ms resolution time globally
- Handle 1 billion DNS queries per day
- Support A, AAAA, CNAME, MX, TXT, NS records
- Health-check-based failover (if server is down, return backup IP)
- GeoDNS (return different IPs based on user's location)
- 100% uptime (DNS down = entire internet breaks for your domain)

## Architecture
```
User Query: "api.example.com"
  │
  ▼
┌─────────────────┐
│ Recursive        │   ISP's DNS resolver (caches results)
│ Resolver (ISP)   │
└────────┬────────┘
         │ (if not cached)
         ▼
┌─────────────────┐
│ Authoritative   │   YOUR DNS servers (globally distributed)
│ DNS Servers     │
│                 │   ┌──────────┐ ┌──────────┐ ┌──────────┐
│                 │   │  US PoP  │ │  EU PoP  │ │ Asia PoP │
│                 │   │ (Anycast)│ │ (Anycast)│ │ (Anycast)│
│                 │   └──────────┘ └──────────┘ └──────────┘
└────────┬────────┘
         │
         ▼
  Response: A record → 93.184.216.34 (TTL: 300 seconds)

Health-Check Failover:
  health_checker → pings primary server every 10 sec
  If primary DOWN → DNS returns BACKUP server IP
  TTL = 60 seconds → propagates in 1 minute
```

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **DNS is a SPOF** | If DNS is down, EVERYONE can't reach your site | **Anycast** across 30+ global PoPs. Even if 10 PoPs go down, 20 still serve. Multiple NS records (ns1, ns2, ns3, ns4). |
| **DDoS on DNS** | Attacker floods DNS servers with 10B queries/sec | **Anycast absorbs** — traffic spreads across all PoPs. **Rate limiting** per source IP. **Response Rate Limiting (RRL)** — limit identical responses to the same IP. |
| **Stale DNS cache** | You changed IP from old_server to new_server. ISP still caches old IP. | Set **low TTL** (60 seconds) before planned migrations. After migration, increase TTL back to 300s. Can't control ISP cache directly — they respect the TTL. |
| **GeoDNS accuracy** | User in India routed to US server (wrong geo lookup) | Use **multiple GeoIP databases** (MaxMind + IP2Location). Combine with **latency-based routing** — measure actual RTT, not just IP-based guess. |
| **DNS propagation delay** | Changed A record but some users still hit old IP for 48 hours | This is ISP caching. Lower TTL BEFORE the change. After 48h, all caches expire. Use TTL = 60s if fast failover is critical. |

# 25. Problem 18: Distributed Cache System (Redis Cluster / Memcached)

## Requirements
- In-memory key-value cache for 100M+ keys
- Sub-millisecond latency (< 1ms for 99th percentile)
- Support for strings, hashes, sorted sets, lists
- Horizontal scaling (add nodes without downtime)
- Data replication (no cache data loss on node failure)
- TTL-based expiry
- Cache eviction policies (LRU, LFU)

## Architecture
```
App Server → Cache Client (with consistent hashing) → Redis Cluster
                                                          │
                                           ┌──────────────┼──────────────┐
                                           ▼              ▼              ▼
                                     ┌──────────┐  ┌──────────┐  ┌──────────┐
                                     │  Shard 1  │  │  Shard 2  │  │  Shard 3  │
                                     │  Primary  │  │  Primary  │  │  Primary  │
                                     │ keys 0-5K │  │keys 5K-10K│  │keys 10K+│
                                     └─────┬─────┘  └─────┬─────┘  └─────┬─────┘
                                           │              │              │
                                     ┌─────▼─────┐  ┌─────▼─────┐  ┌─────▼─────┐
                                     │  Replica   │  │  Replica   │  │  Replica   │
                                     │  (standby) │  │  (standby) │  │  (standby) │
                                     └───────────┘  └───────────┘  └───────────┘

Key Distribution:
  hash("user:123") → Slot 5234 → Shard 2 (handles slots 5001-10000)
  
  Redis has 16,384 hash slots. Each shard owns a range of slots.
  Adding a new shard → Redis migrates some slots to the new node.
```

## Key Design Decisions
| Decision | Choice | Why |
| :--- | :--- | :--- |
| Partitioning | **Hash-slot based (Redis Cluster)** | 16,384 slots distributed across nodes. Adding/removing nodes = slot migration, not full rehash. |
| Replication | **Async replication to 1 replica** | Each primary has a standby. If primary dies, replica auto-promotes. Async = low write latency. |
| Eviction | **allkeys-lru** | When memory is full, evict least recently used keys first. Most caches follow Pareto — 20% of keys serve 80% of reads. |
| Persistence | **RDB snapshots + AOF** | RDB for fast restarts. AOF for durability (logs every write). On restart: load RDB + replay AOF. |

## 🔥 Where Does It Break?

| Bottleneck | Why | Fix |
| :--- | :--- | :--- |
| **Hot key** | One key (`product:iphone15`) gets 500K reads/sec → single shard bottleneck | **Read replicas** for hot shards. Or **local in-memory cache** (L1 cache) on the app server with 5-second TTL. Most reads don't even hit Redis. |
| **Cache stampede** | TTL expires on popular key. 10K threads hit DB simultaneously to repopulate. | **Lock-based refresh**: first thread acquires lock, fetches from DB, populates cache. Other threads wait and read from cache. Or **stale-while-revalidate**: serve stale, refresh in background. |
| **Node failure** | Primary node dies. 1 million keys lost until replica promotes. | **Async replication** means ~1 second of writes may be lost. For critical data (sessions), use **synchronous replication** (WAIT command). Auto-failover via Redis Sentinel in < 30 seconds. |
| **Memory full** | 100GB memory limit hit. New writes rejected. | **Eviction policies** (LRU/LFU). Monitor memory usage. Alert at 80%. Add nodes proactively. Set appropriate TTLs so expired keys are cleaned. |
| **Thundering herd on cold start** | Cache is empty after restart. 100% of traffic hits DB. | **Cache warming**: pre-load top 10K hot keys from DB on startup. Use RDB snapshots so restart loads most data from disk, not DB. |
| **Network partition** | App server can't reach Redis. Fail open (serve without cache) or fail closed (return errors)? | **Fail open** for most caches — serve from DB directly (slower but works). For rate limiting cache: **fail closed** (block requests rather than allow unlimited). Have local fallback counters. |
