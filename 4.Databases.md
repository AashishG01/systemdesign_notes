## 1. Indexing — The Speed Demon

### 1.1 What is an Index?

An **Index** is a separate, smaller data structure that a database maintains *alongside* the main table. Its sole purpose is to make data retrieval (SELECT queries) dramatically faster.

> **💡 Intuition — The Library Analogy:**
> Imagine a library with 10 million books scattered randomly on shelves.
> *   **Without an Index:** To find "Harry Potter," you must walk through every single shelf and check every single book's title. This is a **Full Table Scan** — O(N) time.
> *   **With an Index (Card Catalog):** You walk to the alphabetized card catalog, find the card for "Harry Potter" under "H," and it tells you: "Shelf 42, Position 7." You walk directly there. This is an **Index Lookup** — O(log N) time.

### 1.2 How Does an Index Actually Work? (B-Tree Algorithm)

The most common index structure used by PostgreSQL, MySQL, Oracle, and SQL Server is the **B-Tree (Balanced Tree)**.

**The Algorithm:**
1.  The database takes all the values in the indexed column (e.g., all emails in the `users` table).
2.  It sorts them alphabetically and builds a tree structure where each node can have multiple children.
3.  Every leaf node stores a **pointer** (the physical disk address) back to the actual row in the main table.

```
              ┌──────────────────────┐
              │     [M]              │   ← Root Node
              └──────┬───────┬───────┘
                     │       │
         ┌───────────┘       └───────────┐
         ▼                               ▼
   ┌──────────┐                    ┌──────────┐
   │  [D, H]  │                    │  [R, W]  │   ← Internal Nodes
   └──┬──┬──┬─┘                    └──┬──┬──┬─┘
      │  │  │                         │  │  │
      ▼  ▼  ▼                         ▼  ▼  ▼
    [A-C][E-G][I-L]               [N-Q][S-V][X-Z]  ← Leaf Nodes (with disk pointers)
```

**Searching for email = "john@doe.com" (starts with 'J'):**
1.  Compare 'J' with Root 'M' → J < M → Go Left.
2.  Compare 'J' with 'D' and 'H' → J > H → Go to the third child.
3.  Arrive at leaf `[I-L]` → Scan the leaf, find 'J', follow the pointer → **Row found!**
4.  **Total steps: 3.** In a table of 100 Million rows, a B-Tree with a branching factor of ~100 would need only **4-5 disk reads** to find any single row.

### 1.3 Types of Indexes

| Index Type | What it Does | When to Use |
| :--- | :--- | :--- |
| **Primary Index** | Automatically created on your Primary Key (`id`). Data on disk is physically sorted by this key. | Always exists. You don't manually create this. |
| **Secondary Index** | Created manually on non-Primary-Key columns (e.g., `email`, `username`). It's a separate B-Tree pointing back to the main table. | When you frequently `WHERE` filter or `ORDER BY` a specific column. |
| **Composite Index** | An index on multiple columns simultaneously, e.g., `CREATE INDEX idx ON orders(user_id, created_at)`. | When your queries filter by `WHERE user_id = X AND created_at > Y`. The order of columns in the index *matters*. |
| **Unique Index** | Like a Secondary Index, but it additionally enforces that no two rows can have the same value. | `email` columns, `username` columns — anything that must be globally unique. |
| **Hash Index** | Uses a Hash Table instead of a B-Tree. O(1) exact lookups. | When you *only* do exact `=` lookups and *never* range queries (`>`, `<`, `BETWEEN`). |
| **Full-Text Index** | Specialized for searching natural language text. | `WHERE description LIKE '%wireless headphones%'` — searching product descriptions, blog posts. |

### 1.4 The Trade-Off: Why Not Index Everything?

Every index you create has a cost:
1.  **Write Penalty:** Every single `INSERT`, `UPDATE`, or `DELETE` must now update *both* the main table AND every B-Tree index on that table. If you have 10 indexes on a table, a single INSERT must do 11 writes (1 for the table + 10 for the indexes).
2.  **Storage Cost:** Each B-Tree index takes up physical disk space. On a 50GB table with 5 indexes, you might have 100GB+ of index data alone.

> **⚡ The Golden Rule:** Index columns you `WHERE` on frequently. Never blindly index every column.

## 2. Query Optimization Basics

### 2.1 What is Query Optimization?

The **Query Optimizer** is the brain inside your database engine. When you write a SQL query, there are *many* possible ways to execute it. The optimizer's job is to find the fastest plan.

> **💡 Intuition — The GPS Analogy:**
> You want to drive from New York to Boston. There are 50 different possible routes. Some go through highways (fast), some go through tiny side roads (slow), some are shorter but have traffic.
> *   **The Query Optimizer is your GPS.** It evaluates all 50 routes, estimates the traffic and distance for each, and picks the one with the lowest estimated "cost." That cost is measured in disk reads, CPU time, and memory usage.

### 2.2 Common Optimization Techniques (What YOU Can Control)

| Technique | Bad Query ❌ | Good Query ✅ | Why? |
| :--- | :--- | :--- | :--- |
| **Avoid `SELECT *`** | `SELECT * FROM users` | `SELECT id, name FROM users` | `SELECT *` reads ALL columns from disk, including the 5MB `profile_photo` BLOB you didn't even need. |
| **Use Indexes on WHERE** | `SELECT * FROM orders WHERE status = 'shipped'` (no index on `status`) | `CREATE INDEX idx_status ON orders(status);` then run the query. | Without the index, the DB scans all 10M rows. With it, it jumps directly to 'shipped' rows. |
| **Avoid Functions on Indexed Columns** | `WHERE YEAR(created_at) = 2024` | `WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'` | Wrapping a column in a function (`YEAR()`) forces the DB to compute the function on *every single row*, bypassing the index entirely. |
| **Use `LIMIT`** | `SELECT * FROM logs ORDER BY created_at DESC` | `SELECT * FROM logs ORDER BY created_at DESC LIMIT 100` | Without LIMIT, the DB sorts and returns millions of rows. With LIMIT, it stops after 100. |
| **Use `EXISTS` over `IN` for subqueries** | `WHERE id IN (SELECT user_id FROM orders)` | `WHERE EXISTS (SELECT 1 FROM orders WHERE orders.user_id = users.id)` | `IN` materializes the entire subquery result into memory first. `EXISTS` stops after finding the first match. |

### 2.3 The EXPLAIN Plan (Reading the DB's Mind)

Every major database lets you prefix a query with `EXPLAIN` (or `EXPLAIN ANALYZE` in PostgreSQL) to see the exact execution plan the optimizer chose.

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'john@doe.com';
```

```
Index Scan using idx_users_email on users  (cost=0.43..8.45 rows=1 width=120)
  Index Cond: (email = 'john@doe.com'::text)
Planning Time: 0.1 ms
Execution Time: 0.05 ms   ← blazing fast!
```

**Key terms to look for:**
| Term | Meaning | Good or Bad? |
| :--- | :--- | :--- |
| **Seq Scan** | Sequential Scan — Reads *every single row* in the table. | 🔴 **Bad** (unless the table is tiny). You are missing an index. |
| **Index Scan** | Uses the B-Tree index to jump directly to the row. | 🟢 **Good.** This is what you want. |
| **Index Only Scan** | Even better — the index itself contains all the data needed, so the DB doesn't even need to read the main table. | 🟢🟢 **Excellent.** |
| **Bitmap Heap Scan** | A hybrid — the DB uses the index to find *which pages* contain matching rows, then reads those pages from disk. | 🟡 **Okay.** Common for queries returning many rows (e.g., 1000+). |
| **Sort** | The DB had to sort data in memory or on disk. | 🟡 Watch out — large sorts can spill to disk and be extremely slow. |
| **Hash Join / Nested Loop / Merge Join** | Different algorithms the DB uses to JOIN two tables. | Depends on context. Hash Join is fast for large data. Nested Loop is fast for small indexed lookups. |

## 3. ACID Properties — The 4 Sacred Promises

Every relational database (PostgreSQL, MySQL, Oracle) guarantees these 4 properties for every transaction. If even *one* is violated, the data is corrupt.

### 3.1 Atomicity — "All or Nothing"

**Definition:** A transaction is an indivisible unit. Either *every* operation in the transaction succeeds, or *none* of them do. There is no "half-finished" state.

> **💡 Real-World Scenario — Bank Transfer:**
> You transfer ₹10,000 from Account A to Account B.
> *   Step 1: `UPDATE accounts SET balance = balance - 10000 WHERE id = 'A';`
> *   Step 2: `UPDATE accounts SET balance = balance + 10000 WHERE id = 'B';`
> *   **The Danger:** What if the server crashes *after* Step 1 but *before* Step 2? ₹10,000 vanished from A but never appeared in B. Money has disappeared.
> *   **Atomicity Saves You:** The database wraps both steps in a TRANSACTION. If Step 2 fails, the database automatically **ROLLBACK**s Step 1. Account A gets its ₹10,000 back. No money is ever lost.

### 3.2 Consistency — "Rules Are Never Broken"

**Definition:** A transaction moves the database from one valid state to another valid state. All data integrity constraints (foreign keys, unique constraints, CHECK constraints) are respected.

> **Example:** You have a rule: `CHECK (balance >= 0)`. If Account A has ₹5,000 and you try to transfer ₹10,000 (which would make balance -₹5,000), the database rejects the entire transaction. It refuses to transition to an invalid state.

### 3.3 Isolation — "Transactions Don't See Each Other's Mess"

**Definition:** Multiple transactions running at the same time are isolated from each other. Transaction A cannot see the uncommitted, "dirty" changes of Transaction B.

> **💡 Real-World Scenario — Seat Booking:**
> User A and User B both try to book Seat #5 on a flight at the exact same time.
> *   Without Isolation: Both read "Seat #5 is available," both book it, and the airline sells 1 seat to 2 people.
> *   With Isolation: The database ensures that once User A starts modifying Seat #5, User B is either blocked or sees a consistent snapshot.

### 3.4 Durability — "Once Committed, It's Permanent"

**Definition:** Once a transaction is successfully committed (`COMMIT`), its changes are permanently saved to disk, even if the server immediately crashes 1 millisecond later.

> **How it works:** The database uses a **Write-Ahead Log (WAL)**. Before actually modifying the data on disk, the DB writes the intended change to a log file first. If the server crashes mid-write, on restart it replays the WAL log to recover the committed data.

## 4. Isolation Levels — Controlling Concurrency

Isolation (the "I" in ACID) isn't a binary on/off switch. It's a spectrum with 4 levels, each trading performance for safety.

### 4.1 The 3 Concurrency Anomalies (Problems to Prevent)

Before understanding the levels, you must understand the problems they prevent:

| Anomaly | What Happens | Example |
| :--- | :--- | :--- |
| **Dirty Read** | Transaction A reads uncommitted data from Transaction B. If B rolls back, A used "phantom" data that never existed. | A reads "Balance = ₹0" because B is in the middle of a transfer but hasn't committed yet. B then rolls back. A panics over a balance that was never real. |
| **Non-Repeatable Read** | Transaction A reads a row twice, and gets *different* values because Transaction B modified and committed it in between A's two reads. | A reads `price = ₹100`. B updates it to `₹150` and commits. A reads again and gets `₹150`. The price "changed" mid-transaction. |
| **Phantom Read** | Transaction A runs a range query twice (e.g., `WHERE age > 25`), and gets a *different number of rows* because Transaction B inserted a new matching row in between. | A counts 10 users with age > 25. B inserts a new user aged 30 and commits. A counts again and gets 11. A new "phantom" row appeared. |

### 4.2 The 4 Isolation Levels

| Level | Dirty Read | Non-Repeatable Read | Phantom Read | Performance | Use Case |
| :--- | :---: | :---: | :---: | :--- | :--- |
| **Read Uncommitted** | ⚠️ Possible | ⚠️ Possible | ⚠️ Possible | 🟢 Fastest | Almost never used. Only for rough analytics where approximate data is fine (e.g., "approximately how many users are online?"). |
| **Read Committed** | ✅ Prevented | ⚠️ Possible | ⚠️ Possible | 🟢 Fast | **Default in PostgreSQL & Oracle.** The most common setting. Good for typical web apps. |
| **Repeatable Read** | ✅ Prevented | ✅ Prevented | ⚠️ Possible | 🟡 Medium | **Default in MySQL (InnoDB).** When you need to read the same data twice in a transaction and it must be identical. |
| **Serializable** | ✅ Prevented | ✅ Prevented | ✅ Prevented | 🔴 Slowest | When absolute correctness is critical: Banking, Financial Transactions, Airline Seat Booking. Every transaction behaves as if it ran alone, sequentially, one after another. |

> **💡 Interview Tip:** You must know that PostgreSQL's default is **Read Committed** and MySQL InnoDB's default is **Repeatable Read**. Interviewers love asking this.

### 4.3 How Isolation Levels Are Implemented (Under the Hood)

Databases use two major strategies:

1.  **Pessimistic Concurrency (Locking):** The DB physically locks the rows being read/written. Other transactions must *wait* until the lock is released. Used heavily in Serializable level.
2.  **Optimistic Concurrency (MVCC — Multi-Version Concurrency Control):** The DB keeps *multiple versions* of each row. Transaction A sees version 1 of the data (a snapshot), while Transaction B can freely modify the data and create version 2. Neither blocks the other.

> **PostgreSQL uses MVCC**. This is why PostgreSQL is extraordinarily good at handling many concurrent readers and writers without locking them out. MySQL InnoDB also uses MVCC but combines it with traditional locking for certain operations.

## 5. Locking — Who Gets to Touch the Data?

Locks are the mechanism databases use to prevent two transactions from corrupting the same data simultaneously.

### 5.1 Why Do We Need Locks?

> **💡 Intuition — The Google Doc Analogy:**
> Imagine you and your colleague are editing the same sentence in a Google Doc at the exact same millisecond. You type "Hello" while they type "Goodbye" over the same words. Without any conflict resolution, the result is garbled nonsense like "HGoeloldbye." 
> A **Lock** is like forcing one person to wait until the other finishes their edit before touching the same sentence.

### 5.2 Types of Locks

| Lock Type | What it Does | Blocks Who? | Example |
| :--- | :--- | :--- | :--- |
| **Shared Lock (S Lock / Read Lock)** | Allows multiple transactions to **read** the same row simultaneously. | Blocks anyone trying to **write** to that row. | 100 users are viewing a product page. All 100 hold a Shared Lock on the `products` row. They can all read it. But if an admin tries to update the price, they must wait. |
| **Exclusive Lock (X Lock / Write Lock)** | Gives ONE transaction the sole right to **read AND write** a row. No one else can touch it. | Blocks **everyone** (readers AND writers). | User A is transferring money. The DB puts an Exclusive Lock on Account A's row. No one else can even read the balance until the transfer is fully committed. |
| **Row-Level Lock** | Locks only the specific row(s) being modified. Other rows in the same table are untouched. | Only transactions touching the *same row*. | The **gold standard** for concurrency. PostgreSQL and MySQL InnoDB use row-level locking by default. |
| **Table-Level Lock** | Locks the *entire table*. No one can read or write *any* row in the table. | **Everyone** accessing the table. | Extremely coarse-grained. Used by older engines like MyISAM (MySQL). Terrible for concurrency. Avoid. |
| **Optimistic Lock** | Not a real DB lock. The application adds a `version` column. Before updating, it checks if the version matches. If not, the update is rejected ("someone else changed it first"). | No one is blocked. Conflicts are detected *after* the work, not before. | Perfect for systems with lots of reads and very rare writes (e.g., editing a user profile). |
| **Pessimistic Lock** | A real DB lock. The application explicitly asks the DB to lock the row using `SELECT ... FOR UPDATE`. | All other transactions trying to access that row. | Used when collisions are *very likely*, like booking the last concert ticket. You *must* lock it before reading. |

### 5.3 Lock Compatibility Matrix

```
              ┌───────────────────────────────────────┐
              │ Can these two locks coexist on the     │
              │ SAME row at the SAME time?             │
              ├───────────────┬───────────┬────────────┤
              │               │ Shared(S) │ Exclusive(X)│
              ├───────────────┼───────────┼────────────┤
              │  Shared (S)   │   ✅ YES  │   ❌ NO    │
              │  Exclusive(X) │   ❌ NO   │   ❌ NO    │
              └───────────────┴───────────┴────────────┘
```

**Key Takeaway:** Multiple readers can coexist (S + S = ✅). But a writer (X) blocks everyone, including other writers and readers.

## 6. Deadlocks — When Everyone Waits Forever

### 6.1 What is a Deadlock?

A **Deadlock** occurs when two or more transactions are each waiting for the other to release a lock, creating a circular dependency where *no one can ever proceed*.

> **💡 Intuition — The Narrow Bridge:**
> Two cars approach a single-lane bridge from opposite sides at the same time. Car A enters from the left and needs the right side to exit. Car B enters from the right and needs the left side to exit. Neither can reverse. Both are stuck forever.

### 6.2 Deadlock Diagram

```
    Transaction A                    Transaction B
    ─────────────                    ─────────────
    Step 1: LOCK Row #1 ✅           Step 1: LOCK Row #2 ✅
            (Holds Lock on Row #1)           (Holds Lock on Row #2)

    Step 2: Wants to LOCK Row #2     Step 2: Wants to LOCK Row #1
            ⏳ WAITING...                    ⏳ WAITING...
            (Row #2 is locked by B!)         (Row #1 is locked by A!)

                        ╔═══════════════╗
                        ║   DEADLOCK!   ║
                        ║ Both wait     ║
                        ║ forever.      ║
                        ╚═══════════════╝
```

### 6.3 Real-World Deadlock Example

**Scenario:** A banking app has two concurrent transfers:
*   **Transaction A:** Transfer ₹100 from Account #1 → Account #2.
*   **Transaction B:** Transfer ₹200 from Account #2 → Account #1.

| Time | Transaction A | Transaction B |
| :---: | :--- | :--- |
| T1 | `UPDATE accounts SET balance = balance - 100 WHERE id = 1;` → **Locks Row 1** | `UPDATE accounts SET balance = balance - 200 WHERE id = 2;` → **Locks Row 2** |
| T2 | `UPDATE accounts SET balance = balance + 100 WHERE id = 2;` → ⏳ **BLOCKED!** (Row 2 is locked by B) | `UPDATE accounts SET balance = balance + 200 WHERE id = 1;` → ⏳ **BLOCKED!** (Row 1 is locked by A) |
| T3 | **💀 DEADLOCK DETECTED** | **💀 DEADLOCK DETECTED** |

### 6.4 How Databases Handle Deadlocks

1.  **Detection:** The DB runs a background **Wait-For Graph** algorithm. It builds a directed graph of "who is waiting for whom." If it finds a **cycle** in the graph, a deadlock exists.
2.  **Resolution:** The DB picks one of the deadlocked transactions as the **victim** (usually the one that has done the least work) and forcefully **ROLLBACK**s it. The other transaction can now proceed.
3.  **Your Application's Job:** Your code must catch the deadlock error and **retry** the failed transaction.

### 6.5 How to PREVENT Deadlocks

| Strategy | How it Works |
| :--- | :--- |
| **Lock in a Consistent Order** | Always lock resources in the same global order. If every transaction locks Account #1 before Account #2 (sorted by ID), the circular dependency can never form. |
| **Keep Transactions Short** | The shorter the transaction duration, the less time locks are held, and the lower the chance of conflict. |
| **Use Timeouts** | Set `lock_timeout = '5s'`. If a transaction can't acquire a lock within 5 seconds, it automatically gives up and retries instead of waiting forever. |
| **Avoid `SELECT ... FOR UPDATE` when possible** | Use **Optimistic Locking** (version numbers) instead of pessimistic locking to eliminate the need for explicit locks. |

## 7. Transactions — The Complete Picture

### 7.1 What is a Transaction?

A **Transaction** is a sequence of one or more SQL operations that are treated as a **single, atomic unit of work**. Either all operations succeed and the changes are permanently saved (`COMMIT`), or if any operation fails, all changes are undone (`ROLLBACK`).

### 7.2 Transaction Lifecycle

```sql
BEGIN;                                              -- Start the transaction

UPDATE accounts SET balance = balance - 500 WHERE id = 1;  -- Step 1: Debit
UPDATE accounts SET balance = balance + 500 WHERE id = 2;  -- Step 2: Credit

-- Now decide:
COMMIT;    -- ✅ Success! Both changes are permanently saved to disk.
-- OR
ROLLBACK;  -- ❌ Something went wrong. Undo BOTH changes. DB goes back to its original state.
```

### 7.3 Savepoints — Partial Rollbacks

Sometimes you want to undo *part* of a transaction without aborting the entire thing.

```sql
BEGIN;

INSERT INTO orders (user_id, product_id) VALUES (1, 101);  -- Step 1: Create order ✅

SAVEPOINT before_payment;                                   -- Mark a checkpoint

UPDATE wallets SET balance = balance - 999 WHERE user_id = 1;  -- Step 2: Deduct wallet

-- Oops! Payment failed (insufficient balance).
ROLLBACK TO SAVEPOINT before_payment;                       -- Undo ONLY Step 2. Step 1 (order) is safe.

-- We can try an alternative payment method here...
UPDATE credit_cards SET charged = true WHERE user_id = 1;   -- Step 3: Charge credit card instead

COMMIT;  -- Both Step 1 and Step 3 are committed. Step 2 was gracefully undone.
```

### 7.4 Auto-Commit Mode

In most databases, if you don't explicitly write `BEGIN`, every single SQL statement is automatically wrapped in its own tiny transaction and immediately committed. This is called **Auto-Commit mode**.

*   `INSERT INTO users (name) VALUES ('Alice');` → Automatically committed immediately.
*   This is fine for simple operations but dangerous for multi-step operations like bank transfers where you need atomicity across multiple statements.

## 8. Normalization vs Denormalization

### 8.1 What is Normalization?

**Normalization** is the process of organizing your database tables to **eliminate data redundancy** (storing the same data in multiple places) and **prevent update anomalies**.

> **💡 Intuition — The Filing Cabinet:**
> Imagine you store every customer's full address inside every single order they've ever placed. If the customer moves, you must find and update the address in *every single order row* (hundreds of rows). Miss one? Now you have two conflicting addresses for the same customer. **Normalization** says: store the address in ONE place (the `customers` table) and just reference it by `customer_id` from the `orders` table.

### 8.2 The Normal Forms (1NF → 2NF → 3NF)

#### First Normal Form (1NF) — "No Repeating Groups"
**Rule:** Every cell in the table must contain a *single, atomic value*. No lists, no arrays, no comma-separated values.

| ❌ Violates 1NF | ✅ Satisfies 1NF |
| :--- | :--- |
| `student_id=1, courses="Math, Physics, CS"` | Row 1: `student_id=1, course="Math"` |
|  | Row 2: `student_id=1, course="Physics"` |
|  | Row 3: `student_id=1, course="CS"` |

#### Second Normal Form (2NF) — "No Partial Dependencies"
**Rule:** Must be in 1NF + every non-key column must depend on the *entire* Primary Key (not just part of it). This only matters for **composite keys**.

| ❌ Violates 2NF (Composite PK: `student_id, course_id`) | Problem |
| :--- | :--- |
| `student_id, course_id, student_name, course_name, grade` | `student_name` depends ONLY on `student_id`, not on the full PK (`student_id, course_id`). This is a partial dependency. |

**Fix:** Split into 3 tables: `students(id, name)`, `courses(id, name)`, `enrollments(student_id, course_id, grade)`.

#### Third Normal Form (3NF) — "No Transitive Dependencies"
**Rule:** Must be in 2NF + no non-key column should depend on *another non-key column*.

| ❌ Violates 3NF | Problem |
| :--- | :--- |
| `employee_id, department_id, department_name` | `department_name` depends on `department_id`, not on `employee_id`. If HQ renames "Engineering" to "Product Development," you'd have to update every employee row in that department. |

**Fix:** Split `department_name` into a separate `departments` table. The `employees` table only stores the `department_id` foreign key.

### 8.3 What is Denormalization?

**Denormalization** is the *intentional* process of adding redundancy back into your tables to **improve read performance** at the cost of write complexity.

> **💡 Intuition:** Normalization optimizes for **data integrity and writes**. Denormalization optimizes for **read speed**. In System Design, you almost always encounter trade-offs between the two.

### 8.4 When to Denormalize

| Scenario | Why Denormalize |
| :--- | :--- |
| **Read-Heavy Dashboard** | A CEO dashboard shows `order_id, customer_name, product_name, total`. With full normalization, this requires JOINing 3 tables (orders, customers, products) on every single page load. With denormalization, you pre-store `customer_name` directly in the `orders` table, eliminating the JOIN entirely. |
| **NoSQL Databases** | NoSQL databases like MongoDB have no JOINs at all. You *must* embed related data inside the same document (denormalize). |
| **Caching Layers** | When you put data into Redis, you store a pre-computed, denormalized JSON blob of the final response, not 3 separate normalized tables. |

### 8.5 Normalization vs Denormalization Summary

| Aspect | Normalization | Denormalization |
| :--- | :--- | :--- |
| **Data Redundancy** | ❌ Zero redundancy | ✅ Intentional redundancy |
| **Write Performance** | ✅ Fast writes (update one place) | ❌ Slower writes (update multiple places) |
| **Read Performance** | ❌ Slower reads (need JOINs) | ✅ Faster reads (no JOINs) |
| **Data Integrity** | ✅ High (single source of truth) | ⚠️ Risk of inconsistency |
| **Best For** | OLTP (Online Transaction Processing) — banking, e-commerce orders | OLAP (Online Analytical Processing) — dashboards, reports, analytics |

## 9. SQL vs NoSQL — When to Use What

This is the most common database question in every System Design interview. The answer is never "one is better." It's always "It depends."

### 9.1 SQL (Relational Databases)

**Examples:** PostgreSQL, MySQL, Oracle, SQL Server, SQLite.

**Core Philosophy:** Data is organized in **tables with rows and columns**, connected by **relationships** (Foreign Keys). All operations must follow **ACID** guarantees.

**Best For:**
*   Structured, well-defined data with clear relationships (Users → Orders → Products).
*   When data integrity is **non-negotiable** (Banking, Insurance, Healthcare).
*   When you need complex queries with multi-table JOINs, aggregations, and transactions.

### 9.2 NoSQL (Non-Relational Databases)

**Examples:** MongoDB, Redis, Cassandra, DynamoDB, Neo4j.

**Core Philosophy:** Data is stored in flexible formats (JSON documents, key-value pairs, graphs). There is no fixed schema. Prioritizes **scalability and speed** over strict consistency.

**Best For:**
*   Unstructured or semi-structured data (JSON logs, sensor data, social media feeds).
*   When you need **massive horizontal scalability** (millions of writes per second).
*   When your data model changes frequently (agile, early-stage startups).

### 9.3 The Decision Matrix

| Question to Ask | If SQL Wins → | If NoSQL Wins → |
| :--- | :--- | :--- |
| **Is the data highly relational?** (Users, Orders, Products linked by FK) | ✅ SQL — JOINs are purpose-built for this. | ❌ |
| **Do I need ACID transactions?** (Money, medical records) | ✅ SQL — Rock-solid guarantees. | ❌ |
| **Is the schema well-defined and stable?** | ✅ SQL — Tables with strict columns. | ❌ |
| **Do I need to scale writes to millions/sec?** | ❌ | ✅ NoSQL — Horizontal sharding by design. |
| **Is the data unstructured?** (Logs, IoT sensor data) | ❌ | ✅ NoSQL — Schema-less flexibility. |
| **Do I need ultra-low latency reads?** (< 1ms) | ❌ | ✅ NoSQL (Redis) — In-memory, O(1) lookups. |
| **Is the data a graph?** (Social networks, recommendations) | ❌ | ✅ NoSQL (Neo4j) — Graph traversals are native. |

> **💡 The Real-World Answer:** Most production systems use **BOTH**. For example, Uber uses PostgreSQL for trip records (relational, ACID) and Redis for real-time driver locations (ultra-fast, key-value).

## 10. NoSQL Systems — The 4 Families

NoSQL is not one thing. It's a category containing 4 fundamentally different types of databases. Choosing the wrong one is a critical design mistake.

### 10.1 Key-Value Stores

**What it is:** The simplest NoSQL model. It's literally a giant hash map: `Key → Value`. Nothing else. No schema, no columns, no relationships.

**Examples:** **Redis**, Memcached, Amazon DynamoDB (can also be Document DB).

| Key | Value |
| :--- | :--- |
| `user:123:session` | `{ "logged_in": true, "cart": [101, 205] }` |
| `product:456:views` | `1,532,897` |
| `rate_limit:ip:192.168.1.1` | `47` |

**Operations:** Only 3: `GET(key)`, `SET(key, value)`, `DELETE(key)`. All are O(1).

**When to Use:**
*   **Caching:** Store the results of expensive DB queries in Redis. Serve them in 0.5ms instead of 50ms.
*   **Session Management:** Store user session tokens.
*   **Rate Limiting:** Track how many API calls an IP address has made in the last minute.
*   **Leaderboards / Counters:** Redis's `INCR` command atomically increments a counter.

**When NOT to Use:** When you need JOINs, complex queries, or relationships. You cannot query by value, only by key.

### 10.2 Document Databases

**What it is:** Stores data as self-contained **JSON-like documents**. Each document can have a completely different structure. Think of it as a Key-Value store where the value is always a rich, queryable JSON object.

**Examples:** **MongoDB**, CouchDB, Amazon DocumentDB, Firestore.

```json
// Document in the "products" collection
{
  "_id": "prod_456",
  "name": "Wireless Headphones",
  "price": 2999,
  "category": "Electronics",
  "reviews": [
    { "user": "Alice", "rating": 5, "text": "Amazing sound!" },
    { "user": "Bob", "rating": 4, "text": "Good battery life." }
  ],
  "specs": {
    "weight": "250g",
    "bluetooth": "5.3",
    "battery_hours": 40
  }
}
```

**When to Use:**
*   **Content Management Systems (CMS):** Blog posts, articles — each can have different fields.
*   **Product Catalogs:** An electronics product has `bluetooth_version`, a shirt has `size`. Different schemas per document.
*   **User Profiles:** Embed addresses, preferences, and order history inside a single document for ultra-fast reads (no JOINs).
*   **Real-Time Analytics:** Store event logs as JSON documents.

**When NOT to Use:** When you have highly relational data requiring multi-document JOINs and ACID transactions across collections.

### 10.3 Column-Family (Wide-Column) Databases

**What it is:** Instead of storing data row-by-row (like SQL), it stores data **column-by-column**. This is incredibly efficient when you need to read specific columns across billions of rows.

**Examples:** **Apache Cassandra**, HBase, Google Bigtable, ScyllaDB.

```
Traditional Row Storage (SQL):        Column Storage (Cassandra):
┌────┬───────┬─────┬────────┐         Column "name":  [Alice, Bob, Carol, ...]
│ id │ name  │ age │ salary │         Column "age":   [30, 25, 28, ...]
├────┼───────┼─────┼────────┤         Column "salary":[80k, 60k, 75k, ...]
│ 1  │ Alice │ 30  │ 80k    │
│ 2  │ Bob   │ 25  │ 60k    │         ✅ To calculate AVG(salary) for
│ 3  │ Carol │ 28  │ 75k    │         1 Billion employees, the DB only
└────┴───────┴─────┴────────┘         reads the "salary" column, skipping
                                       name and age entirely. Blazing fast.
```

**When to Use:**
*   **Time-Series Data:** IoT sensor readings, stock price tickers, server metrics (billions of rows per day).
*   **Write-Heavy Workloads:** Cassandra can handle **millions of writes per second** because it uses a Log-Structured Merge (LSM) Tree instead of a B-Tree (append-only writes, no in-place updates).
*   **Analytics / Aggregations:** When you run queries like `AVG(salary)` or `SUM(revenue)` on a single column across billions of rows.

**When NOT to Use:** When you need complex JOINs or ad-hoc, flexible querying. Cassandra requires you to design your tables around your *exact* query patterns ahead of time.

### 10.4 Graph Databases

**What it is:** Data is stored as **Nodes** (entities) and **Edges** (relationships). The database is purpose-built for traversing connections between entities.

**Examples:** **Neo4j**, Amazon Neptune, ArangoDB, JanusGraph.

```
    ┌─────────┐                  ┌─────────┐
    │  Alice  │──── FRIENDS ────▶│   Bob   │
    │ (User)  │                  │ (User)  │
    └────┬────┘                  └────┬────┘
         │                            │
      LIKES                       PURCHASED
         │                            │
         ▼                            ▼
    ┌──────────┐                ┌──────────────┐
    │ Product  │                │   Product    │
    │ Headphone│                │   Laptop     │
    └──────────┘                └──────────────┘
```

**When to Use:**
*   **Social Networks:** "Show me friends of friends of Alice who also like Jazz music." SQL would require 3-4 deeply nested JOINs. A Graph DB does this in milliseconds using a simple traversal.
*   **Recommendation Engines:** "Users who bought X also bought Y." Netflix, Amazon, Spotify all rely on graph-like traversals.
*   **Fraud Detection:** "Is this credit card transaction linked to a known fraudulent account through a chain of intermediary accounts?"
*   **Knowledge Graphs:** Google's search knowledge panel ("Who is the director of Inception?") is powered by a graph database.

**When NOT to Use:** For simple CRUD operations, transactional banking, or typical e-commerce (use SQL instead).

### 10.5 The Cheat Sheet: Which NoSQL to Pick?

| Need | Pick This | Example Product |
| :--- | :--- | :--- |
| Blazing fast cache / sessions | Key-Value | Redis |
| Flexible schema / CMS / catalogs | Document DB | MongoDB |
| Billions of rows / time-series / IoT | Wide-Column | Cassandra |
| Social graphs / recommendations / fraud | Graph DB | Neo4j |

## 11. Replication vs Sharding (Conceptually)

These two are the pillars of scaling a database beyond a single machine. They solve *different problems*.

### 11.1 Replication (Copies for Availability)

**What it is:** Creating exact **copies** of your entire database on multiple servers.

```
                    ┌────────────────────┐
    Writes ────────▶│   MASTER (Primary) │
                    │   [Full Data Copy] │
                    └────────┬───────────┘
                             │ Replicates data
                    ┌────────┴───────────┐
                    │                    │
                    ▼                    ▼
            ┌──────────────┐    ┌──────────────┐
 Reads ────▶│  REPLICA #1  │    │  REPLICA #2  │◀──── Reads
            │ [Full Copy]  │    │ [Full Copy]  │
            └──────────────┘    └──────────────┘
```

**What it Solves:**
1.  **High Availability:** If the Master crashes, Replica #1 is instantly promoted to the new Master. Zero downtime.
2.  **Read Scalability:** Route all heavy `SELECT` queries to the Replicas, freeing the Master to focus exclusively on writes.

**What it Does NOT Solve:** Storage limits. Every replica holds the *entire* dataset. If your data is 10TB, every server needs 10TB.

### 11.2 Sharding (Splitting for Scalability)

**What it is:** Splitting the dataset into **chunks (shards)** and distributing them across multiple servers. Each server holds only a *portion* of the total data.

```
                    ┌─────────────────────────┐
                    │     SHARD ROUTER         │
                    │ (Determines which shard) │
                    └──────┬──────┬──────┬─────┘
                           │      │      │
                    ┌──────┘      │      └──────┐
                    ▼             ▼              ▼
            ┌─────────────┐ ┌──────────────┐ ┌──────────────┐
            │   SHARD 1   │ │   SHARD 2    │ │   SHARD 3    │
            │ Users A-H   │ │ Users I-P    │ │ Users Q-Z    │
            │ (3.3TB)     │ │ (3.3TB)      │ │ (3.3TB)      │
            └─────────────┘ └──────────────┘ └──────────────┘
```

**What it Solves:**
1.  **Storage Scalability:** 10TB of data? Split it across 3 servers, each holding 3.3TB.
2.  **Write Scalability:** Writes for users A-H only go to Shard 1, not touching Shards 2 or 3.

**What it Does NOT Solve:** Availability. If Shard 2 goes down, all users I-P are completely offline. (Solution: Combine sharding WITH replication — each shard has its own replicas).

### 11.3 Replication vs Sharding Summary

| Aspect | Replication | Sharding |
| :--- | :--- | :--- |
| **Data Copied?** | ✅ Yes — every server has the full dataset | ❌ No — each server has a unique *portion* |
| **Solves** | High Availability, Read Scalability | Storage Limits, Write Scalability |
| **Complexity** | Low-Medium | Very High (cross-shard queries, rebalancing) |
| **Use When** | You need failover / disaster recovery | Your single DB can't hold all the data or handle all the writes |

## 12. Query Planning — How the DB Engine Thinks

### 12.1 What is a Query Plan?

When you write a SQL query, you're telling the database **what** data you want, not **how** to get it. The **Query Planner** (also called the Query Optimizer) generates an **execution plan** — a step-by-step recipe for retrieving the data as efficiently as possible.

### 12.2 The 3 Phases of Query Execution

```
  Your SQL Query
       │
       ▼
┌──────────────┐
│   1. PARSER  │ → Checks syntax. Converts SQL text into an Abstract Syntax Tree (AST).
└──────┬───────┘
       ▼
┌──────────────┐
│  2. PLANNER  │ → The brain. Evaluates multiple possible plans, estimates cost for each,
│  (Optimizer) │   and picks the cheapest one. Uses table statistics (row counts, value
└──────┬───────┘   distribution, index availability) to estimate costs.
       ▼
┌──────────────┐
│ 3. EXECUTOR  │ → Executes the chosen plan step by step. Returns the results.
└──────────────┘
```

### 12.3 Join Algorithms (How the DB Joins Two Tables)

When your query does `SELECT * FROM orders JOIN users ON orders.user_id = users.id`, the DB must pick a **Join Algorithm**.

| Algorithm | How it Works | Best When |
| :--- | :--- | :--- |
| **Nested Loop Join** | For each row in Table A, scan Table B to find matches. Like two nested `for` loops: `for a in A: for b in B: if match`. O(N × M). | One table is very small (< 100 rows), or the inner table has an index on the join column. |
| **Hash Join** | Build a Hash Table from the smaller table (Table A). Then scan the larger table (Table B) and probe the hash table for matches. O(N + M). | Large tables without indexes on the join column. The smaller table must fit in memory. |
| **Merge Join (Sort-Merge)** | Sort both tables by the join column, then walk through them simultaneously (like merging two sorted arrays). O(N log N + M log M). | Both tables are already sorted (or indexed) on the join column. Extremely efficient for pre-sorted data. |

### 12.4 The Cost-Based Optimizer

Modern databases (PostgreSQL, MySQL 8+, Oracle) use a **Cost-Based Optimizer (CBO)**. Here's the key idea:

1.  The optimizer generates *many* possible plans.
2.  For each plan, it estimates a **cost** (measured in arbitrary "cost units" representing disk I/O and CPU effort).
3.  It picks the plan with the **lowest estimated cost**.

**Where does it get its estimates?** From **table statistics**. The DB maintains metadata like:
*   `n_rows`: Total number of rows in the table.
*   `n_distinct`: Number of unique values in a column.
*   `most_common_vals`: The most frequent values and their frequencies.

> **💡 Critical Tip:** If your queries are suddenly slow, it might be because the table statistics are stale (outdated). Run `ANALYZE table_name;` in PostgreSQL to force a refresh of the statistics.

## 13. Real-World Scenarios in Software Development

### Scenario 1: E-Commerce Product Page (Amazon)

**Challenge:** Millions of users browse product pages every second. Each page loads the product name, price, images, reviews, and "frequently bought together" recommendations.

**Database Architecture:**
*   **PostgreSQL (SQL):** Stores the core transactional data — `users`, `orders`, `payments`, `products`, `inventory`. All writes go here with full ACID guarantees (you can't sell an item that's out of stock).
*   **Redis (Key-Value Cache):** Caches the fully rendered product page JSON. 99% of page loads never touch PostgreSQL. Cache TTL is 60 seconds.
*   **MongoDB (Document DB):** Stores product reviews as flexible JSON documents (each review can have different fields — some have images, some have video, some have neither).
*   **Neo4j (Graph DB):** Powers the "Frequently Bought Together" and "Customers who bought this also bought..." recommendation engine by traversing purchase graphs.

**Key Database Decisions:**
*   Product `inventory_count` uses **Pessimistic Locking** (`SELECT ... FOR UPDATE`) to prevent overselling.
*   The `orders` table is **Normalized** (3NF) for write integrity but a **Denormalized** materialized view is created for the admin dashboard.
*   **Composite Index** on `orders(user_id, created_at DESC)` to power "My Orders" page sorted by date.

---

### Scenario 2: Banking Transfer System

**Challenge:** Two users transfer money simultaneously. The system must guarantee that money never disappears or duplicates.

**Database Architecture:**
*   **PostgreSQL with Serializable Isolation Level.** This is non-negotiable. Money *must* have the strongest ACID guarantees.
*   **Transactions:** Every transfer is wrapped in `BEGIN ... COMMIT` with both the debit and credit operations inside the same transaction (Atomicity).
*   **Deadlock Prevention:** All transfers lock accounts in ascending order of `account_id` to prevent circular waits.
*   **Write-Ahead Log (WAL):** Guarantees Durability — even if the server crashes mid-transfer, the committed data is recovered from the WAL on restart.

---

### Scenario 3: Real-Time IoT Dashboard (Smart Factory)

**Challenge:** 50,000 sensors in a factory each send a temperature/pressure reading every second. That's 50,000 writes/second, 4.3 billion rows/day. An engineer needs to query "What was the average temperature of Sensor #42 over the last 24 hours?"

**Database Architecture:**
*   **Apache Cassandra (Wide-Column):** Handles the massive write throughput using its LSM-Tree (append-only) write path. Data is partitioned by `sensor_id` (each sensor's data lives on a specific shard).
*   **Data Model:** The table is designed around the *exact query pattern*: `PRIMARY KEY (sensor_id, timestamp)`. This allows Cassandra to retrieve all readings for a specific sensor in chronological order without scanning the entire dataset.
*   **No JOINs, No Normalization:** All relevant data (sensor name, location, reading value) is denormalized into a single wide row.

---

### Scenario 4: Social Media Feed (Instagram/Twitter)

**Challenge:** When you open Instagram, you see a personalized feed of posts from people you follow, ranked by relevance.

**Database Architecture:**
*   **PostgreSQL:** Stores user accounts, follow relationships, and post metadata.
*   **Redis (Sorted Sets):** Pre-computes and caches each user's personalized feed. When User A posts, the system pushes the post ID into the Redis sorted sets of all of A's followers (this is called "Fan-Out on Write").
*   **Amazon S3 + CDN:** All images and videos are stored in S3 and served via a global CDN, never from the database.
*   **Neo4j:** Powers "Suggested Users" by traversing the follower graph ("Friends of friends you might know").

## 14. Interview Questions & Problems with Solutions

### ❓ Q1: "You have a `users` table with 100 million rows. The query `SELECT * FROM users WHERE email = 'x@y.com'` takes 8 seconds. How do you fix it?"

**Answer:**
1.  Run `EXPLAIN ANALYZE` on the query. You will almost certainly see a **Seq Scan** (full table scan).
2.  Create a **B-Tree Index** on the `email` column: `CREATE INDEX idx_email ON users(email);`
3.  Re-run the query. `EXPLAIN ANALYZE` should now show an **Index Scan**. The query will drop from 8 seconds to under 5 milliseconds.
4.  **Bonus:** If `email` must be unique, create a `UNIQUE INDEX` instead. This both speeds up lookups AND enforces data integrity.

---

### ❓ Q2: "Two users simultaneously try to buy the last available concert ticket. How do you prevent double-booking?"

**Answer:**
Use **Pessimistic Locking**:
```sql
BEGIN;
-- Lock the ticket row. No other transaction can read or modify it until we COMMIT.
SELECT * FROM tickets WHERE id = 42 AND status = 'available' FOR UPDATE;

-- If the row was returned, the ticket IS available. Book it.
UPDATE tickets SET status = 'sold', buyer_id = 123 WHERE id = 42;
COMMIT;
```
If User B runs the same query a nanosecond later, the `SELECT ... FOR UPDATE` will **block** and wait until User A's transaction commits. When User B's query finally runs, the `WHERE status = 'available'` condition will fail (because User A already set it to `sold`), and User B gets zero rows back. **No double-booking.**

---

### ❓ Q3: "What isolation level would you use for a financial reporting system that generates end-of-day balance reports?"

**Answer:**
**Repeatable Read** at minimum, ideally **Serializable**.

*   The report reads the balances of all accounts. While the report is running (which takes 30 minutes), other transactions are actively depositing and withdrawing money.
*   With **Read Committed** (the default in PostgreSQL), the report might read Account A's balance at the start (before a transfer) and Account B's balance at the end (after the transfer). The total would be wrong.
*   With **Repeatable Read**, the report takes a consistent **snapshot** of the entire database at the moment it starts running. All reads within the report see the exact same data, regardless of what other transactions do.

---

### ❓ Q4: "Our `orders` table has 500 million rows. We query it by `user_id` and by `created_at`. Should we create two separate indexes or one composite index?"

**Answer:**
It depends on the query pattern.
*   **If the most common query is:** `WHERE user_id = 123 AND created_at > '2024-01-01'`
    → Create **one Composite Index**: `CREATE INDEX idx ON orders(user_id, created_at);`
    This is far more efficient than two separate indexes because the B-Tree can narrow down to user 123 first, then within that subset, efficiently scan by date.
*   **If you also query by `created_at` alone** (e.g., "all orders today"):
    → Create an **additional separate index**: `CREATE INDEX idx_date ON orders(created_at);`
    The composite index `(user_id, created_at)` **cannot** be used for a query that only filters by `created_at` because `user_id` is the leading column. The **Leftmost Prefix Rule** requires the leading column to be present.

---

### ❓ Q5: "We're building a system that must handle 2 million writes per second from IoT sensors. SQL or NoSQL?"

**Answer:**
**NoSQL — specifically Apache Cassandra or ScyllaDB.**

*   **Why not SQL?** A single PostgreSQL server maxes out at approximately 10,000-50,000 writes/sec. Even with sharding, the overhead of ACID transaction logging, B-Tree index rebalancing, and write-ahead logging makes it physically impossible to sustain 2 million writes/sec.
*   **Why Cassandra?** Cassandra uses a **Log-Structured Merge Tree (LSM-Tree)** instead of B-Trees. Writes are append-only (sequentially written to an in-memory buffer called a *Memtable*, then flushed to disk). This makes writes extraordinarily fast.
*   **Trade-off:** Reads in Cassandra can be slower and more complex. You must design your data model around your read queries ahead of time (no ad-hoc JOINs).

---

### ❓ Q6: "Why shouldn't I just index every column in my table?"

**Answer:**
Three reasons:
1.  **Write Penalty:** Every `INSERT`, `UPDATE`, or `DELETE` on the table must also update *every* index. If you have 15 indexes, a single INSERT triggers 16 write operations (1 table + 15 indexes). Your write throughput drops dramatically.
2.  **Storage Bloat:** Indexes consume disk space. A 50GB table with 10 indexes might consume 200GB+ of total disk space.
3.  **The Optimizer Gets Confused:** With too many overlapping indexes, the Query Planner might choose a suboptimal plan or spend excessive time evaluating which index to use.

> **Golden Rule:** Only index columns used in `WHERE`, `JOIN ON`, `ORDER BY`, and `GROUP BY` clauses of your most frequent queries.

---

### ❓ Q7: "Explain the difference between Optimistic and Pessimistic Locking. When would you use each?"

**Answer:**

| Aspect | Optimistic Locking | Pessimistic Locking |
| :--- | :--- | :--- |
| **Mechanism** | Application adds a `version` column. On update, check `WHERE version = old_version`. If it fails, someone else modified the row first. Retry. | Database physically locks the row with `SELECT ... FOR UPDATE`. Other transactions must wait. |
| **Blocking?** | ❌ No blocking. Conflicts are detected at commit time. | ✅ Yes. Other transactions are blocked and must wait. |
| **Best For** | Systems with **lots of reads and rare write conflicts** (editing a user profile, CMS content). | Systems where **write conflicts are very likely** (booking the last ticket, decrementing the last inventory item). |
| **Failure Mode** | If a conflict occurs, the application must **retry** the operation. | If a deadlock occurs, the DB kills one transaction and the application must **retry**. |

---

### ❓ Q8: "You have a `products` table and a `reviews` table. The product page needs to show `product_name`, `avg_rating`, and the latest 5 reviews. How do you optimize this?"

**Answer:**
1.  **Denormalize `avg_rating`:** Instead of computing `AVG(rating)` from the `reviews` table every page load (which scans thousands of rows), pre-compute and store it directly in the `products` table as a `avg_rating DECIMAL(3,2)` column. Update it asynchronously whenever a new review is submitted.
2.  **Index for latest reviews:** Create: `CREATE INDEX idx_reviews_product_date ON reviews(product_id, created_at DESC);`
3.  **Cache the full result:** Store the final JSON response (product + 5 reviews) in **Redis** with a 60-second TTL. 99% of page loads never hit the database.

This transforms a slow, multi-JOIN, aggregation-heavy query into a single Redis `GET` that returns in 0.3ms.
