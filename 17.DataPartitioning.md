## 1. What is Data Partitioning?

**Data Partitioning (Sharding)** is the process of splitting a large database into smaller, more manageable pieces called **partitions** or **shards**, distributed across multiple servers.

> **рҹ’Ў Analogy:** A library with 1 million books on a single shelf is impossible to search. Split it into 26 sections (A-Z by author's last name). Each section is on a different shelf. Now 26 librarians can serve 26 readers simultaneously.

## 2. Horizontal vs Vertical Partitioning

### Horizontal Partitioning (Sharding)

Split **rows** across servers. Each shard has ALL columns but only SOME rows.

```
User Table (100M rows):

Shard 1: Users where user_id % 4 == 0  (25M rows, all columns)
Shard 2: Users where user_id % 4 == 1  (25M rows, all columns)
Shard 3: Users where user_id % 4 == 2  (25M rows, all columns)
Shard 4: Users where user_id % 4 == 3  (25M rows, all columns)
```

### Vertical Partitioning

Split **columns** across servers. Each partition has ALL rows but only SOME columns.

```
User Table:

Server A (frequently accessed): user_id, name, email
Server B (rarely accessed):     user_id, bio, profile_picture, preferences
Server C (large blobs):         user_id, resume_pdf, avatar_image

Each server stores data for ALL users, but only specific columns.
```

### When to Use Which

| Type | Use When |
| :--- | :--- |
| **Horizontal** | Table has too many ROWS (100M+). Need to scale reads AND writes. |
| **Vertical** | Table has too many COLUMNS. Some columns accessed rarely or are very large. |
| **Both** | Large tables with both many rows and many columns (e.g., user profiles at Facebook scale). |

## 3. Partitioning Strategies

### Strategy 1: Hash-Based Partitioning вӯҗ

```
shard_id = hash(partition_key) % number_of_shards

Example: partition_key = user_id
  hash(user_123) % 4 = 2 вҶ’ Shard 2
  hash(user_456) % 4 = 0 вҶ’ Shard 0
  hash(user_789) % 4 = 3 вҶ’ Shard 3

вң… Even distribution (hash function spreads data uniformly)
вқҢ Adding/removing shards requires rehashing EVERYTHING (see Consistent Hashing)
```

### Strategy 2: Range-Based Partitioning

```
Partition by a range of the partition key:

  Shard 1: user_id 1 - 25,000,000
  Shard 2: user_id 25,000,001 - 50,000,000
  Shard 3: user_id 50,000,001 - 75,000,000
  Shard 4: user_id 75,000,001 - 100,000,000

вң… Range queries are efficient (all users 1-25M are on the same shard)
вқҢ Hot spots: if new users are always assigned incrementing IDs,
   Shard 4 gets ALL new writes while Shards 1-3 are idle.
```

### Strategy 3: Geographic Partitioning

```
Partition by geography:

  Shard US:     Users in United States
  Shard EU:     Users in Europe
  Shard ASIA:   Users in Asia
  Shard LATAM:  Users in Latin America

вң… Data locality (US users' data is on a US server вҶ’ low latency)
вң… Legal compliance (EU data stays in EU вҶ’ GDPR compliant)
вқҢ Uneven distribution (US might have 60% of users, LATAM only 5%)
вқҢ Cross-region queries are slow (finding a friend in another region)
```

### Strategy 4: Directory-Based Partitioning

```
A lookup service maintains a mapping: key вҶ’ shard

  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
  в”Ӯ   DIRECTORY SERVICE  в”Ӯ
  в”Ӯ                      в”Ӯ
  в”Ӯ  user_123 вҶ’ Shard 2  в”Ӯ
  в”Ӯ  user_456 вҶ’ Shard 1  в”Ӯ
  в”Ӯ  user_789 вҶ’ Shard 3  в”Ӯ
  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

вң… Maximum flexibility (move any key to any shard)
вң… Easy to rebalance (just update the directory)
вқҢ Directory is a SPOF and bottleneck (every request hits it first)
вқҢ Extra latency (lookup before every query)
```

## 4. Consistent Hashing (The Deep Dive)

### 4.1 The Problem with Naive Hashing

```
4 shards: shard = hash(key) % 4

hash("user_1") % 4 = 2 вҶ’ Shard 2
hash("user_2") % 4 = 0 вҶ’ Shard 0
hash("user_3") % 4 = 1 вҶ’ Shard 1

Now add a 5th shard: shard = hash(key) % 5

hash("user_1") % 5 = 3 вҶ’ Shard 3  вҶҗ MOVED! Was on Shard 2!
hash("user_2") % 5 = 2 вҶ’ Shard 2  вҶҗ MOVED! Was on Shard 0!
hash("user_3") % 5 = 1 вҶ’ Shard 1  вҶҗ Same (lucky)

~75% of keys need to be MOVED to different shards. Massive data migration!
```

### 4.2 The Consistent Hashing Solution

```
Imagine a RING (0 to 2^32):

                    0
                в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
           S3 в—Ҹв”Җв”Ө       в”ңв”Җв—Ҹ S1
               в”Ӯ       в”Ӯ
               в”Ӯ       в”Ӯ
           S2 в—Ҹв”Җв”Ө       в”ңв”Җв—Ҹ S4
                в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

  Servers (S1-S4) are placed on the ring at hash(server_ip) positions.
  Keys are placed on the ring at hash(key) positions.
  
  A key belongs to the FIRST server you encounter going CLOCKWISE.
  
  key_A at position 50   вҶ’ clockwise вҶ’ hits S1 at position 100   вҶ’ Shard S1
  key_B at position 200  вҶ’ clockwise вҶ’ hits S2 at position 300   вҶ’ Shard S2

ADDING S5 at position 250:
  Only keys between S4 (position 150) and S5 (250) need to move to S5.
  All other keys stay where they are!
  
  Only ~20% of keys move (1/N), not 75%! вң…
```

### 4.3 Virtual Nodes

```
Problem: With 4 servers, some might get 60% of keys and others 10% (uneven).

Solution: Each real server has 100-200 "virtual nodes" spread across the ring.

  S1 вҶ’ V1_1, V1_2, V1_3, ..., V1_150  (150 virtual nodes for S1)
  S2 вҶ’ V2_1, V2_2, V2_3, ..., V2_150
  
  150 virtual nodes per server вҶ’ nearly perfect even distribution.

Used by: Cassandra, DynamoDB, Memcached
```

## 5. Problems with Sharding

| Problem | Description | Mitigation |
| :--- | :--- | :--- |
| **Cross-shard JOINs** | User on Shard 1, their orders on Shard 3 вҶ’ can't JOIN | Denormalize (store user info WITH orders). Or do application-level joins. |
| **Cross-shard transactions** | ACID transactions don't work across shards | Use Saga pattern or 2-phase commit (expensive). |
| **Hotspot / Uneven distribution** | Celebrity user gets 10x more traffic вҶ’ that shard is overloaded | Better partition key choice. Or split the hot shard further. |
| **Resharding** | Adding shards requires data migration | Consistent hashing minimizes migration. Plan capacity ahead. |
| **Operational complexity** | Backups, schema migrations, monitoring Г— N shards | Automation. Use managed databases (DynamoDB, CockroachDB) that handle sharding. |
| **ID generation** | Auto-increment doesn't work across shards | Use Snowflake IDs or UUIDs. |

> **рҹ’Ў Golden Rule:** **Don't shard until you have to.** Read replicas, caching, and query optimization solve most scaling issues. Shard only when you've exhausted everything else.

## 6. Real-World Examples

| Company | Partition Strategy | Partition Key | Why |
| :--- | :--- | :--- | :--- |
| **Instagram** | Hash-based | `user_id` | All of a user's posts, stories, and likes on one shard |
| **Slack** | Hash-based | `workspace_id` | All messages for a workspace on one shard (no cross-shard queries for chat) |
| **Uber** | Geographic | `city_id` | Rides in Delhi and NYC never interact. City-level sharding is natural. |
| **Stripe** | Hash-based | `merchant_id` | All of a merchant's payments on the same shard for fast lookups |
| **Cassandra** | Consistent Hashing | Configurable | Data automatically distributed across nodes. Virtual nodes for balance. |

## 7. Interview Questions

### вқ“ Q1: "How would you shard a users table?"

**Answer:** "Partition key = `user_id`. Hash-based: `shard = hash(user_id) % N`. All of a user's data (profile, settings, preferences) lives on one shard вҶ’ no cross-shard queries for user-specific operations. Use consistent hashing to make adding shards easy."

### вқ“ Q2: "An e-commerce orders table is too large. How do you shard it?"

**Answer:** "Depends on the access pattern:
- If queries are 'find orders FOR a user' вҶ’ shard by `user_id`. All of a user's orders on one shard.
- If queries are 'find orders BY date range' вҶ’ range-based partitioning by `order_date`. Each shard holds one month.
- Hybrid: Shard by `user_id`, but add a global index for date-range queries."

### вқ“ Q3: "What's the difference between partitioning and replication?"

**Answer:** "**Partitioning** splits data вҖ” each shard has DIFFERENT data. Increases capacity. **Replication** copies data вҖ” each replica has the SAME data. Increases availability and read throughput. Use both together: shard for capacity, replicate each shard for durability."
