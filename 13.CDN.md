## 1. What is a CDN?

A **Content Delivery Network** is a geographically distributed network of servers (called **edge servers** or **PoPs — Points of Presence**) that cache and serve content from locations **closer to the user**, reducing latency dramatically.

> **💡 Intuition — The Pizza Chain:**
> Without CDN: Every pizza order goes to the central kitchen in Delhi. A customer in Chennai waits 4 hours for delivery.
>
> With CDN: The chain opens local branches in every city. The Chennai branch keeps ready-made popular pizzas. Customer gets their pizza in 20 minutes. The central kitchen only handles rare specialty orders.

```
WITHOUT CDN:
  User (Mumbai) ──── 200ms ──── Origin Server (Virginia, US)
  
WITH CDN:
  User (Mumbai) ──── 10ms ──── CDN Edge (Mumbai PoP) ──(cache miss)── Origin Server
                                      │
                              Cache HIT? → Serve instantly (10ms total)
                              Cache MISS? → Fetch from origin, cache it, serve (200ms first time)
```

## 2. How CDN Works — Step by Step

```
Step 1: User in Mumbai requests: GET /images/product-123.jpg

Step 2: DNS resolves cdn.example.com → nearest CDN PoP (Mumbai)
        (CDN providers use Anycast DNS to route to closest server)

Step 3: CDN Edge Server (Mumbai) checks its cache:
        ┌──────────────────────┐
        │ Cache HIT ✅          │ → Return cached image instantly (10ms)
        │                       │
        │ Cache MISS ❌          │ → Fetch from Origin Server (Virginia)
        │   → Store in cache    │    (200ms first time only)
        │   → Set TTL (1 hour)  │
        │   → Return to user    │
        └──────────────────────┘

Step 4: Next user in Mumbai requests same image → Cache HIT → 10ms ✅
        All subsequent requests from Mumbai region served from cache.
```

## 3. Pull CDN vs Push CDN

| Feature | Pull CDN | Push CDN |
| :--- | :--- | :--- |
| **How it works** | Edge pulls content from origin on first request (lazy loading) | You explicitly push content to all edge servers before any request |
| **First request** | Cache MISS → slow (fetches from origin) | Always cached → always fast |
| **Storage** | Only caches what's requested (efficient) | All content on all edges (potentially wasteful) |
| **Best for** | Dynamic, large catalogs, unpredictable traffic | Static assets, known-popular content, video streaming |
| **Example** | Cloudflare, AWS CloudFront (default) | Netflix pre-positions video files to edge servers worldwide |

```
PULL CDN:
  1st request for /images/rare-product.jpg → MISS → fetch from origin → cache
  2nd request → HIT → serve from edge ✅
  
  ✅ Pro: Only popular content is cached. Storage efficient.
  ❌ Con: First user always gets a slow response (cold cache).

PUSH CDN:
  You upload /videos/trailer.mp4 → CDN pushes to 200 edge servers worldwide
  1st request → HIT immediately ✅ (already cached everywhere)
  
  ✅ Pro: Zero cold-cache misses. Always fast.
  ❌ Con: You pay for storage at 200 locations even if nobody in Africa watches it.
```

## 4. CDN Caching Strategies

### 4.1 What to Cache vs What NOT to Cache

| Cache ✅ | Don't Cache ❌ |
| :--- | :--- |
| Static assets (JS, CSS, images, fonts) | User-specific data (profile, cart, dashboard) |
| Product images, thumbnails | API responses with real-time data (stock prices) |
| Video files, audio files | POST/PUT/DELETE responses |
| Public API responses (product catalog) | Authenticated content (varies per user) |
| HTML pages (if not personalized) | Content that changes every second |

### 4.2 Cache-Control Headers

```http
# Cache for 1 year (static assets with versioned filenames)
Cache-Control: public, max-age=31536000, immutable

# Cache for 5 minutes (product catalog page)
Cache-Control: public, max-age=300

# Don't cache at all (user dashboard)
Cache-Control: no-store, no-cache, must-revalidate

# Cache, but revalidate with origin before serving (stale-while-revalidate)
Cache-Control: public, max-age=60, stale-while-revalidate=30
```

| Directive | Meaning |
| :--- | :--- |
| `public` | CDN and browser can both cache it |
| `private` | Only the browser can cache it (not CDN) — user-specific content |
| `max-age=300` | Cache is valid for 300 seconds |
| `no-store` | Don't cache at all. Ever. |
| `immutable` | Content will NEVER change (versioned assets like `app.abc123.js`) |
| `stale-while-revalidate` | Serve stale cache while fetching fresh copy in background |

## 5. Cache Invalidation at the Edge

The hardest problem in CDN: **"How do you update cached content across 200 edge servers worldwide?"**

| Strategy | How it Works | Latency | Cost |
| :--- | :--- | :--- | :--- |
| **TTL Expiry** | Set `max-age=300`. After 5 min, edge fetches fresh content. | 0-5 min stale | Free |
| **Purge API** | Call CDN's purge API: `POST /purge { url: "/product/123" }` | Seconds (propagates to all edges) | Per-purge cost |
| **Versioned URLs** | Change `/app.js` to `/app.v2.js`. Old cache is irrelevant. | Instant | Free |
| **Stale-While-Revalidate** | Serve stale, fetch fresh in background. Next request gets updated. | 1 request stale | Free |
| **Cache Tags** | Tag cached items: `tag: "product-123"`. Purge by tag. | Seconds | Per-purge cost |

> **💡 Best Practice:** Use **versioned URLs** for static assets (JS, CSS, images) → never need to invalidate. Use **short TTL + stale-while-revalidate** for dynamic content (product pages). Use **purge API** only for emergency fixes.

## 6. CDN Architecture Patterns

### Pattern 1: Static Assets Only (Most Common)
```
App Server serves HTML + API
CDN serves: /static/*, /images/*, /fonts/*

Setup: S3 (origin) → CloudFront (CDN) → User
       App Server → User (for API/HTML)
```

### Pattern 2: Full-Site CDN (Modern)
```
ALL requests go through CDN. CDN decides what to cache.

User → CDN → (Cache HIT → serve) or (Cache MISS → Origin Server)

Benefits: DDoS protection, SSL termination, WAF at the edge.
Used by: Almost every large website (via Cloudflare, Fastly).
```

### Pattern 3: Multi-Origin CDN
```
CDN routes to different origins based on path:

  /api/*     → API Server (no caching)
  /images/*  → S3 Bucket (cache forever)
  /videos/*  → Video Streaming Server (chunked delivery)
  /*         → Web App Server (cache 60 seconds)
```

## 8. Interview Questions

### ❓ Q1: "How would you serve images for a global e-commerce site?"

**Answer:** "Upload images to S3 (origin). Put CloudFront CDN in front. Set `Cache-Control: public, max-age=86400` (24 hours). Use versioned filenames: `product-123-v2.jpg`. When image updates, upload with new version → old cache becomes irrelevant. CDN edge servers worldwide serve images in < 20ms to any user globally."

---

### ❓ Q2: "You just updated a product price, but the CDN is still showing the old price. How do you fix it?"

**Answer:** "Three options:
1. **Immediate:** Call the CDN purge API to clear the cached page for that product URL.
2. **Preventive:** Use short TTL (`max-age=60`) for product pages. Max 60-second staleness.
3. **Hybrid:** Use `stale-while-revalidate=30`. Serve stale while fetching fresh in background.
For critical price changes, use the purge API for instant effect."

---

### ❓ Q3: "CDN vs Redis Cache — what's the difference?"

**Answer:**
| | CDN | Redis |
| :--- | :--- | :--- |
| **Where** | Edge servers worldwide (close to user) | Your data center (close to your app server) |
| **What** | HTTP responses, files, images, HTML pages | Database query results, sessions, computed data |
| **Reduces** | Network latency (user → server) | Database load (app → DB) |
| **Protocol** | HTTP/HTTPS | TCP (Redis protocol) |
| **Use together:** | CDN caches the HTTP response. Redis caches the DB query that generated it. Both reduce load at different layers."
