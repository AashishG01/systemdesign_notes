# Part 1: Core Concepts

---

## 1. What is Rate Limiting?

### 1.1 The Definition

**Rate Limiting** is a technique to control the number of requests a client can make to a server within a defined time window. Requests that exceed the limit are rejected with an error response.

> **💡 Intuition — The Nightclub Bouncer:**
> A nightclub has a capacity of 200 people. A bouncer stands at the door counting everyone who enters. Person #201 is told: "We're at capacity. Come back in 30 minutes."
>
> The bouncer doesn't care if you're a VIP or a regular. If the club is full, you wait. That's a basic rate limiter. A *smarter* bouncer (tiered rate limiting) lets VIPs bypass the line — that's API key-based rate limiting.

### 1.2 Rate Limiting in One Diagram

```
                        ┌────────────────────────┐
  Request #1 ──────────▶│                        │──────────▶ Backend Server ✅
  Request #2 ──────────▶│     RATE LIMITER       │──────────▶ Backend Server ✅
  Request #3 ──────────▶│                        │──────────▶ Backend Server ✅
  ...                   │  Rule: Max 100 req/min │
  Request #100 ────────▶│                        │──────────▶ Backend Server ✅
  Request #101 ────────▶│                        │──▶ ❌ 429 Too Many Requests
  Request #102 ────────▶│                        │──▶ ❌ 429 Too Many Requests
                        └────────────────────────┘
```

## 2. Why Rate Limiting is Critical

### 2.1 The 5 Problems Rate Limiting Solves

| Problem | Without Rate Limiting | With Rate Limiting |
| :--- | :--- | :--- |
| **DDoS Attack** | Attacker sends 10M requests/sec → servers crash → entire site offline | Attacker's IP is limited to 100 req/sec → servers stay healthy |
| **Abusive Bots** | Scraper bot downloads your entire product catalog in 5 minutes | Bot is throttled to 10 req/min → can't scrape faster than a human |
| **Buggy Client** | Mobile app has a bug — enters infinite loop, sends 1000s of duplicate requests | Client rate-limited after 100 req/min → servers are protected |
| **Fair Usage** | One power user consumes 90% of server resources, starving everyone else | Each user limited to 1000 req/hour → resources shared fairly |
| **Cost Control** | Third-party API charges per request. Buggy code makes 10M accidental calls → $50K bill | Outbound rate limiter caps at 1000 req/min → costs controlled |

## 3. Where to Apply Rate Limiting

```
┌───────────────────────────────────────────────────────────────────────┐
│                    RATE LIMITING LAYERS                               │
│                                                                       │
│  Layer 1: CDN / Edge (Cloudflare, AWS Shield)                        │
│  ├── Blocks volumetric DDoS attacks before they reach your servers   │
│  ├── IP-based rate limiting at the network edge                      │
│                                                                       │
│  Layer 2: API Gateway (Kong, NGINX, AWS API Gateway)                 │
│  ├── Per-user, per-API-key rate limiting                             │
│  ├── Per-endpoint rate limiting (different limits for /search vs /buy)│
│                                                                       │
│  Layer 3: Application Code (Express middleware, Spring filter)       │
│  ├── Business-logic-specific limits                                  │
│  ├── "Max 5 password attempts per hour" / "Max 3 posts per minute"   │
│                                                                       │
│  Layer 4: Database (Connection limits, query throttling)             │
│  ├── Max connections per application                                 │
│  ├── Query timeout limits                                            │
└───────────────────────────────────────────────────────────────────────┘
```

> **💡 Best Practice:** Apply rate limiting at **multiple layers**. The CDN blocks massive DDoS floods. The API Gateway enforces per-user API quotas. The application enforces business rules (3 login attempts).

## 4. Fixed Window Counter

### 4.1 How it Works

Divide time into **fixed windows** (e.g., 1-minute intervals). Count requests per window. Reset the counter at the start of each new window.

```
Window: 1 minute
Limit: 5 requests per window

Timeline:
  12:00:00 ──────────────────── 12:01:00 ──────────────────── 12:02:00
  │                             │                             │
  │ Req 1 ✅ (count = 1)       │ Counter RESET (count = 0)   │
  │ Req 2 ✅ (count = 2)       │ Req 1 ✅ (count = 1)       │
  │ Req 3 ✅ (count = 3)       │ Req 2 ✅ (count = 2)       │
  │ Req 4 ✅ (count = 4)       │                             │
  │ Req 5 ✅ (count = 5)       │                             │
  │ Req 6 ❌ (REJECTED!)       │                             │
```

### 4.2 Implementation

```javascript
// Fixed Window Rate Limiter using Redis
async function fixedWindowRateLimit(userId, limit = 100, windowSec = 60) {
  const windowKey = Math.floor(Date.now() / (windowSec * 1000));
  const key = `rate_limit:${userId}:${windowKey}`;
  
  const count = await redis.incr(key);
  
  if (count === 1) {
    await redis.expire(key, windowSec);  // Auto-delete after window ends
  }
  
  return count <= limit;  // true = allowed, false = rejected
}
```

### 4.3 The Boundary Problem ⚠️

```
Limit: 5 requests per minute

  12:00:00 ──────────── 12:00:59 │ 12:01:00 ──────────── 12:01:59
                                 │
        ... Req 1-5 at 12:00:55  │  Req 6-10 at 12:01:01 ...
            (5 requests ✅)      │  (Counter reset! 5 more ✅)
                                 │
  Result: 10 requests in 6 seconds! We allowed DOUBLE the limit.
```

**Problem:** A burst of 5 requests at the END of window 1 + a burst of 5 requests at the START of window 2 = 10 requests in just 6 seconds, even though the limit is 5/minute.

## 5. Sliding Window Log

### 5.1 How it Works

Instead of fixed time windows, store the **exact timestamp** of every request. To check the limit, count all timestamps within the last N seconds.

```
Limit: 5 requests per 60 seconds

  Request at 12:00:10 → Log: [12:00:10]
  Request at 12:00:20 → Log: [12:00:10, 12:00:20]
  Request at 12:00:30 → Log: [12:00:10, 12:00:20, 12:00:30]
  Request at 12:00:40 → Log: [12:00:10, 12:00:20, 12:00:30, 12:00:40]
  Request at 12:00:50 → Log: [12:00:10, 12:00:20, 12:00:30, 12:00:40, 12:00:50]
  Request at 12:00:55 → Count timestamps in [12:00:00 - 12:00:55] = 5.
                         Already at limit! ❌ REJECTED

  Request at 12:01:15 → Remove timestamps before 12:00:15.
                         Log: [12:00:20, 12:00:30, 12:00:40, 12:00:50]
                         Count = 4. Under limit! ✅ ALLOWED
```

### 5.2 Pros and Cons

| ✅ Pros | ❌ Cons |
| :--- | :--- |
| **Perfectly accurate.** No boundary burst problem. | **High memory usage.** Stores every timestamp. At 10K users × 100 req/min × 60 sec = 60 million timestamps in memory. |
| Smoothly rolling window. | Cleanup logic required (remove old timestamps). |

---

## 6. Sliding Window Counter

### 6.1 How it Works

A **hybrid** of Fixed Window and Sliding Window Log. It uses the counts from the **current window** and the **previous window**, then calculates a weighted average based on how far into the current window you are.

```
Limit: 100 requests per minute

Previous window (12:00 - 12:01): 84 requests
Current window  (12:01 - 12:02): 36 requests so far

Current time: 12:01:15 → 25% into the current window (15/60)

Weighted count = (previous_count × overlap%) + current_count
               = (84 × 75%) + 36
               = 63 + 36
               = 99

99 < 100 → ✅ ALLOWED

Next request:
Weighted count = 63 + 37 = 100 → ❌ REJECTED (at limit)
```

### 6.2 Why This Works

It approximates a true sliding window using only **2 counters** (previous window count + current window count) instead of storing every timestamp. Very memory-efficient with nearly perfect accuracy.

## 7. Token Bucket ⭐

### 7.1 How it Works

Imagine a bucket that holds a maximum of N **tokens**. Tokens are added at a steady rate (e.g., 10 tokens/second). Each request consumes 1 token. If the bucket is empty, the request is rejected.

```
Bucket Capacity: 10 tokens
Refill Rate: 2 tokens per second

Time 0.0s: [●●●●●●●●●●] 10 tokens (full)
  Request → ✅ [●●●●●●●●●○] 9 tokens
  Request → ✅ [●●●●●●●●○○] 8 tokens
  Request → ✅ [●●●●●●●○○○] 7 tokens
  Request → ✅ [●●●●●●○○○○] 6 tokens
  Request → ✅ [●●●●●○○○○○] 5 tokens

Time 1.0s: 2 tokens refilled → [●●●●●●●○○○] 7 tokens
  Request → ✅ [●●●●●●○○○○] 6 tokens

Time 2.0s: 2 tokens refilled → [●●●●●●●●○○] 8 tokens
  (No requests)

Time 3.0s: 2 tokens refilled → [●●●●●●●●●●] 10 tokens (capped at max)
```

### 7.2 The Key Property: Burst Tolerance

The Token Bucket **allows short bursts** up to the bucket capacity. If the bucket has 10 tokens, a client can make 10 rapid requests in 1 second. After that, they're rate-limited to the refill rate (2/sec).

**This is perfect for real-world traffic patterns** where users naturally come in bursts (load a page → rapid API calls → idle → repeat).

### 7.3 Implementation

```javascript
class TokenBucket {
  constructor(capacity, refillRate) {
    this.capacity = capacity;           // Max tokens in bucket
    this.tokens = capacity;             // Start full
    this.refillRate = refillRate;        // Tokens added per second
    this.lastRefillTime = Date.now();
  }

  allowRequest() {
    this.refill();
    
    if (this.tokens >= 1) {
      this.tokens -= 1;
      return true;    // ✅ Allowed
    }
    return false;      // ❌ Rejected
  }

  refill() {
    const now = Date.now();
    const elapsed = (now - this.lastRefillTime) / 1000;  // seconds
    const tokensToAdd = elapsed * this.refillRate;
    
    this.tokens = Math.min(this.capacity, this.tokens + tokensToAdd);
    this.lastRefillTime = now;
  }
}

// Usage:
const limiter = new TokenBucket(10, 2);  // 10 max, refill 2/sec
if (!limiter.allowRequest()) {
  return res.status(429).json({ error: 'Rate limit exceeded' });
}
```

## 8. Leaky Bucket

### 8.1 How it Works

Requests enter a **fixed-capacity queue** (bucket). The queue is drained (processed) at a **constant rate**, like water leaking from a bucket with a small hole. If the queue is full, new requests overflow and are rejected.

```
Bucket Capacity: 5 requests
Drain Rate: 1 request per second (constant)

Requests pour in:
  ┌─────────┐
  │ [Req 5] │ ← Newest (just arrived)
  │ [Req 4] │
  │ [Req 3] │
  │ [Req 2] │
  │ [Req 1] │ ← Oldest (next to be processed)
  └────┬────┘
       │ Leaks at constant rate: 1 req/sec
       ▼
  Processing: Req 1 → Req 2 → Req 3 → ... (evenly spaced)

  Req 6 arrives while bucket is full → ❌ OVERFLOW (rejected)
```

### 8.2 Token Bucket vs Leaky Bucket

| Feature | Token Bucket | Leaky Bucket |
| :--- | :--- | :--- |
| **Bursts** | ✅ Allows bursts (up to bucket capacity) | ❌ No bursts. Output is always at a constant rate |
| **Processing Rate** | Variable (can drain tokens as fast as they exist) | Fixed (always processes at the leak rate) |
| **Behavior** | "Here are 10 tokens. Use them whenever." | "I'll process exactly 1 request per second, no faster." |
| **Best For** | User-facing APIs (burst-tolerant) | Backend processing where a constant load on downstream is critical |
| **Used By** | AWS API Gateway, Stripe, most public APIs | Network traffic shaping, NGINX `limit_req` (leaky bucket internally) |

---

## 9. Algorithm Comparison — When to Use What

| Algorithm | Accuracy | Memory | Burst Handling | Complexity | Best For |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Fixed Window** | ⚠️ Boundary burst issue | ✅ Very low (1 counter) | ❌ Double burst at boundary | ✅ Simplest | Internal APIs, non-critical limits |
| **Sliding Window Log** | ✅ Perfect | ❌ Very high (all timestamps) | ✅ No boundary issue | Medium | Small-scale, precision-critical |
| **Sliding Window Counter** | ✅ Near-perfect | ✅ Low (2 counters) | ✅ No boundary issue | Medium | ⭐ Production APIs (best balance) |
| **Token Bucket** | ✅ Good | ✅ Low (token count + timestamp) | ✅ Allows controlled bursts | Medium | ⭐ User-facing public APIs |
| **Leaky Bucket** | ✅ Good | ✅ Low (queue + pointer) | ❌ Smooths all bursts | Medium | Traffic shaping, constant-rate processing |

> **💡 Interview Answer:** "For most production APIs, I'd use **Token Bucket** (allows natural user bursts while enforcing an average rate) or **Sliding Window Counter** (accurate with minimal memory). For traffic shaping where I need perfectly smooth output, I'd use **Leaky Bucket**."

## 10. Rate Limiting Granularity

| Level | Identifier | Example Rule | Use Case |
| :--- | :--- | :--- | :--- |
| **Per IP** | Client's IP address | 100 req/min per IP | DDoS protection, anonymous rate limiting |
| **Per User** | Authenticated User ID (from JWT) | 1000 req/hour per user | Fair usage among authenticated users |
| **Per API Key** | API Key (from header) | Free: 100/min, Pro: 10K/min | Tiered pricing for API products |
| **Per Endpoint** | URL path | `GET /search`: 30/min. `POST /upload`: 5/min | Protect expensive endpoints |
| **Per Region** | GeoIP location | 500K req/min per country | Prevent regional abuse, comply with laws |
| **Global** | System-wide | 1M req/min total across all users | Protect backend infrastructure from total overload |

### Combined Rule Example

```
Rate Limit Rules (evaluated in order):

1. Global:          Max 500,000 req/min total
2. Per IP:          Max 100 req/min per IP address
3. Per API Key:
   - Free tier:     Max 60 req/min
   - Pro tier:      Max 5,000 req/min
   - Enterprise:    Max 100,000 req/min
4. Per Endpoint:
   - POST /api/ai/generate:  Max 10 req/hour (expensive GPU operation)
   - GET  /api/search:       Max 60 req/min
   - GET  /api/health:       No limit
```

## 11. Distributed Rate Limiting

### 11.1 The Problem

With a single server, rate limiting is easy — just a counter in local memory. But with **10 horizontally scaled API servers**, each with their own local counter, the user's requests are distributed across servers:

```
Rate Limit: 100 req/min

  Server 1: User A has made 30 requests (local counter)
  Server 2: User A has made 25 requests (local counter)
  Server 3: User A has made 25 requests (local counter)
  Server 4: User A has made 25 requests (local counter)
  
  Total: 105 requests! But each server thinks the user is under the limit.
  ❌ Rate limit is INEFFECTIVE.
```

### 11.2 The Solution: Centralized Counter (Redis)

All API servers share a single **Redis instance** for rate limit counters.

```
  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ Server 1 │  │ Server 2 │  │ Server 3 │  │ Server 4 │
  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
       │              │              │              │
       └──────────────┼──────────────┼──────────────┘
                      │
              ┌───────▼───────┐
              │    REDIS      │
              │               │
              │ user:A = 105  │ ← Single source of truth
              │               │   All 4 servers read/write here
              └───────────────┘

  Server 1 receives a request from User A:
    → INCR "rate_limit:user:A:window:42" in Redis
    → Redis returns: 106
    → 106 > 100 → ❌ REJECTED (429)
```

### 11.3 Race Condition: The INCR Trick

```javascript
// ❌ WRONG — Race condition
const count = await redis.get(key);        // Read: 99
// Another server also reads 99 at this exact moment
if (count >= 100) return reject();
await redis.set(key, count + 1);           // Both servers set to 100
// Result: Both requests pass! Limit exceeded.

// ✅ CORRECT — Atomic operation
const count = await redis.incr(key);       // Atomically: read + increment
if (count === 1) {
  await redis.expire(key, 60);             // Set TTL on first request
}
if (count > 100) return reject();
// Redis INCR is atomic — no race condition possible.
```

### 11.4 What if Redis Goes Down?

| Strategy | Behavior | Risk |
| :--- | :--- | :--- |
| **Fail Open** | If Redis is unreachable, allow ALL requests through (no rate limiting) | Risk of abuse during Redis downtime, but system stays available |
| **Fail Closed** | If Redis is unreachable, reject ALL requests | Safe from abuse, but legitimate users can't access the system |
| **Local Fallback** | Fall back to approximate local in-memory rate limiting per server | Imperfect limits, but reasonable protection |

> **💡 Industry Standard:** Most systems **fail open** — availability trumps protection. A few minutes without rate limiting is better than rejecting all users. Some high-security systems (banking) fail closed.

## 12. HTTP Response Design

### 12.1 The Standard Response

```http
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
Retry-After: 30

X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1709312460

{
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "You have exceeded the rate limit of 100 requests per minute.",
    "retry_after_seconds": 30
  }
}
```

### 12.2 Rate Limit Headers Explained

| Header | Meaning | Example |
| :--- | :--- | :--- |
| `X-RateLimit-Limit` | Maximum requests allowed in the window | `100` |
| `X-RateLimit-Remaining` | Requests remaining in the current window | `73` (you've used 27) |
| `X-RateLimit-Reset` | Unix timestamp when the window resets | `1709312460` |
| `Retry-After` | Seconds the client should wait before retrying | `30` |

> **💡 Why include these headers on EVERY response?** Good API design returns rate limit headers on **successful responses too** (not just 429s). This lets clients monitor their consumption and self-throttle before hitting the limit.

## 13. Rate Limiting with Redis — Implementation

### 13.1 Sliding Window Counter in Redis

```javascript
const WINDOW_SIZE = 60;  // 60 seconds
const MAX_REQUESTS = 100;

async function slidingWindowRateLimit(userId) {
  const now = Date.now();
  const windowStart = now - (WINDOW_SIZE * 1000);
  const key = `rate_limit:${userId}`;
  
  // Use Redis Sorted Set: score = timestamp, member = unique request ID
  const pipeline = redis.pipeline();
  
  // 1. Remove all entries older than the window
  pipeline.zremrangebyscore(key, 0, windowStart);
  
  // 2. Count remaining entries in the window
  pipeline.zcard(key);
  
  // 3. Add current request (if allowed)
  pipeline.zadd(key, now, `${now}-${Math.random()}`);
  
  // 4. Set TTL so the key auto-deletes (memory cleanup)
  pipeline.expire(key, WINDOW_SIZE);
  
  const results = await pipeline.exec();
  const requestCount = results[1][1];  // Result of ZCARD
  
  if (requestCount >= MAX_REQUESTS) {
    // Undo the ZADD (we already added it optimistically)
    await redis.zrem(key, results[2][1]);
    return { allowed: false, remaining: 0 };
  }
  
  return { allowed: true, remaining: MAX_REQUESTS - requestCount - 1 };
}
```

### 13.2 Token Bucket in Redis (Lua Script for Atomicity)

```lua
-- Token Bucket Rate Limiter (Redis Lua Script)
-- Atomic: entire script runs as a single Redis command

local key = KEYS[1]
local capacity = tonumber(ARGV[1])     -- Max tokens
local refill_rate = tonumber(ARGV[2])  -- Tokens per second
local now = tonumber(ARGV[3])          -- Current timestamp (ms)

-- Get current state
local data = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens = tonumber(data[1]) or capacity
local last_refill = tonumber(data[2]) or now

-- Calculate tokens to add since last refill
local elapsed = (now - last_refill) / 1000
local new_tokens = math.min(capacity, tokens + (elapsed * refill_rate))

-- Try to consume 1 token
if new_tokens >= 1 then
    new_tokens = new_tokens - 1
    redis.call('HMSET', key, 'tokens', new_tokens, 'last_refill', now)
    redis.call('EXPIRE', key, capacity / refill_rate * 2)
    return 1  -- ALLOWED
else
    redis.call('HMSET', key, 'tokens', new_tokens, 'last_refill', now)
    redis.call('EXPIRE', key, capacity / refill_rate * 2)
    return 0  -- REJECTED
end
```

> **💡 Why Lua?** Redis executes Lua scripts atomically — no other command can run in between. This prevents the read-then-write race condition that breaks rate limiting.

## 14. Throttling vs Rate Limiting vs Debouncing

| Concept | What it Does | Example |
| :--- | :--- | :--- |
| **Rate Limiting** | Hard cap: reject requests beyond the limit | "Max 100 API calls/min. Request #101 gets 429." |
| **Throttling** | Slow down processing instead of rejecting. Requests are delayed or queued. | "Process max 50 req/sec. Request #51 waits in a queue instead of being rejected." |
| **Debouncing** | Delay execution until activity stops. Only the LAST event fires. | "User types in search box. Don't fire API call on every keystroke — wait until they stop typing for 300ms." |

```
RATE LIMITING:     ✅✅✅✅✅❌❌❌  (hard reject after limit)
THROTTLING:        ✅✅✅✅✅⏳⏳⏳  (delay, process later)
DEBOUNCING:        ──────────────✅  (only fire once after quiet period)
```

## 15. DDoS Protection & WAF

### 15.1 Rate Limiting vs DDoS Protection

| Concern | Rate Limiting | DDoS Protection |
| :--- | :--- | :--- |
| **Target** | Individual abusive clients | Massive distributed botnets (millions of IPs) |
| **Volume** | Hundreds of req/sec from one IP | Millions of req/sec from thousands of IPs |
| **Layer** | Application layer (L7) | Network layer (L3/L4) + Application (L7) |
| **Solution** | API Gateway / Application code | CDN / WAF / Cloud shield (Cloudflare, AWS Shield) |

### 15.2 Defense in Depth Against DDoS

```
Layer 1: DNS / Anycast (Cloudflare)
├── Absorbs volumetric attacks (terabits/sec) at the edge
├── Traffic never reaches your servers

Layer 2: WAF (Web Application Firewall)
├── Blocks known attack patterns (SQL injection, XSS)
├── IP reputation scoring (known botnet IPs)
├── Geo-blocking (block entire countries if needed)

Layer 3: API Gateway Rate Limiting
├── Per-IP: 100 req/min
├── Per-User: 1000 req/hour
├── Per-Endpoint: custom limits

Layer 4: Application Rate Limiting
├── Business logic limits (3 login attempts/hour)
├── Anti-automation (CAPTCHA after 5 failed logins)
```

## 16. API Quota Management

### 16.1 Rate Limit vs Quota

| Concept | Time Window | Purpose |
| :--- | :--- | :--- |
| **Rate Limit** | Short (per second, per minute) | Prevent traffic spikes, protect server resources |
| **Quota** | Long (per day, per month) | Enforce billing tiers, control usage costs |

### 16.2 Tiered Quota Example (Like Stripe, Twilio, OpenAI)

| Tier | Rate Limit | Monthly Quota | Price |
| :--- | :--- | :--- | :--- |
| **Free** | 10 req/min | 1,000 req/month | $0 |
| **Starter** | 60 req/min | 50,000 req/month | $29/month |
| **Pro** | 600 req/min | 500,000 req/month | $99/month |
| **Enterprise** | 6,000 req/min | Unlimited | Custom pricing |

```
API Gateway checks:
  1. Is the request within the rate limit (short-term)?  → If no → 429
  2. Is the request within the monthly quota (long-term)? → If no → 403 with upgrade prompt
  3. Both pass? → Forward to backend ✅
```

## 17. Graceful Degradation Under Load

Instead of hard-rejecting requests at the rate limit, progressively degrade the service quality:

| Load Level | Response Strategy |
| :--- | :--- |
| **Normal** (< 70% capacity) | Full response. All features enabled. |

## 19. Interview Questions & Problems with Solutions

### ❓ Q1: "Design a rate limiter for a distributed system with 50 API servers."

**Answer:**
"The fundamental challenge is that requests are spread across 50 servers, so a local counter on each server is useless.

**Architecture:**
1.  Use a **centralized Redis** instance as the single source of truth for all rate limit counters.
2.  Algorithm: **Sliding Window Counter** for accuracy, or **Token Bucket** if burst tolerance is needed.
3.  Every API server calls `redis.incr()` (atomic) on each request. If the returned count > limit, return 429.
4.  Use Redis Lua scripts to make multi-step operations (check + increment + set TTL) atomic.
5.  **Fail-open policy:** If Redis is temporarily unreachable, allow requests through rather than blocking all users.
6.  Redis **Cluster** with read replicas for high availability of the rate limiter itself."

---

### ❓ Q2: "How would you handle a scenario where the rate limit needs to be different for each API endpoint?"

**Answer:**
"Use **per-endpoint rate limiting** at the API Gateway:

```
const ENDPOINT_LIMITS = {
  'GET:/api/search':       { limit: 30,  window: 60 },    // Expensive query
  'POST:/api/upload':      { limit: 5,   window: 60 },    // Heavy I/O
  'GET:/api/products':     { limit: 200, window: 60 },    // Lightweight read
  'POST:/api/ai/generate': { limit: 10,  window: 3600 },  // Expensive GPU compute
};

// Redis key: rate_limit:{userId}:{endpoint}:{windowId}
```

The API Gateway extracts the method + path, looks up the limit, and applies the appropriate rate limit per user per endpoint."

---

### ❓ Q3: "Fixed Window has a boundary burst problem. How do you fix it without using the full Sliding Window Log (too much memory)?"

**Answer:**
"Use the **Sliding Window Counter** algorithm. It keeps only 2 counters:
- Previous window count (e.g., 84 requests in 12:00-12:01)
- Current window count (e.g., 36 requests in 12:01-12:02)

At 12:01:15 (25% into the current window), the weighted estimate is:
`(84 × 75%) + 36 = 99`. This approximates a true sliding window using minimal memory. The accuracy is within 0.003% of a true sliding window according to Cloudflare's analysis."

---

### ❓ Q4: "A malicious user is rotating through thousands of IP addresses to bypass your per-IP rate limit. How do you stop them?"

**Answer:**
"Per-IP rate limiting is insufficient against distributed attackers. Layer additional strategies:
1.  **Per-Account rate limiting:** If authenticated, limit by user ID, not IP. Can't create infinite accounts (require email/phone verification).
2.  **Behavioral analysis:** Track request patterns (abnormal timing, sequential scraping, identical User-Agent) and flag suspicious behavior.
3.  **CAPTCHA challenges:** After N requests from suspicious sources, require CAPTCHA verification.
4.  **Device fingerprinting:** Track browser fingerprint (canvas, WebGL, fonts) — harder to rotate than IPs.
5.  **WAF with IP reputation:** Services like Cloudflare maintain databases of known botnet IPs and block them automatically."

---

### ❓ Q5: "Your rate limiter uses Redis. Redis goes down for 5 minutes. What happens?"

**Answer:**
"It depends on the **failure policy**:
- **Fail Open:** All requests are allowed through. For 5 minutes, there's no rate limiting. Risk: a bot could abuse the system. But legitimate users are unaffected. **This is the standard choice for most APIs.**
- **Fail Closed:** All requests are rejected. Risk: the entire API is down for 5 minutes — even though the backend services are perfectly healthy. Only use this for high-security systems.
- **Local Fallback:** Each server falls back to an in-memory approximate rate limiter. Limits are per-server (imperfect) but better than nothing. This is the most resilient approach.

Prevention: Use **Redis Sentinel** or **Redis Cluster** for automatic failover. A replica is promoted within seconds."
